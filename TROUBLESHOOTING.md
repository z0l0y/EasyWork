## 1. Agent 不遵守 EasyWork 规则

### 症状
Agent 跳过需求分析直接写代码、顺手重构无关模块、注释用英文、不做自审查。

### 原因
EasyWork 的 skill 文件没有被正确加载，或者 Agent 的上下文中 EasyWork 规则被后续对话冲淡。

### 自救步骤

**即时纠正**（在当前对话中）：
```
"请重新加载 .claude/skills/easywork/fullchain-dev-workflow/SKILL.md，
然后遵守其反模式清单继续执行。"
```

**预防措施**：
- 在 CLAUDE.md 的 **最顶部**（而非末尾）添加 EasyWork 引用。Claude Code 会给文件开头更高的注意力权重
- 如果 Agent 忽略特定约束（如"注释用中文"），直接说"检查 code-implement 技能的铁律1"
- 每次开始新的大任务时，用 `/clear` 清空上下文后再触发

**验证方法**：
新会话中先问："你现在加载了哪些 skill？"如果 Agent 列出了 EasyWork 的编排中枢，则加载成功。

## 2. Agent 卡在 CODE↔REVIEW 循环

### 症状
Agent 写完代码 → REVIEW 发现问题 → 修 → REVIEW 又发现新问题 → 再修 → REVIEW 还发现问题……超过 3 轮循环。

### 原因
EasyWork 有硬性限制（最多 3 轮回退循环）。如果触发了这个限制，说明修复方向可能有根本性问题——需求理解偏差、设计层面有缺陷、或者 REVIEW 标准过于严格。

### 自救步骤

Agent 应该已经挂起并输出类似：
```
【异常挂起 — REVIEW】
问题：CODE↔REVIEW 回退循环已达 3 次上限
```

如果 Agent **没有主动挂起**，你说：
```
"你已经来回改了 3 轮了。停止修改。分析一下循环的根因是什么，给我 2-3 个不同方向的方案选择。"
```

**你应该做的**：
1. 回顾 READ 阶段的需求理解——修复方向是否一开始就错了？
2. 检查 REVIEW 的中发现问题是否真的是"阻断级"——有些性能/可维护性问题可以在未来迭代解决
3. 考虑是否应该缩小修改范围，先打补丁（Hotfix），再规划正式重构

## 3. 上下文不够读完当前 Skill

### 症状
Agent 说"上下文不足"、输出被截断、或者跳过大量步骤。

### 原因
9 个 skill 全部加载会消耗大量 token。EasyWork 虽有"单技能加载模式"，但长对话中即便是加载单个 skill 也可能紧张。

### 自救步骤

**立即行动**：
1. 让 Agent 输出当前状态快照：
   ```
   "保存当前 EasyWork 状态快照"
   ```
2. 执行 `/clear` 清空上下文
3. 粘贴快照 JSON 到新对话：
   ```
   "从以下 EasyWork 状态快照恢复，继续执行：{JSON}"
   ```

**如果连状态快照都来不及保存**，手动告诉 Agent 当前进度：
```
"上一个会话中我们做到了 {步骤名称}，
已完成 {已完成步骤列表}，
中间产出：{关键发现}。
请从 {下一步} 继续。"
```

**预防措施**：
- 长任务优先选择"分步执行模式"，每 2-3 步主动保存快照并 `/clear`
- 对于搜索大量文件、运行大量测试的步骤，主动说"用子 Agent 执行这一步"
- 在项目中预先添加 `.claude/settings.json`，设置 `contextBudget` 阈值

## 4. 项目没有测试框架，EXAMINE 无法执行

### 症状
Agent 在 EXAMINE 阶段说"找不到测试命令"或"项目没有测试"。

### 原因
不是所有项目都有自动化测试。EasyWork 的策略是：能找到测试就跑，找不到就补最少量的测试。

### 自救步骤

**Agent 应该做的**（检查它是否做了）：
1. 搜索 `package.json` scripts、Makefile、CI 配置中的测试命令
2. 搜索 `*.test.*`、`*.spec.*`、`__tests__/` 目录
3. 确实找不到时，输出：
   ```
   【EXAMINE 阻断 - 测试环境不可用】
   选项：A) 跳过测试仅凭静态审查  B) 先搭建测试环境  C) 你手动验证
   ```

**如果 Agent 没有按 SOP 询问**，你说：
```
"项目没有测试。跳过 EXAMINE，用 REVIEW 的静态审查结果代替。"
```

**你的选择指南**：
- **Bug修复/功能开发** → 选 B 或 C（不要跳过验证）
- **快速迭代/原型** → 选 A（先上线，后续补测试）
- **微调/文案** → 选 A（不需要测试）

## 5. Agent 画了渲染不出来的 Mermaid 图

### 症状
Mermaid 代码在渲染时显示空白或语法错误。

### 常见错误及修复

| 错误现象 | 常见原因 | 修复方式 |
|---------|---------|---------|
| 完全空白 | 节点 ID 包含特殊字符 | 将节点 ID 改为纯英文 |
| 部分节点不显示 | 忘记关闭 subgraph | 检查 `end` 关键字 |
| 箭头方向错乱 | 混用 `-->` 和 `--->` | 统一使用 `-->` |
| 连线标签不显示 | `-->|标签|` 中标签含竖线 | 用 `-->|"标签文本"|` 包裹 |
| 节点文字被截断 | 文字中含括号 | 用引号包裹：`A["含(括号)的文字"]` |

**自救步骤**：
1. 把 Mermaid 代码复制到 [Mermaid Live Editor](https://mermaid.live/) 验证
2. 根据错误提示修改后，把修正版发给 Agent：
   ```
   "你的 Mermaid 图第 X 行有语法错误。修复后用 mermaid.live 验证了再给我。"
   ```
3. 如果问题反复出现，说"用文字描述代替 Mermaid，我自己画"

## 6. JSON 快照恢复失败

### 症状
粘贴状态快照后，Agent 说"无法解析"或找不到前序步骤的产出。

### 原因
快照 JSON 格式有误（换行符被截断、手动编辑时删掉了关键字段）、或前序步骤的 `step_outputs` 中缺少必填字段。

### 自救步骤

**检查快照是否完整**：
最小的可用快照应包含：
```json
{
  "easywork_version": "2.4",
  "task_type": "...",
  "current_step": 3,
  "completed_steps": ["READ", "CODE"],
  "skipped_steps": [],
  "step_outputs": {
    "read_output": { "goal": "...", ... },
    "code_output": { "files_changed": [...], ... }
  }
}
```

如果某个 `step_outputs` 为空或缺失，Agent 会无法继续。这时：
```
"快照中缺少 {step} 的产出。请重新执行 {step}，然后继续。"
```

**预防**：不要手动编辑快照 JSON。每次只复制粘贴 Agent 输出的完整 JSON 块。

## 7. 安装后 Agent 没有加载技能

### 症状
在 CLAUDE.md 中配置了 EasyWork，但 Agent 的行为和没有安装一样。

### 检查清单

- [ ] `CLAUDE.md` 中的路径是否写成了绝对路径（如 `C:\Users\xxx\EasyWork\`）但项目在别的机器上？
  - **修复**：改用相对路径 `.claude/skills/easywork/`（先运行安装脚本）
- [ ] 技能目录是否确实在 `.claude/skills/easywork/` 下？
  - **验证**：`ls .claude/skills/easywork/` 应看到 10 个子目录
- [ ] `SKILL.md` 文件是否存在且非空？
  - **验证**：`wc -l .claude/skills/easywork/fullchain-dev-workflow/SKILL.md`
- [ ] Claude Code 版本是否支持 skills？
  - **检查**：Claude Code v1.0+ 支持项目级 skills

## 8. Agent 擅自跳过了某个步骤

### 症状
Agent 在进度卡中直接标记了 `[skip]`，但没有先询问你。

### 原因
任务分类器推断该步骤可跳过（如纯理解任务跳过了 CODE/REVIEW/EXAMINE）。分类器有默认识别逻辑，但不是 100% 准确。

### 自救步骤

**如果要追加步骤**：
```
"不要跳过 {步骤名}，把它加回来。"
```

**如果要跳过更多步骤**：
```
"跳过 {步骤名}，不需要执行。"
```

EasyWork 的设计原则是：**分类器的默认建议只是建议，用户的指令是最终裁决**。

## 9. 拆分提交后发现拆错了

### 症状
GIT 阶段产出了拆分方案，但实际提交后发现某些文件放错了单元，或者拆得太碎/太粗。

### 自救步骤

**如果还没有实际提交**（Agent 只是输出了方案）：
```
"调整拆分方案：把 {文件} 从 {单元A} 移到 {单元B}，因为 {理由}。"
```

**如果已经执行了 `git commit`**：
```bash
git reset --soft HEAD~1    # 撤销最近 1 个提交，改动回到暂存区
git reset HEAD              # 全部移出暂存区
# 然后重新告诉 Agent：'按新方案重新拆分并提交'
```

**如果已经 push 了**：
```bash
git revert <commit-hash>    # 先回滚
# 然后重新让 Agent 生成拆分方案
```

**让 Agent 记住你的偏好**：
```
"以后拆分提交时，{配置类文件} 不要和 {逻辑类文件} 混在一起。"
```

## 通用自救命令速查

| 场景 | 对 Agent 说的话 |
|------|---------------|
| 跳过当前步骤 | "跳过这一步，直接做下一步" |
| 重做某一步 | "重新执行 {步骤名}" |
| 保存进度明天继续 | "保存状态快照，输出 JSON" |
| 从断点恢复 | "从以下快照继续：{粘贴JSON}" |
| 追加要跳过的步骤 | "把 {步骤名} 也加回来" |
| 强制走全流程 | "忽略分类建议，走完整 10 步" |
| 只看不改 | "这个任务是纯理解，不要改任何代码" |
| 紧急停止 | "停止当前操作，保存状态" |

## 还不行？

如果以上方案都无法解决你的问题，请检查：

1. **Agent 用的是哪个模型**——某些小型模型可能无法理解复杂的多步骤指令。建议使用 Claude Opus 4.8+ 或同等能力模型
2. **CLAUDE.md 中是否有冲突的指令**——其他 skill 或规则可能与 EasyWork 的约束冲突
3. **对话历史是否过长**——超过 200 轮对话后即使 `/clear` 也很难恢复。建议新开对话
