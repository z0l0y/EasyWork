# ⑤ AI 时代的反模式速查

> **核心主张**：知道不该做什么，比知道该做什么更重要。AI 时代有 AI 时代专属的陷阱。

---

## 🚨 反模式 1：跳过思考，直接要结果

### 表现

```
❌ "帮我修复这个 bug。"
   （没有上下文，没有根因分析，没有期望结果）

❌ "帮我加一个缓存。"
   （没说为什么要加，没说哪里要加，没说预期效果）

❌ "帮我重构这个模块。"
   （没说为什么重构，没说现有代码哪里不满意）
```

### 为什么危险

AI 会**永远给出答案**——即使你的问题本身就是错的。如果你跳过了"想清楚要什么"这一步，AI 就会替你定义问题，而它可能定义错了。你花时间实现了一个完美的方案，但解决的问题根本不存在。

### 正确做法

套用五问法，至少回答 WHY 和 WHAT 再触发 AI。

---

## 🚨 反模式 2：假性理解

### 表现

```
AI 给出了一个方案，你看了一遍，觉得"大概懂了"，然后就接受了。

但实际上：
- 你只是识别了其中的模式（哦，这是 React 的 useEffect 模式）
- 你并没有理解这段代码在你的上下文中的行为
- 你不知道它的边界条件
- 你不知道它失败时会发生什么
```

### 为什么危险

假性理解让人产生"在干活"的幻觉，实际上只是在搬运。长此以往，你对自己系统的理解只会越来越浅，出问题的概率越来越高。

### 自检问题

```
□ 我能用一句话解释这段代码在做什么吗？
□ 我能说出来什么情况下它会出错吗？
□ 我知道它为什么这样写，而不是另一种写法吗？
□ 如果删掉其中一行，会怎样？
□ 三个月后我还能看懂吗？
```

## 🚨 反模式 3：过度依赖流水线

### 表现

```
什么事都走完整全链流水线：
  "帮我修复一个拼写错误"
  → 需求理解 → 方案设计 → 代码实现 → 代码审查 → 测试 → 提交拆分
  （一个 30 秒的修改变成了 15 分钟的流水线仪式）
```

### 为什么危险

- **浪费上下文**：每次流水线都在消耗 token 和注意力
- **形式大于内容**：为了走流程而走流程，忘记了流程是工具不是目的
- **决策疲劳**：在小事上消耗了太多决策精力，大事上反而没精力想了

### 正确做法

用 Cynefin 框架判断问题复杂度，匹配恰当的流程深度：

| 问题类型 | 流程深度 | 示例 |
|---------|---------|------|
| 拼写错误 / 变量命名 | 直接改 | 不需要流水线 |
| 简单的 CRUD / 配置修改 | 实现 + 快速审查 | 2 步 |
| 中等复杂度功能 | 需求 → 实现 → 审查 → 测试 | 4 步 |
| 核心模块 / 架构改动 | 完整流程 | 全链路 |

## 🚨 反模式 4：不做偏差记录

### 表现

```
开发过程中发现：
- "嗯，这个跟预期不一样，但先记在脑子里……"
- "这个报错有点奇怪，但绕过去了……"
- "这个字段文档里没有，但看起来能用……"

不做记录，事后也忘了。
```

### 为什么危险

- 偏差是**最宝贵的信息**——它告诉你"世界不是你以为的那样"
- 不记录偏差 = 放弃了学习的机会
- 过了两个月，同样的问题再次出现，又从零开始排查

### 正确做法

偏差立刻记录，一句话就行（见 §04 习惯 2）。

## 🚨 反模式 5：知识不沉淀

### 表现

```
做了一个复杂的 debug，花了 3 小时。做完了，修好了，松了一口气。
然后关掉窗口，开始下一个任务。

没有留下：
- 根因是什么
- 解决步骤是什么
- 这个坑的识别特征是什么（下次怎么快速定位）
- 这个经验对团队其他人有什么用
```

### 为什么危险

- 三个月后，同一个坑，同一个人（或另一个人），再踩一次
- 团队知识没有积累，每个人都是零基础
- "老人"的优势没有转化为团队资产

### 正确做法

每完成一个耗时 > 30 分钟的任务，花 3 分钟写一条知识条目（见 §04 习惯 3）。

---

## 🚨 反模式 6：把 AI 的输出当作最终答案

### 表现

```
AI 生成了代码 → 直接复制粘贴 → 跑起来了 → 完工。
没有验证：边界条件、错误处理、性能、安全性、兼容性。
```

### 为什么危险

AI 的目标是**生成看起来合理的回答**，不是**生成正确的回答**。这两者有本质区别。

AI 会：
- 生成看起来合理但实际有 bug 的代码
- 使用过时或不再推荐的 API
- 漏掉错误处理
- 忽略安全最佳实践
- 引入你没要求的依赖

### 正确做法

AI 的输出 = **初稿**，不是**终稿**。必须经过你的审查、验证、测试。（见 §04 模板 3）

## 🚨 反模式 7：泛化不当

### 表现

```
这次需要处理"用户登录超时"的场景，
AI 或你自己顺手把"所有接口"的超时处理都改了。

结果：解决了一个问题，引入了三个新问题。
```

### 为什么危险

泛化增加了改动半径。你没有机会验证泛化部分在所有场景下的正确性。

### 正确做法

- 先做"刚好够解决当前问题"的改动
- 验证通过后，再考虑是否需要泛化
- 泛化必须有明确的动机（而不是"以后可能会用到"）
- 遵循 YAGNI（You Ain't Gonna Need It）原则

---

## 🚨 反模式 8：用速度换思考

### 表现

```
因为 AI 可以让"开始写代码"的速度变快 10 倍，
所以你觉得"快速出代码"比"想清楚再写"更重要。

一天能写 2000 行代码，但其中 1500 行后来要重构。
```

### 为什么危险

写代码的成本趋近于零，但**维护代码的成本并没有降低**。代码越多，负担越重。AI 让你更容易制造更多的技术债。

### 正确做法

- 想清楚再写，比写得多更重要
- 删代码比加代码更值得庆祝
- "这个功能不写会怎样？"——有时候最好的代码是不存在的代码

---

## 🚨 反模式 9：在自己的回音室里使用 AI

### 表现

```
遇到的每个问题都用相同的方式问 AI，得到相似的答案，陷入固定的思维模式。
```

### 为什么危险

AI 的回答风格受你的提问方式影响。如果你的提问总是同一个套路，AI 的回答也会是同一个套路。你就被困在了自己的认知回音室里。

### 正确做法

- 试试"扮演反对者"："请对我的这个方案提出最严厉的批评。"
- 试试"倒置法提问"："这个方案最可能失败的原因是什么？"
- 试试 EasyWork 的 `/easywork:selfcheck`（CTO 拷打模式）
- 拿 AI 的答案去问另一个 AI（或同一个人换个角度问）
