# ① 五问法：动手前先想清楚

> **核心主张**：做任何事之前，先回答五个问题。少一个，就是在碰运气。

---

## 五问框架

```
                    ┌──────────────┐
                    │   ① 原因     │  ← 为什么要干？为什么之前没干？
                    │   WHY       │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   ② 动作     │  ← 具体要干什么？边界在哪？
                    │   WHAT      │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   ③ 方法     │  ← 打算怎么干？有没有更好的方式？
                    │   HOW       │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   ④ 结果     │  ← 想要什么结果？怎么验证做到了？
                    │   RESULT    │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   ⑤ 影响     │  ← 会有什么后果？未来还会改什么？
                    │   IMPACT    │
                    └──────────────┘
```

## ① 原因（WHY）—— 为什么要干这个？

> **不问原因就动手，方向错了越努力越错。**

### 必须回答的三个子问题

| # | 问题 | 如果不能回答…… |
|---|------|---------------|
| 1 | 当前发生了什么，促使我必须做这件事？ | 你在解决一个不存在的问题 |
| 2 | 为什么之前没有这样做？（技术债？不知道？做不到？） | 你可能在重复造轮子，或者忽略历史教训 |
| 3 | 不做会怎样？做晚了会怎样？ | 你无法判断这件事的真实优先级 |

### 示例

```
❌ 坏："我要把数据库从 MySQL 换成 PostgreSQL。"
   → 为什么要换？MySQL 哪里不够用了？是性能问题还是功能需求？
   → 之前为什么选 MySQL？那个决策的条件现在还成立吗？

✅ 好："当前 MySQL 5.7 不支持 JSON 聚合函数（原因），
   业务需要实时生成嵌套 JSON 报表（触发条件），
   之前选 MySQL 是因为当时没有 JSON 需求（历史原因）。
   如果不换，报表功能需要应用层 N+1 查询，性能预估差 10 倍（不做会怎样）。"
```

### 配合 EasyWork

```
/easywork:requirements  → 帮助理解"用户到底遇到了什么问题"
/easywork:project       → 帮助理解"当前系统为什么长这样"
/easywork:trace         → 帮助追踪"这个改动的上游触发源"
```

## ② 动作（WHAT）—— 具体要干什么？

> **范围不清，要么做得太多（浪费），要么做得太少（返工）。**

### 必须回答的三个子问题

| # | 问题 | 如果不能回答…… |
|---|------|---------------|
| 1 | 这次改动的**精确边界**是什么？（改哪些文件、哪些模块、哪些接口） | 范围蔓延，越做越大 |
| 2 | 这次**不动**什么？（明确排除项） | 可能顺手改了不相关的，引入新问题 |
| 3 | 改动的最小可交付单元是什么？（MVP 切片） | 没法分步交付，只能一把梭 |

### 示例

```
❌ 坏："优化一下用户模块的性能。"
   → "用户模块"是多大范围？哪些接口？什么算"优化"？
   → 不动什么？权限逻辑要不要碰？缓存策略要不要改？

✅ 好："优化 /api/user/list 接口（边界：仅此一个接口），
   目标是将 P99 延迟从 800ms 降到 200ms 以内（可衡量），
   本次不动权限校验逻辑、不动缓存策略（排除项），
   最小交付：添加数据库索引 + 移除 N+1 查询（MVP）。"
```

### 配合 EasyWork

```
/easywork:decompose  → 把大需求拆成可独立交付的小任务
/easywork:compare    → 对比不同实现路径的影响面
```

---

## ③ 方法（HOW）—— 打算怎么干？

> **同样的事，路径不同，代价差十倍。**

### 必须回答的三个子问题

| # | 问题 | 如果不能回答…… |
|---|------|---------------|
| 1 | 我打算怎么做？步骤是什么？ | 执行过程变成布朗运动 |
| 2 | 有没有其他方式？为什么选这个不选那个？ | 选了第一个想到的方案，不一定是好方案 |
| 3 | 失败了怎么回滚？Plan B 是什么？ | 出了事只能慌乱补救 |

### 示例

```
❌ 坏："加个缓存就行了。"
   → 什么缓存？本地还是 Redis？过期策略？缓存穿透怎么办？
   → 有没有其他方案？数据库索引？读写分离？ES 异构？

✅ 好："方案 A（推荐）：Redis 缓存，TTL 5min，Cache-Aside 模式。
   备选方案 B：MySQL 覆盖索引，P99 预估降到 400ms（不如 A 但零依赖）。
   选 A 因为 QPS 已到 2000，索引优化天花板不够。
   回滚：缓存开关配置热关闭，降级到直读数据库。
   风险：缓存雪崩 → 解决：TTL 加随机抖动 0-60s。"
```

### 配合 EasyWork

```
/easywork:implement   → 方案→代码变更，含改动原因和影响面
/easywork:diagram     → 画出方案架构，一图胜千言
/easywork:compare     → 多方案对比评分矩阵
```

## ④ 结果（RESULT）—— 想要什么结果？怎么验证？

> **说不清"怎样算做完"，就永远做不完。**

### 必须回答的三个子问题

| # | 问题 | 如果不能回答…… |
|---|------|---------------|
| 1 | 成功的**可量化标准**是什么？（数字，不是感觉） | 验收变成玄学 |
| 2 | 怎么验证？测试步骤是什么？ | 上线即开盲盒 |
| 3 | 与之前相比，差异体现在哪里？怎么让相关方感知到？ | 做了好事没人知道，出了事人人追责 |

### 示例

```
❌ 坏："接口变快了就行。"
   → "变快"是多少？从多少到多少？谁说了算？
   → 怎么证明变快了？压测报告？APM 截图？

✅ 好："P99 延迟：800ms → ≤200ms（可量化标准），
   QPS 上限：1000 → ≥3000。
   验证方式：JMeter 压测脚本（100 并发，持续 5min），
   对比改动前后 APM 截图 + 压测报告。
   感知方式：运维大屏延迟曲线下降、用户反馈表单加载速度提升。"
```

### 配合 EasyWork

```
/easywork:examine    → 测试覆盖率 + 质量验证
/easywork:review     → 七维度代码审查
/easywork:coverage   → 五维测试覆盖 + 盲区检测
/easywork:test       → 接口联调测试 + 预期结果预计算
```

## ⑤ 影响（IMPACT）—— 会有什么后果？

> **只想第一步不想第二步，迟早要还技术债。**

### 必须回答的三个子问题

| # | 问题 | 如果不能回答…… |
|---|------|---------------|
| 1 | 这个改动会影响谁？（上下游、其他团队、用户） | 改了 A，炸了 B |
| 2 | 未来三个月内，这个模块还可能怎么改？ | 今天的设计可能阻碍明天的需求 |
| 3 | 如果这个改动出问题了，最坏的后果是什么？ | 没有应急预案 |

### 示例

```
❌ 坏："这个改动只影响我自己的模块，不会有问题的。"
   → 你的 API 有没有人调用？你的数据表有没有人读？
   → 你的接口签名改了，前端的调用方知道吗？

✅ 好："影响面：
   - 上游：前端需要配合升级 API 版本（已通知 @前端负责人）
   - 下游：报表服务读这张表，索引变更可能影响其查询计划
   - 未来：Q3 计划做多租户改造，本表结构要预留 tenant_id
   - 最坏情况：索引创建期间表锁，导致写入中断 < 3s（已评估）
   - 回滚：DROP INDEX 即可，无数据迁移风险"
```

### 配合 EasyWork

```
/easywork:retro       → 5-Whys 根因分析 + Trade-offs
/easywork:selfcheck   → CTO 四阶段盘问（认知缺口 + 汇报就绪）
/easywork:meta        → 网状自治扩散，发现隐藏影响面
```

## 🔗 五问之间的关系

这五个问题不是独立的，它们互相约束：

```
原因(Why) 决定 动作(What) 的优先级
动作(What) 决定 方法(How) 的粒度
方法(How) 决定 结果(Result) 的形态
结果(Result) 决定 影响(Impact) 的评估方式
影响(Impact) 反过来修正 原因(Why) —— "搞了半天影响这么大，当初为什么要搞？"
```

> ⚠️ **常犯错误**：把五个问题顺序回答完就算完事。实际上它们是**迭代**的——回答⑤时发现影响面太大，就应该回头修正①，重新评估"到底要不要做"。
