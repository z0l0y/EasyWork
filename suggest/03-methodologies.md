# ③ 业界方法论补全

> **核心主张**：你不需要记住所有方法论。但了解它们，可以在需要时调出来用。
> 每个方法论都是一把**思维工具刀**，合适的时候拿出来，能少走很多弯路。

---

## 方法论矩阵：什么时候用什么？

```
                     问题清晰度
                    ┌─────────┐
                    │ 清晰    │ → 5 Whys / PDCA / ADR
                    │ 模糊    │ → OODA / Wardley Mapping / Cynefin
                    └─────────┘
                     时间压力
                    ┌─────────┐
                    │ 紧迫    │ → OODA / BLUF / 启发式
                    │ 充裕    │ → 第一性原理 / 二阶思维 / 预演失败
                    └─────────┘
                     决策影响
                    ┌─────────┐
                    │ 可逆    │ → 快速决策 / MVP / 曳光弹
                    │ 不可逆  │ → 第一性原理 / ADR / 预演失败 / 倒置法
                    └─────────┘
```

## 🏭 5 Whys（五问法）— 找根因

> **来源**：丰田生产系统 (Toyota Production System)，大野耐一

### 核心思想

碰到问题，连续问五个"为什么"，直到找到根本原因而非表面症状。

### 怎么用

```
表面问题："支付接口超时了。"
① 为什么超时？            → 数据库查询花了 5 秒
② 为什么查询花了 5 秒？    → 没有走索引，全表扫描
③ 为什么没有走索引？       → 查询条件用了函数包裹字段：WHERE DATE(created_at) = ...
④ 为什么用了函数包裹？     → 需求要求按"自然日"查询，开发不知道可以改写为范围查询
⑤ 为什么开发不知道？       → 团队没有 SQL 性能审查规范，code review 不检查执行计划

根因：规范缺失，而非个人失误。
行动：PR 模板增加 SQL 审查项 + 团队分享一次 MySQL 索引失效案例。
```

### 注意事项

- 不一定是恰好 5 个，**多一个少一个都行**，关键是一直追到系统性问题
- 不要止步于"人犯错了"——追问为什么这个人会犯错（流程？工具？培训？）
- 不要变成 blame game——目标是改进系统，不是找人背锅

## 🔄 PDCA（戴明环）— 持续改进

> **来源**：W. Edwards Deming，质量管理大师

### 核心思想

```
   Plan（计划）
    │
    ├─→ Do（执行）
    │     │
    │     ├─→ Check（检查）
    │     │     │
    │     │     ├─→ Act（行动/改进）
    │     │     │     │
    │     │     │     └─→ 回到 Plan
    │     │     │
    │     │     └─→ [问题?] → 调整后继续
    │     │
    │     └─→ [问题?] → 调整后继续
    │
    └─→ [重大偏差] → 重新计划
```

### 与五问法 + 时间维度的关系

| PDCA | 对应时间维度 | 对应五问 |
|------|------------|---------|
| Plan | 事前 | WHY + WHAT + HOW |
| Do | 事中 | HOW（执行） |
| Check | 事中→事后 | RESULT（验收） |
| Act | 事后 | IMPACT（改进） |

### 常见误区

- ❌ 只 PD 不 CA：计划了、执行了，不检查、不改进
- ❌ Act 变成"下次注意"：没有具体行动，没有跟踪闭环
- ❌ 一次 PDCA 就结束：PDCA 是循环，没有终点

## 🎯 OODA 循环 — 快速适应

> **来源**：John Boyd，美国空军上校，空战战术理论家

### 核心思想

```
Observe（观察）→ Orient（判断）→ Decide（决策）→ Act（行动）→ 循环
```

### 关键洞察

Boyd 强调的不是"每一步做得多好"，而是**循环速度**。谁能更快完成 OODA，谁就能先发制人——你的行动让对方还在判断时就已经过时了。

### 在软件开发中的应用

```
传统方式：
  需求分析（2天）→ 技术方案（1天）→ 开发（5天）→ 测试（3天）
  → OODA 循环周期：11 天

敏捷方式：
  用户故事 → 小粒度拆分 → 1天开发 → 当天测试 → 第二天反馈
  → OODA 循环周期：1 天

优势：11 天内你能做 11 次调整，传统方式只能做 1 次。
```

### 与 EasyWork 的关系

EasyWork 的"点·线·网"三级编排本质上就是在**加速 OODA 循环**：
- 点模式：快速单步 OODA
- 线模式：2-3 步串联 OODA
- 网模式：多维度并行 OODA

## 🧱 第一性原理 — 打破惯性

> **来源**：亚里士多德，近代被 Elon Musk 广泛推广

### 核心思想

把问题分解到最基本的真理，然后从那里重新构建，而不是基于已有的做法（类比）来推理。

```
类比思维（大多数人）：
  "别人都是这样做的，所以我们也这样做。"
  "以前是这样做的，所以继续这样做。"

第一性原理思维：
  "这件事的基本物理/逻辑约束是什么？"
  "如果从零开始，我会怎么设计？"
  "现在做的这件事，哪些部分是基于事实，哪些是基于惯性？"
```

### 怎么用

```
问题："用户反馈登录太慢。"
类比思维："加个 CDN，多搞几台机器。"
第一性原理：
  1. 登录的本质是什么？ → 身份验证 + Session 创建
  2. 耗时分解到原子操作：
     - 网络往返：20ms
     - 密码哈希：5ms
     - 数据库查询：800ms ← 瓶颈
     - Session 写入 Redis：10ms
  3. 数据库查询为什么耗时 800ms？
     → 没有索引 + 关联了不必要的表
  4. 解决：加索引 + 移除不必要的 JOIN → 从 800ms 降到 30ms
  5. 不需要加机器，不需要买 CDN。
```

### 适用场景

- 技术选型（"为什么选 A 不选 B？A 解决了什么根本问题？"）
- 架构重构（"如果今天重新设计，会怎么做？"）
- 打破惯性（"一直这样做 ≠ 应该这样做"）

## 🔮 二阶思维 — 考虑后果的后果

> **来源**：Howard Marks（投资家）、Ray Dalio（桥水基金）

### 核心思想

```
一阶思维（大多数人）：
  "做了 A → B 会发生。"

二阶思维：
  "做了 A → B 会发生 → B 发生之后 → C 会发生。"

三阶思维：
  "做了 A → B → C → D 会发生。"
```

### 怎么用

```
场景：为了修复一个 bug，你决定把一个 public 方法的签名改了。

一阶：bug 修好了，方法工作了。✅
二阶：所有调用这个方法的地方都编译失败了。❌
三阶：修了所有调用方 → 其中一个调用方改了之后，
      它的下游也坏了 → 最终影响面扩散到 3 个团队。

如果一开始就用二阶思维：
  "修 bug 要改方法签名。改签名会影响调用方。
   → 查询所有调用方（IDE Find Usages）
   → 发现 12 处引用，其中 3 处在其他团队的代码中
   → 方案改为：保留旧方法 deprecate，新增一个正确签名的方法
   → 通知调用方迁移，给一个迭代的缓冲期"
```

### 训练方法

每次做决策，加问一句：**"然后呢？"**（And then what?）

## 🪞 倒置法（Inversion）— 从反面思考

> **来源**：卡尔·雅可比（数学家："倒置，总是倒置"），查理·芒格广泛推广

### 核心思想

与其问"怎么成功？"，不如问"怎么失败？"然后避开所有失败的原因。

```
正向：我怎么写出好代码？
倒置：怎样的代码一定是烂代码？ → 不写那样的。

正向：怎么让项目按时交付？
倒置：项目延期都是什么原因导致的？ → 逐一预防。

正向：怎么让团队高效协作？
倒置：怎样的团队一定低效？ → 避免那些特征。
```

### 与预演失败（Premortem）的关系

倒置法是思维工具，预演失败是倒置法的具体实践。

## 📐 Cynefin 框架 — 先判断问题类型，再选方法

> **来源**：Dave Snowden，认知科学家

### 五种问题域

```
┌────────────┬────────────┬──────────────────────────────┐
│   域       │  关系      │  策略                        │
├────────────┼────────────┼──────────────────────────────┤
│ 清晰 Clear │ 因果明确   │ 感知→归类→响应               │
│            │            │ （最佳实践，标准流程）         │
├────────────┼────────────┼──────────────────────────────┤
│ 繁杂 Complicated │ 因果可分析 │ 感知→分析→响应          │
│            │            │ （专家分析，多种好答案）       │
├────────────┼────────────┼──────────────────────────────┤
│ 复杂 Complex │ 因果仅可回溯 │ 探测→感知→响应            │
│            │            │ （先做小实验，涌现结果）       │
├────────────┼────────────┼──────────────────────────────┤
│ 混沌 Chaotic │ 无清晰因果 │ 行动→感知→响应              │
│            │            │ （先止血，再分析）            │
├────────────┼────────────┼──────────────────────────────┤
│ 混乱 Disorder │ 不确定     │ 拆分成子问题再分类          │
└────────────┴────────────┴──────────────────────────────┘
```

### 为什么这很重要？

**对清晰问题用探索策略（Complex approach）是浪费时间。**
**对复杂问题用标准流程（Clear approach）是假装在解决问题。**

```
清晰域例子："加一个 CRUD API。" → 标准实现流程即可，不需要探索
复杂域例子："重构遗留系统核心模块。" → 需要试探性重构 + 增量验证
混沌域例子："线上数据库删库了。" → 先恢复服务，别急着分析根因
```

### Cynefin 与 EasyWork 技能选择

| 问题域 | 建议 EasyWork 策略 |
|--------|-------------------|
| 清晰 | 单技能直达（"帮我加个 API"） |
| 繁杂 | 线模式（需求理解→实现→审查→测试） |
| 复杂 | 网模式 / 场景画布（多维探索 + 增量交付） |
| 混沌 | 先用人工判断稳定局势，再用技能辅助 |
| 混乱 | 先手动拆解问题，再逐个分类处理 |

## 🏗️ ADR（架构决策记录）— 记录 Why 而非 What

> **来源**：Michael Nygard，ThoughtWorks

### 核心思想

代码告诉你"做了什么"，但**不告诉你"为什么这样做"**。ADR 补上这一环。

### ADR 模板

```markdown
# ADR-XXX：{简短标题}

## 状态
[提议中 / 已采纳 / 已废弃 / 已替代]

## 背景
当前发生了什么，促使我们必须做一个架构决策？

## 决策
我们决定做什么？

## 备选方案
还考虑过哪些方案？为什么没选？
- 方案 A：...（为什么不选）
- 方案 B：...（为什么不选）

## 后果
### 正面
- ...
### 负面
- ...
### 风险
- ...

## 未来考量
什么情况下这个决策应该被重新审视？
```

### 为什么 ADR 很重要？

三个月后，有人看到一段奇怪的代码，问："当初谁写的？为什么不直接 XXX？"

如果没有 ADR，回答是："不知道，可能是乱写的。"
如果有 ADR，回答是："当时因为 XXX 限制，只能这样。那个限制在 Q3 会解除——TODO 注释里标了。"

## 🧠 双系统理论 — 觉察自己的思考模式

> **来源**：Daniel Kahneman，《思考，快与慢》（Thinking, Fast and Slow）

### 系统 1 vs 系统 2

```
系统 1（快思维）：
- 自动的、直觉的、不费力的
- "这个简单，这样改就行了！"
- 适合：熟悉的、成型的问题
- 陷阱：模式误配——看起来像，其实不是

系统 2（慢思维）：
- 刻意的、分析的、费力的
- "等等，让我想想这个改动的影响面……"
- 适合：陌生的、复杂的问题
- 陷阱：懒惰——大脑默认不启动系统 2
```

### 怎么用

```
当你觉得"这个很简单，这样改就行了"——
  问自己：我是真的理解了，还是只是在做模式匹配？

当你遇到不熟悉的问题却想用熟悉的方法解决时——
  停下来，这是系统 1 在偷懒。启动系统 2。

当 EasyWork Agent 给出一个看起来"显然正确"的方案时——
  不要直接接受。用系统 2 审视一遍：它真的理解我的上下文吗？
```

## 📚 速查：什么情况用什么工具？

| 你的处境 | 推荐方法论 | 一句话 |
|---------|-----------|--------|
| 碰到一个 bug，不知道根因 | 5 Whys | 连问五个"为什么" |
| 准备开始一个新任务 | 五问法 §01 + 预演失败 | 动手前先想清楚 + 如果失败了会是什么原因 |
| 执行过程中发现偏差 | OODA | 观察→判断→决策→行动，加速循环 |
| 做了一个重要技术决策 | ADR | 记录为什么这样选，不只是选了啥 |
| 技术选型，要从多个方案选一个 | 第一性原理 + 二阶思维 | 打破惯性 + 考虑后果的后果 |
| 项目做完了，要复盘 | PDCA + 5 Whys | 检查 vs 计划 + 追问根因 |
| 总觉得自己在做重复劳动 | PDCA（Act 环节） | 没有把改进落地成流程/工具 |
| 觉得某个方案"显然"是对的 | 双系统（启动系统 2） + 倒置法 | 如果这个方案会失败，最可能的原因是什么？ |
| 面对一个全新领域，不知道从哪开始 | Cynefin | 先判断问题域类型，再选策略 |
| 要给团队或领导汇报 | BLUF | 结论先行，细节按需展开 |

---

> 💡 **记住**：方法论是帮你**想得更清楚**的，不是用来**秀知识面**的。选一个顺手的，用到成为本能，比知道十个但一个都不用好。
