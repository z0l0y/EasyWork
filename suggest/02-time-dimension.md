# ② 时间维度：事前·事中·事后

> **核心主张**：每一个动作都有时间坐标。分不清事前、事中、事后，就会把本该事前想的事情拖到事中慌乱，把本该事后复盘的事情忘到九霄云外。

---

## 三阶段总览

```
事前（BEFORE）              事中（DURING）              事后（AFTER）
─────────────────────────────────────────────────────────────────────
想清楚                      做对                        学到
│                           │                           │
├─ ① 为什么干？              ├─ ① 按计划执行              ├─ ① 结果 vs 预期？
├─ ② 具体干什么？            ├─ ② 每个步骤验证            ├─ ② 偏差原因？
├─ ③ 打算怎么干？            ├─ ③ 偏差立刻记录            ├─ ③ 下次怎么改？
├─ ④ 想要什么结果？          ├─ ④ 碰到意外，停下来         ├─ ④ 沉淀成知识
├─ ⑤ 会有什么影响？          │    重新评估                 ├─ ⑤ 告知相关方
│                           │                           │
输出：行动计划              输出：执行记录               输出：改进措施
```

---

## 事前（BEFORE）—— 想清楚再动手

> **事前多花 10 分钟，事中少花 2 小时，事后少背 3 年锅。**

### 事前检查清单

```
□ [WHY]  我清楚为什么要做这件事吗？
□ [WHY]  我知道为什么之前没有做吗？
□ [WHAT]  我明确这次要改什么、不改什么吗？
□ [WHAT]  最小可交付单元切出来了吗？
□ [HOW]   方案比较过了吗？选这个方案的原因是什么？
□ [HOW]   失败了怎么回滚？Plan B 是什么？
□ [RESULT] 成功的可量化标准写下来了吗？
□ [RESULT] 验证步骤明确了吗？
□ [IMPACT] 影响面评估过了吗？（上下游/其他模块/用户）
□ [IMPACT] 最坏情况是什么？能承受吗？
□ [COMM]   需要提前通知谁？改完后需要告知谁？
```

### 事前最容易被跳过的三件事

| 被跳过的事 | 后果 | 怎么补救 |
|-----------|------|---------|
| **为什么不之前做？** | 忽略历史决策背景，可能踩同一个坑两次 | 查 git log、查 ADR、问老同事 |
| **不动什么？** | 范围蔓延，越改越多 | 在任务描述中明确写"本次不改 X" |
| **失败了怎么办？** | 没有回滚方案，出了事只能硬扛 | 每个任务必须配回滚步骤 |

### 🛠️ 业界方法补充：预演失败（Premortem）

> 源自 Gary Klein，诺贝尔奖得主 Kahneman 推荐。

在项目开始前，做一个反向思维实验：

```
假设我们穿越到 6 个月后，这个项目彻底失败了。
现在，请写下失败的原因——所有你能想到的。

然后，针对每一个失败原因，问：我们现在能做什么来防止它发生？
```

这个方法的价值在于：**人天然乐观，预演失败强迫你正视风险。**

## 事中（DURING）—— 边做边验证

> **事中不是"埋头干就完了"。事中是持续验证、持续校准的过程。**

### 事中三大纪律

#### 纪律 1：按计划执行，但不要盲从计划

```
每完成一个步骤，问自己：
□ 当前产出是否符合预期？
□ 有没有发现计划中没有覆盖的新情况？
□ 如果有新发现，计划要不要调整？
```

#### 纪律 2：偏差立刻记录，不要"等做完再说"

```
❌ 坏："这个地方跟预期不太一样，但先记在脑子里，搞完了再说。"
   → 十分钟后忘记了，或者记不清具体偏差是多少。

✅ 好："API 返回格式比文档多了两个字段（extra_field_1, extra_field_2），
   已截图保存，不确定是新增还是文档遗漏，代码中先做兼容处理，
   事后确认后决定是否清理。—— 2026-07-01 14:23"
```

> 💡 偏差记录不需要长篇大论，一句话就行。关键是**立刻写、当场写、趁热写。** EasyWork 的知识库 Hook 会在每轮对话结束时自动捕获你的发现，但前提是你得说出来。

#### 纪律 3：碰到意外，停下来重新评估

```
如果执行过程中发现：
- 影响面比预期大（比如要改的文件多了 3 倍）
- 方案有致命缺陷（比如选的技术栈有兼容性问题）
- 上下游冲突（比如同事刚好也在改同一个模块）

→ 不要硬推。停下来，回到事前重新评估五问。
```

### 🛠️ 业界方法补充：OODA 循环

> 源自美国空军上校 John Boyd，后来被广泛用于商业决策。

```
  ┌──────────┐
  │ Observe  │ ← 观察：当前发生了什么？
  │  观察    │
  └────┬─────┘
       │
  ┌────▼─────┐
  │ Orient   │ ← 判断：这对我的计划意味着什么？
  │  判断    │
  └────┬─────┘
       │
  ┌────▼─────┐
  │ Decide   │ ← 决策：我应该调整计划吗？怎么调？
  │  决策    │
  └────┬─────┘
       │
  ┌────▼─────┐
  │   Act    │ ← 行动：执行调整后的计划
  │  行动    │
  └──────────┘
       │
       └──→ 循环回到 Observe
```

**关键洞察**：Boyd 的"OODA Loop"本意是**比对手更快地完成这个循环**。在软件开发中，"对手"就是**复杂性和不确定性**。你循环得越快，越能提前发现问题和调整方向。

## 事后（AFTER）—— 复盘 → 沉淀 → 改进

> **事后不是"搞完了就完了"。事后是学习发生的时刻。**

### 事后三件事

#### 1. 结果对比

```
对比维度：
□ 实际结果 vs 预期结果：一致吗？
□ 实际时间 vs 预估时间：偏差多少？为什么？
□ 实际影响 vs 预估影响：有没有意外影响？
```

#### 2. 根因分析（不只是"出了什么问题"）

```
不要停留在："登录慢了，因为数据库查询慢。"
要追问：
1. 为什么数据库查询慢？（没有索引）
2. 为什么没有索引？（表结构是上一个人建的，没有文档）
3. 为什么没有文档？（没有强制要求）
4. 为什么没有强制要求？（团队没有代码审查包含性能检查的规范）
5. 为什么没有这个规范？（没有人在事前推动过）

第 5 层才是真正的根因 → 行动：在 PR 模板中加入性能审查检查项。
```

#### 3. 知识沉淀

```
做完一件事，至少沉淀一条知识：
□ 这个坑别人可能还会踩 → 写到项目的 TROUBLESHOOTING 里
□ 这个决策以后可能被质疑 → 写一条 ADR
□ 这个经验同类项目需要 → 写一条 EasyWork 知识条目
□ 这个模式反复出现了 → 考虑提取成模板或自动化
```

### 事后最容易被跳过的三件事

| 被跳过的事 | 后果 | 怎么补救 |
|-----------|------|---------|
| **根因分析只停在表面** | 头痛医头，下次换个表现形式再犯 | 强制追问到第 5 层"为什么" |
| **知识不沉淀** | 同一个坑，下一个人（或三个月后的自己）再踩一次 | 建一个"已踩过的坑"文档 |
| **不告知相关方** | 你改了 API 签名，前端同事上线才报错 | 发送变更通知，附 changelog |
