# ④ AI 时代的实践指南

> **核心主张**：思维链不是背出来的，是用出来的。这一章把前面的框架翻译成日常可执行的动作。

## 原则 0：AI 是副驾驶，你才是机长

```
┌─────────────────────────────────────────────┐
│                                             │
│   你（机长）         AI（副驾驶）             │
│   ─────────         ────────────             │
│   ✅ 决定飞哪         ✅ 帮你导航             │
│   ✅ 判断能不能飞     ✅ 提醒风险             │
│   ✅ 最终拍板         ✅ 执行计算             │
│   ✅ 承担后果         ❌ 不能承担后果          │
│                                             │
└─────────────────────────────────────────────┘
```

**任何时候，如果你对 AI 说"就这样吧，你觉得对就行"——你已经放弃了机长的职责。**

## 日常工作的思维链模板

### 模板 1：接到一个新任务

```
步骤 1：启动系统 2（不要秒回"好的"）
步骤 2：套用五问法
步骤 3：判断问题域类型（Cynefin）
步骤 4：选 EasyWork 策略
步骤 5：预演失败（最坏会怎样？）
步骤 6：开始执行

示例对话：

老板："排查一下为什么用户反馈登录很慢。"

你（系统 2 启动，五问开始）：
  WHY  → 为什么现在排查？有具体用户投诉还是 APM 告警？影响面多大？
  WHAT → 只排查后端登录接口？还是含前端页面加载？PC端还是移动端？
  HOW  → APM 看火焰图？数据库慢日志？网络链路追踪？
  RESULT → 给出根因报告 + 修复建议？还是直接修好？
  IMPACT → 如果需要加索引，会不会锁表？需要停机窗口吗？

然后："好的，我先确认几件事：（列出五问的答案选项给老板确认）"
```

### 模板 2：让 AI 帮你写代码

```
步骤 1：自己先想清楚五问（不要直接扔需求给 AI）
步骤 2：用自然语言描述五问的答案给 AI
步骤 3：Review AI 的输出——用倒置法（如果这段代码会出 bug，最可能在哪？）
步骤 4：验证——用 §01 的结果标准来验收
步骤 5：沉淀——本次有什么值得记录的认知？

示例对话（你→AI）：

❌ 坏："帮我写一个用户登录接口。"
   → 你没想清楚，AI 只能猜，猜错了你也不知道。

✅ 好：
  "我需要实现一个用户登录接口，背景是：
   ① WHY：当前登录使用 Session，需要改为 JWT 以支持移动端无状态认证
   ② WHAT：只改 /api/auth/login 接口的 token 生成部分，不动密码加密和用户校验
   ③ HOW：用 jsonwebtoken 库，RS256 算法，access_token 15min + refresh_token 7d
   ④ RESULT：接口返回 { access_token, refresh_token, expires_in }，
              旧版客户端兼容（仍接受旧的 session_id 方式，但返回 deprecation warning）
   ⑤ IMPACT：移动端和 Web 端都需要更新 token 存储逻辑，
              已通知 @前端负责人，计划下周同步上线
   请帮我实现这个接口，注意：
   - 不要改密码校验逻辑
   - 保持向后兼容（旧 header 仍能工作）
   - 生成单元测试覆盖正常流 + token 过期流"
```

结果：
- AI 的输出质量**天壤之别**——因为输入质量天壤之别
- 你不需要重写——因为你已经把边界和约束说清楚了
- Review 成本极低——你只需要对照五问逐项检查

### 模板 3：Review AI 的代码

```
步骤 1：对照你的五问，检查 AI 有没有跑偏
步骤 2：用倒置法——"如果这段代码会出 bug，最可能在哪？"
步骤 3：用二阶思维——"这个改动上线后，然后呢？"
步骤 4：检查 AI 的"盲区"：
  □ 有没有改了你没说可以改的地方？（范围越界）
  □ 有没有引入新的依赖？（供应链风险）
  □ 有没有删除你确认要保留的代码？（"看起来没用到"是 AI 常见借口）
  □ 错误处理是否完整？
  □ 是否有并发安全问题？
  □ 日志/监控埋点是否足够？

步骤 5：如果发现问题，不要只说"这里不对"，要说"这里不对，因为……（五问中的 WHY）"
```

## 🛠️ 《程序员修炼之道》中的关键实践

> **来源**：Dave Thomas & Andy Hunt，《The Pragmatic Programmer》

### 曳光弹开发（Tracer Bullets）

> 不要一次性构建整个系统。先放出一颗"曳光弹"——一条端到端贯穿所有层的极简实现——验证架构假设，然后再迭代。

```
应用到 EasyWork：
  不要上来就走完整的 10 步全链路流程。
  先用一个最小场景验证你的需求理解是对的：
  /easywork:decompose → 拆出最小的那个任务 → /easywork:implement → /easywork:examine
  验证这一个小任务走得通，再铺开到其他任务。
```

### DRY（Don't Repeat Yourself）

> 每一块知识在系统中必须有单一、明确、权威的表示。

```
应用于思维链：
  同一个认知不要重复思考。第一次想清楚一件事，就写下来。
  EasyWork 的知识库就是做这个的——不是存代码，是存"为什么这样设计"的认知。
  下次遇到类似问题，先查知识库，不要重新推导一遍。
```

### 正交性（Orthogonality）

> 一个变化应该只影响一个维度。如果改 A 必须连带改 B、C、D，系统就不正交。

```
应用于你的设计：
  设计模块时问自己：如果我需要改这个配置/接口/数据结构，
  有几个地方需要同步修改？如果多于一个，考虑重构。
```

## 📋 AI 时代程序员的新习惯

### 习惯 1：任务开始时——写"前置思考笔记"

```
不要只在脑子里想。写下来——哪怕只是一段私有的笔记。

格式（30 秒版本）：
---
WHY:   [一句话，为什么做]
WHAT:  [一句话，做什么]
HOW:   [一句话，怎么做 + 备选]
DONE:  [一句话，怎样算完成]
RISK:  [一句话，最坏会怎样]
---

这个格式就是五问的压缩版。30 秒写完，但足够让你在开始前想清楚。
```

### 习惯 2：任务执行中——"偏差日志"

```
不是正规文档，就是随手记的一句话：

"14:30 - 发现 API 返回格式跟文档不一致，比文档多了 payment_method 字段。已做兼容处理。"
"15:10 - 原来估算的 2 小时不够，因为需要顺便修复一个旧 bug。不影响本次范围。"
"16:00 - 跟 @张三 确认了，字段是 V2.1 版本新增的，文档没更新。后续不需要清理。"

价值：
- 复盘时有据可查（偏差不是感觉，是事实）
- 别人接手时知道发生过什么
- 写周报时有实质内容
```

### 习惯 3：任务完成后——"一件事一笔记"

```
做完一个任务，沉淀一条知识：

格式：
---
标题：[一句话，这件事的本质]
日期：[今天]
标签：[技术域] [EasyWork技能]
结论：[核心认知，不超过 3 句话]
如果再来一次：[会有什么不同]
---

一行不超过 3 分钟，但三个月后价值巨大。
```

### 习惯 4：碰到复杂问题——先画再干

```
在开始写代码之前，先用一张图回答三个问题：
1. 现在是什么样？（As-Is）
2. 要变成什么样？（To-Be）
3. 两者之间的 delta 是什么？

EasyWork 工具：/easywork:diagram（多引擎图表）
```

### 习惯 5：做技术决策——写 mini-ADR

```
不是正规的架构文档格式，就是一段话：

"[日期] 选择用 Redis 而不是本地缓存，因为（1）多实例共享（2）有 TTL 需求。
 代价：增加运维复杂度（一个额外服务）。如果不满足这两条中的任一条，
 应该考虑降级回本地缓存。"
```

## 💎 核心原则总结

```
1. 永远自己做五问——AI 不能替你定义问题
2. 输入质量决定输出质量——你描述得越清晰，AI 写得越好
3. Review 用倒置法——假设代码有 bug，找出来
4. 执行用 OODA——越快完成一个循环，越早发现偏差
5. 做完必沉淀——三分钟写一条笔记，三个月后有积累
6. 你是机长——AI 是副驾驶，方向盘在你手里
```

> **最后一条，最重要的**：AI 时代的程序员，核心竞争力不是你用过多少模型、配过多少 MCP，而是**你头脑里的思维框架——知道要构建什么，以及为什么。**
