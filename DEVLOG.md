## v3.1 — 场景画布（2026-07-01）

### 问题
25 个技能是孤岛——一次只能执行一个，或者走固定流程。用户说"只有动作，没有目标"，缺少场景上下文。

### 解决
- **🎨 可视化拖拽编辑器**：`/easywork:canvas` 在浏览器中打开，LiteGraph.js（ComfyUI 同款）渲染，自由拖拽 27 个技能节点、连线、配置参数、保存/加载
- **▶️ 场景执行引擎**：加载场景 YAML → DAG 拓扑排序 → 按依赖顺序/并行执行技能 → HITL 交互暂停点
- **7 个预置场景**：接手遗留项目、线上故障定位、性能排查、技术选型、安全审计、发布检查、新领域学习
- **场景↔Pipeline↔Meta 三层集成**：场景 DAG 可喂入 Pipeline 执行，也可作为 Meta 自治扩散的种子图
- **28 条命令**（+2）：`/easywork:scenario`、`/easywork:canvas`

## v3.0 — 配置驱动 + 多角色通用（2026-06-29）

### 问题
EasyWork 的知识库分类死绑定三个领域（联调/开发/季度O），换了团队就用不了。大量硬编码的领域引用散落在 Python 源码和文档中。

### 解决
- **6 大领域动态定义**：后端/前端/数据/DevOps/移动端/设计，在 `.easywork/config.json` 中自由增删改
- **知识库可选**：`"enabled": false` 即可关闭整个知识库系统
- **消除所有硬编码**：从 Python 代码（hooks、MCP Server）到 Markdown 文档，全部改为 config-driven
- **28 个 SKILL.md 版本统一**：从混用 v1.0/v2.0/v2.12 → 全部 v3.0
- **修复关键 Bug**：`knowledge-flush.py` 中重复的 `classify_domain` 函数（旧版引用了不存在的变量，会导致运行时 NameError）
- **命令文件标准化**：`disable-model-invocation: true` → `false`，28 条命令统一 YAML frontmatter 格式

## v2.11 — 文档保真闸门（2026-06-27）

### 问题
Agent 写文档时会"偷偷"退化——字数越来越少、证据越来越薄、历史版本被覆盖。用户说"上次写得好好的，这次怎么缩水了？"

### 解决
- **文档保真闸门**（铁律#29）：7 项检查——字数退化/证据密度下降/版本丢失/节点缩减/无基准覆写/round_report 冒充 engineering_active/写入模式错配。命中即阻断
- **禁止无基准覆写**（6 条规则）：已有文档默认局部更新、Overwrite 必须带检查清单、Quick Fix 禁止覆写
- **写入模式三级分类**：quick_fix（仅追加）/ normal（结构化合并，默认）/ full_archive（三合法场景 + checklist）
- **文档作用域物理隔离**：round_report（轮次摘要）vs engineering_active（工程档案），不可互混

## v2.10 — 文档拓扑闸门（2026-06-27）

### 问题
Mode B 持续维护模式下，Agent 把内容追加到文件末尾而不是合并到对应节点，导致同样的内容在多个地方出现、版本索引和实际内容不一致。

### 解决
- **文档拓扑闸门**（铁律#28）：7 项拓扑检查——重复节点/模式混用/索引缺失/内容未入节点/孤儿更新记录/历史混淆/索引不一致
- **双模文档结构**：Mode A（审计日志）vs Mode B（持续维护，默认）。Mode B 每个流程节点只出现一次，节点内按版本分层
- **结构化合并**（替代尾部追加）：Fetch→Parse→Pre-check→Distribute→Index→Mark

## v2.9 — 反水文闸门 + ETR 证据链（2026-06-27）

### 问题
Agent 产出大量空泛结论——"功能正常""测试通过""提升用户体验"——没有证据、没有位置、没有风险分析。

### 解决
- **反水文闸门**（铁律#27）：6 类水文信号检测。命中任一即阻断写入
- **ETR 证据标准**：每个关键结论必须包含 Evidence（证据）+ Thinking（推理链）+ Risk（未覆盖风险/副作用/假设）
- **MCR→MCR+**：从"最少有什么"升级为"质量有多好"，每步增加反例表和合格例
- **质量评分**：100 分制，detailed <80 分禁止写入

## v2.8 — 质量门禁循环（2026-06-27）

### 核心新增
- **CODE↔REVIEW 门禁循环**：REVIEW 发现阻断问题 → 必须回退 CODE 修复 → 重新 REVIEW 通过才放行。最多 3 轮
- **READ 需求确认**：编码前必须用自己的话重述需求理解 + 列出澄清问题 + 获用户确认（铁律#24）
- **交互式应用增强**：CLI/TUI/GUI/Web/游戏 → 额外验证首屏/输入反馈/退出路径/stdin 边界/ANSI 兼容
- **回退历史追踪**：状态快照新增 `rollback_history` 字段

## v2.7 — 报告深度分层（2026-06-27）

### 核心新增
- **报告三级深度**：brief / standard / detailed。任务类型自动决定默认值
- **MCR 最小内容要求**：detailed 模式每步骤有硬性最低产出，不满足则阻断写入
- **深度自动升级**：发现安全问题 / 测试根因复杂 / 改动超预估 2 倍 → 自动升级为 detailed
- **流式增量写入**：每步骤完成后立即追加到文档，禁止"浓缩版"替代
