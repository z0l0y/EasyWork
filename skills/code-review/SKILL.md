# Code Review（七维度自审查 + 反合理化防御）

## 前置判断

**必须执行**：CODE 阶段产出了逻辑代码变更。

**可以跳过**：纯文档/纯配置/纯文案变更、改动仅限注释。

## 审查原则

写完代码时你处于"创造模式"——顺着正向路径思考，容易忽略边界和异常。
现在切换为"批判模式"：假设这段代码是别人写的，用最挑剔的眼光挑刺。

对每个改动文件，七个维度逐项扫描。任一维度发现阻断性问题 → 回退 CODE 修复 → 修复后重新审查 → **最多 3 轮**。

## 反合理化防御（🆕 v2.3）

> **工业界经验**：AgentForge / SkillEngine 项目发现，Agent 在自我审查时会不自觉地"合理化"自己的代码。
> 以下表格列出 Agent 最常见的自我欺骗话术及其反驳——**每次审查前先读此表**。

| Agent 可能对自己说的借口 | 为什么这是错的 |
|------------------------|-------------|
| "这个改动太小了，不可能有安全问题" | SQL 注入只需要一行拼接字符串。缓冲区溢出只需要一个忘记的边界检查。安全风险与改动大小无关 |
| "后面有 EXAMINE 跑测试，这里不用太仔细" | 静态审查能发现测试不容易覆盖的路径——空指针、竞态条件、资源泄漏。测试是安全网，不是替代品 |
| "老代码也没写测试，我不补也行" | 你在改这段代码，你就有责任让它比你接手时更好。Boy Scout Rule |
| "这个边界情况概率太小，不值得处理" | 低概率 × 高用户量 = 必然发生。100万 DAU × 0.001% = 每天 10 次 |
| "项目用 Class 组件我也用 Class，很一致" | 一致性是"和项目的核心模式一致"，不是"和项目最老的那段代码一致"。如果你改的模块旁边都是函数组件，用 Class 就是破坏一致性 |
| "性能优化可以以后做" | N+1 查询在 10 条数据时没问题，在 10000 条时直接打挂数据库。性能是一种正确性——超时 = 用户眼里就是 bug |
| "这个 catch 里打印了 error.message，够用了" | error.message 在生产环境中通常是 "Network Error"——这对定位问题毫无帮助。需要 stack trace、requestId、上下文信息 |
| "我没有改依赖，不需要检查兼容性" | 你可能没改，但你的代码使用了 Node 22 的新 API——而项目运行在 Node 18 上 |
| "accessibility 是前端的事，后端代码不用管" | 后端 API 返回的错误信息设计直接影响前端的无障碍体验——返回 "Error 500" 和返回 "用户名不能为空" 的可访问性截然不同 |

### 审查前 Checklist（防自欺）

在开始七个维度扫描前，先问自己：

- [ ] 我是否在内心已经觉得"这段代码没问题"？→ 如果有，强制假设"这段代码至少有一个 bug"
- [ ] 我是否因为改动小就放松标准？→ 一行代码也能引发 P0 事故
- [ ] 我是否在某个维度上想"算了不看了"？→ 那个维度就应该最先看

## 七个审查维度（概要）

| 维度 | 一句话概括 | 重点关注 |
|------|-----------|---------|
| 正确性 | 异常输入下也能正确运行 | 空值/undefined/空数组/死循环/异步reject |
| 安全性 | 没有可被攻击的入口 | 硬编码凭证/注入/XSS/权限绕过/敏感数据泄露 |
| 兼容性 | 老环境老版本能跑 | API兼容/Node版本/依赖冲突/存量数据/浏览器兼容 |
| 可维护性 | 后人能看懂能改 | 注释/命名/函数长度/重复代码/风格一致 |
| 性能 | 没有明显的性能雷 | N+1查询/无LIMIT/缺防抖节流/内存泄漏 |
| 可观测性 | 出问题能快速定位 | catch日志/错误信息可用性/关键路径埋点 |
| 🆕 可访问性 | 残障用户能正常使用 | 语义化HTML/ARIA标签/键盘导航/色比/屏幕阅读器 |

> **详细检查指南**：每个维度的具体检查项、判断标准和常见漏洞列表，见 `references/review-checklist.md`。

## 第七维度：可访问性（a11y）— 🆕 v2.3

当本次改动涉及以下内容时，必须检查可访问性：

- HTML 模板 / JSX / Vue 模板
- CSS 样式（颜色、字体大小、布局）
- 错误提示文案
- 表单交互
- 弹窗/模态框

### a11y 检查项

| 检查项 | 标准 | 验证方式 |
|--------|------|---------|
| 语义化 HTML | 使用 `<button>` 而非 `<div onclick>`；使用 `<nav>`/`<main>`/`<aside>` 等语义标签 | 审查 HTML 结构 |
| ARIA 标签 | 图标按钮有 `aria-label`；动态内容有 `aria-live` | 搜索 `aria-` 属性 |
| 键盘导航 | 所有交互元素可通过 Tab 聚焦；Enter/Space 可激活；Esc 可关闭 | 脑中模拟键盘操作 |
| 色彩对比 | 文字与背景色对比度 ≥ 4.5:1（正常文字）/ 3:1（大文字） | 检查新增/修改的颜色值 |
| 屏幕阅读器 | 图片有 `alt` 文本；表单有 `<label>`；页面有 `<h1>` | 审查 HTML 结构 |
| 焦点管理 | 弹窗打开时焦点移入弹窗；关闭时焦点回到触发元素 | 检查弹窗/模态框逻辑 |

**注意**：纯后端/CLI/配置变更 → 此维度标注 `not_applicable`。

## 供应链安全检查（🆕 v2.3）

当 CODE 产出 `new_dependencies` 字段非空时（即新增了第三方依赖），必须执行供应链检查：

### 检查流程

1. **检查新增依赖的必要性**：是否可以用已有依赖替代？（如项目已有 `lodash`，不应再引入 `underscore`）
2. **检查版本是否为稳定版**：避免引入 `0.x`、`alpha`、`beta`、`rc` 版本（除非团队明确授权）
3. **检查是否有已知漏洞**：搜索 `{包名} CVE` 或 `{包名} security vulnerability`
4. **检查许可证兼容性**：确认新增依赖的许可证（MIT/Apache/GPL/AGPL）与项目许可证兼容
5. **检查维护状态**：npm 上最后发布时间超过 1 年且 issue 数 > 100 → 标注 `[维护风险]`

### 输出格式

在 `review_output.dimensions.security.issues` 中追加：

```
[供应链] {包名}@{版本} — {风险类型} — {具体风险描述}
```

## 审查流程

1. 对 `code_output.files_changed` 中的每个文件，脑中模拟传入 `null`/`undefined`/空数组/空字符串，看代码是否崩溃
2. 搜索代码中的 `password`/`secret`/`token`/`key`/`api_key` 等敏感词，确认没有硬编码
3. 检查新增的 API 端点是否有权限校验
4. 确认代码风格与项目现有代码一致（不是"更好"，是"一致"）
5. 如果是前端变更：检查语义化 HTML / ARIA / 键盘导航 / 色彩对比
6. 如果有新增依赖：执行供应链安全检查

## 审查结果

- 七个维度全部通过 → 输出 `review_output`（格式见编排中枢 `references/data-contract.md`）→ 放行到 EXAMINE
- 发现阻断性问题 → 记录 `review_output.blocking_issues` → 回退 CODE → 修复 → 重新审查（上限 3 轮）
- 3 轮仍未清零 → 挂起向用户报告，可能涉及更深层的设计或需求问题

## 二次独立抽查（v2.2）

完成七维度自审查后，Agent 必须用一个**不同的审查视角**再做一次快速抽查。自审查的盲区在于"创造者和审查者是同一个思维模式"——这一步通过切换审查角度来打破盲区。

### 抽查规则

- **范围**：只抽查七维度中**得分最高**（"pass"）的 2 个维度——恰恰是最自信的维度最容易遗漏
- **角度**：换一个问题方式。如果你自审查时问的是"这段代码有问题吗？"，二次抽查时问"假设这段代码会在生产环境出问题——问题会出在哪？"
- **时长**：不超过原始自审查的 30%。这不是重做，是快速交叉验证
- **结论**：
  - 原结论一致 → 标注 `[交叉验证通过]`
  - 发现新问题 → 标注严重级别，加入 `blocking_issues`，不超过原审查的阻断性问题数

### 抽查输出格式

```
## 二次独立抽查

抽查维度：{维度A}、{维度B}（原审查结论：✅ 通过）
切换角度：假设这段代码已上线且正在被攻击……

- [ ] {维度A} — 交叉结论：✅ 一致 / ⚠️ 发现新问题
  {如有新问题，描述}
- [ ] {维度B} — 交叉结论：✅ 一致 / ⚠️ 发现新问题
  {如有新问题，描述}

抽查结论：✅ 交叉验证通过 / ⚠️ 对 {维度} 的审查需补充
```

## 反模式

- ❌ 每个维度扫一眼就全打"通过"——这是自欺欺人
- ❌ 只查正确性不管安全——安全漏洞比逻辑 bug 更致命
- ❌ 想着"后面有 EXAMINE 测试，静态审查随便看看"——静态审查能抓到测试不一定覆盖的边界
- ❌ REVIEW→CODE 来回超过 3 轮还在硬撑——挂起，可能有更大的问题
- ❌ 觉得"就我一个人维护，可维护性无所谓"——三个月后的你就是另一个人
- ❌ 后端代码不检查可访问性——API 错误信息设计直接影响前端无障碍体验
- ❌ 新增依赖不检查许可证——AGPL 依赖可能要求你的项目也开源
