# 代码审查详细检查项参考

本文档对七维度审查清单中的每一项提供更详细的检查指导和判断标准，供 Agent 在审查时参考。

## 正确性检查指导

### 空值与边界检查
- 对所有函数入参逐一检查：如果调用方传入 null / undefined / 空字符串 / 空数组 / 0，代码是否会崩溃？
- 对异步请求的返回值：检查是否在每个 .then() / await 之后都做了空值判断
- 对数组操作（map / filter / reduce）：数组为空时是否会产生非预期结果？
- 对数字运算：除数是否可能为 0？

### 循环与递归
- for / while 循环：终止条件是否在所有情况下都能达到？
- 递归函数：是否有最大深度限制？是否可能栈溢出？
- 事件监听和定时器：是否在适当的生命周期节点（组件卸载、页面关闭）清除？

### 类型安全
- TypeScript 项目中：是否使用 any 类型规避了类型检查？
- 类型断言（as / !）的使用是否安全？是否有运行时类型不一致的风险？

### 条件分支完整性
- if-else 链是否处理了 else 分支（默认情况）？
- switch-case 是否有 default 分支？
- 多层嵌套的条件判断是否可以提前 return 来简化？

### 异步处理
- Promise 是否同时处理了 resolve 和 reject？
- async/await 是否正确使用了 try-catch？
- 是否存在未 await 的 Promise（可能导致竞态条件）？

## 安全性检查指导

### 硬编码凭证
- 扫描代码中是否出现：password、secret、token、api_key、private_key 等关键词
- 检查这些值是否通过环境变量或配置中心获取
- 注意：测试文件中的假凭证也需要标记（虽然不是安全风险，但是不良实践）

### 注入攻击
- SQL 拼接：是否使用了字符串模板拼接 SQL？应使用参数化查询或 ORM
- 命令执行：child_process.exec / os.system 的参数中是否拼接了用户输入？
- URL 拼接：是否对用户输入的 URL 参数做了编码？

### XSS 防护
- 前端渲染：是否使用了 innerHTML / v-html / dangerouslySetInnerHTML？
- 如果是，渲染的内容是否经过了 HTML 转义（如使用 DOMPurify）？
- 用户输入展示：是否对特殊字符（< > " ' &）做了转义？

### 敏感数据
- 日志中是否打印了用户手机号、身份证、密码、银行卡号等敏感信息？
- API 返回中是否包含了不必要的内部字段（如数据库 ID、内部状态码）？
- 错误信息是否向用户暴露了过多的系统内部信息（如 SQL 语句、文件路径）？

### 权限校验
- 新增的接口是否验证了用户登录状态？
- 是否验证了用户的操作权限（如普通用户不能访问管理接口）？
- 是否存在越权风险（用户 A 能否通过修改 ID 参数访问用户 B 的数据）？

## 兼容性检查指导

### 运行环境兼容
- 查看项目的 browserslist 或 engines 配置，确认目标环境
- 检查使用的 API 在目标环境中是否可用（可参考 MDN 的浏览器兼容性表格）
- 常见低版本不兼容的 JS API：Array.flat()、Object.fromEntries()、String.matchAll()、Promise.allSettled()、可选链 ?.、空值合并 ??

### 依赖兼容
- 新增的依赖包的版本号是否与项目其他依赖冲突？
- 是否引入了与现有依赖功能重复的包（如已有 dayjs 又引入 moment）？
- 依赖包的 license 是否与项目兼容？

### API 向后兼容
- 如果修改了函数签名（参数增减、参数类型变更），是否所有调用方都已更新？
- 如果修改了 API 返回值结构，前端是否做了对应的适配？
- 对于对外暴露的公共 API，是否考虑过版本化（/v1/api vs /v2/api）？

## 可维护性检查指导

### 注释规范
- 注释是否使用中文？
- 注释是否解释了"为什么"而不仅仅是"做了什么"？
- 复杂算法或 hacky 的代码是否有充分的注释说明？
- 是否有没有意义的注释（如 `// 设置 name 为 'John'` 对应代码 `name = 'John'`）？

### 命名
- 变量名是否表达了其用途？避免 a, b, tmp, data, info, item, obj 等无意义命名
- 函数名是否使用动词开头（get, set, fetch, create, update, delete, handle, validate）？
- 布尔值变量是否使用 is/has/can/should 开头？
- 常量是否使用 UPPER_SNAKE_CASE？

### 函数设计
- 单个函数是否超过 50 行？如果是，是否应该拆分？
- 函数是否做了太多件事（单一职责原则）？
- 函数参数是否超过 4 个？如果是，是否应该封装为对象参数？

### 避免重复
- 是否有复制粘贴的代码块（相似度 > 80%）？
- 项目是否已有现成的工具函数可以替代？

## 性能检查指导

### 数据库与 IO
- 循环内是否有数据库查询？应该改用批量查询或 IN 查询
- 数据库查询是否使用了合适的索引？
- 是否有不必要的大字段查询（SELECT * 但只需要其中 2 个字段）？

### 前端性能
- 高频事件（scroll, resize, input, mousemove）是否做了节流或防抖？
- 大列表渲染是否使用了虚拟滚动？
- 图片是否做了懒加载？
- 是否存在不必要的组件重渲染（React 的 memo / Vue 的 computed 是否合理使用）？

### 内存
- 是否存在内存泄漏风险（闭包引用大对象、未清理的定时器、未解绑的事件监听）？
- 一次性加载的数据量是否可能过大？

## 可观测性检查指导

### 日志
- 所有 catch 分支是否有日志输出？
- 日志是否包含足够的上下文（操作名称、关键参数、用户标识）？
- 日志级别是否合理（info / warn / error）？

### 错误处理
- 用户可见的错误提示是否友好且有用（不是"系统错误"而是"网络连接失败，请检查网络后重试"）？
- 是否可以区分"可恢复的错误"和"不可恢复的错误"并做不同处理？

### 监控
- 核心业务操作（支付、登录、关键数据处理）是否有打点？
- 是否有健康检查端点供监控系统使用（如 /health 或 /ping）？

## 可访问性检查指导（🆕 v2.3）

### 语义化 HTML
- 是否使用了 `<button>` 而非 `<div onclick>` 实现按钮？
- 是否使用了 `<nav>`、`<main>`、`<aside>`、`<header>`、`<footer>` 等语义标签？
- 标题层级是否合理（h1 → h2 → h3 不跳级）？

### ARIA 标签
- 纯图标按钮是否有 `aria-label` 描述其功能？
- 动态更新内容区域是否有 `aria-live` 属性通知屏幕阅读器？
- 自定义 UI 组件（如 tabs、accordion、modal）是否有正确的 ARIA role 和状态属性？

### 键盘导航
- 所有交互元素（按钮、链接、表单控件）是否可通过 Tab 键聚焦？
- 是否可以通过 Enter/Space 激活按钮，Esc 关闭弹窗？
- 焦点顺序是否合理（DOM 顺序 = 视觉顺序）？

### 色彩与对比度
- 文字与背景色对比度是否 ≥ 4.5:1（正常文字）/ 3:1（大文字 ≥18px）？
- 是否仅用颜色传达信息（如仅用红色表示错误，没有图标/文字辅助）？

### 屏幕阅读器兼容
- 所有图片是否有有意义的 `alt` 文本（装饰性图片用 `alt=""`）？
- 表单输入框是否有关联的 `<label>`（非仅 placeholder）？
- 页面是否有且仅有一个 `<h1>`？

### 焦点管理
- 弹窗/模态框打开时焦点是否移入弹窗内部？
- 弹窗关闭时焦点是否回到触发元素？
- 路由切换后焦点是否移到页面主要内容？
