# 确认问题维度参考

本文档提供 ASK 阶段六个确认维度的详细参考，帮助 Agent 生成有针对性、有价值的确认问题。

## 维度一：需求与验收覆盖

### 目标
确认本次改动是否完整覆盖了需求中的所有验收标准，是否有遗漏。

### Agent 需要回顾的材料
- READ 阶段输出的需求说明和验收标准
- CODE 阶段的变更清单
- SUM 阶段的总结报告

### 提问角度
- 需求中提到的每个功能点是否都有对应的代码实现？
- 验收标准是否全部通过（或有不适用的情况）？
- 是否有需求中隐含但未明确写出的功能点被遗漏了？
- 用户对最终交付物是否符合预期？

### 示例问题
> "本次改动为登录接口增加了短信验证码登录方式，实现了需求中的第 3 项验收标准。但我注意到需求文档第 2 页提到了'验证码有效期 5 分钟且支持重发'——重发逻辑已实现，但验证码过期的清理机制目前没有做。请确认：验证码过期清理是否需要在本次实现？还是可以放到后续迭代？"

### 警告
不要问泛泛的问题如"你觉得需求都满足了吗？"——用户很难回答这种问题。要基于具体的代码变更和具体的需求条目来提问。

## 维度二：行为变化与兼容性

### 目标
确认用户已知悉本次改动带来的所有行为变化，特别是可能影响下游调用方的变化。

### Agent 需要回顾的材料
- CODE 阶段的变更清单和影响面评估
- REVIEW 阶段的兼容性审查结果

### 提问角度
- 修改的 API 签名是否会影响所有调用方？（前端、其他服务、外部系统）
- 修改的组件行为是否会影响所有使用该组件的页面？
- 是否有需要通知的关联团队？
- 是否存在"静默的行为变化"（没有编译错误但运行时行为发生了变化）？

### 示例问题
> "本次将 UserService.getUserInfo() 的返回值从 `{ name, email }` 扩展为 `{ name, email, phone, avatar }`。这个修改是向后兼容的（旧字段保持不变），所有现有调用方不会受影响。但调用方如果需要使用新增的 phone 和 avatar 字段需要主动适配。请确认：是否需要我列出所有调用 getUserInfo() 的位置，以便你通知相关方？"

### 警告
最容易遗漏的是"静默行为变化"——API 签名没变但内部逻辑变了（如默认排序变了、错误处理方式变了、超时时间变了）。这些必须在此维度明确提问。

## 维度三：数据影响与回滚

### 目标
确认所有数据层面的变更（schema 变更、数据迁移、新字段等）都有明确的回滚方案。

### Agent 需要回顾的材料
- CODE 阶段中涉及数据库/存储的变更
- GIT 阶段中配置与构建维度的拆分

### 提问角度
- 数据库表结构变更是否有 migration 脚本？
- 新增字段是否有默认值以兼容存量数据？
- 数据迁移是否可逆？
- 如果上线后发现问题需要回滚，数据回滚怎么做？
- 回滚操作是否经过测试或预演？

### 示例问题
> "本次在 orders 表新增了 `payment_channel` 字段（VARCHAR(32)，默认值 'wechat'）。Migration 脚本见 `migrations/20260619_add_payment_channel.sql`。回滚时只需执行 `ALTER TABLE orders DROP COLUMN payment_channel`，但需注意：回滚后所有订单的支付渠道信息将丢失，报表系统需要适配。请确认：(1) 这个回滚风险是否可接受？(2) 是否需要在回滚前通知数据团队备份该字段数据？"

### 警告
对于数据变更，没有"小改动"——加一个字段和改一个表结构都是同等严肃的。尤其要注意：包含默认值的新字段虽然不破坏现有功能，但回滚时会丢失该字段的数据。

## 维度四：安全与权限

### 目标
确认本次改动不会引入安全漏洞或权限绕过路径。

### Agent 需要回顾的材料
- REVIEW 阶段的安全维度审查结果
- 涉及认证、授权、数据访问的代码变更

### 提问角度
- 新增的接口是否做了登录验证？
- 不同角色的用户是否能访问不该他们访问的数据？
- 是否有可能通过修改请求参数来访问他人的数据（越权风险）？
- 敏感数据（密码、手机号、身份证号）是否在日志或 API 返回值中暴露？

### 示例问题
> "本次新增的 GET /api/orders/:id/detail 接口使用了 `req.user.id` 来做权限控制（用户只能查看自己的订单）。但我注意到接口没有校验 `:id` 参数是否为数字——虽然 ORM 会做参数化查询不会产生 SQL 注入，但建议加上参数格式校验以防御恶意扫描。请确认：是否需要我在本次加固参数校验？还是现有 ORM 防护已经足够？"

### 警告
安全问题的确认不是推卸责任，而是因为安全边界往往有业务特殊性——什么角色能看什么数据、什么操作需要二次验证——这些只有业务方才能最终决定。

## 维度五：测试覆盖与边界用例

### 目标
让用户了解测试的覆盖范围和已知的未覆盖场景，做出知情的接受/拒绝决定。

### Agent 需要回顾的材料
- EXAMINE 阶段的质量验证报告
- 新增的测试用例清单和未覆盖场景说明

### 提问角度
- 当前测试覆盖了哪些场景？
- 哪些场景因为技术限制未能覆盖？
- 未覆盖的场景是否需要用户手动验证？
- 用户是否接受当前的测试覆盖度？

### 示例问题
> "EXAMINE 阶段运行了 8 个测试用例并全部通过，覆盖了：正常登录流程、密码错误处理、账号锁定、token 刷新。以下 2 个场景未能自动化覆盖：(1) 短信验证码登录——因为需要真实的短信网关，Mock 环境下无法完全模拟网络超时场景；(2) 并发登录——因为本地测试环境不支持压力测试。请确认：是否需要你手动验证这两个场景后再合并代码？"

### 警告
不要遮掩测试覆盖的不足。诚实地告诉用户哪些场景没测到、为什么没测到、可能的风险是什么。隐瞒比不测试更危险。

## 维度六：上线与监控

### 目标
确认有明确的上线计划和出问题时的应急方案。

### Agent 需要回顾的材料
- 全部 8 步的产出
- GIT 阶段的分批提交方案（关联到灰度发布策略）

### 提问角度
- 建议的上线步骤是什么？一次性全量还是分批灰度？
- 上线后需要观察哪些指标？
- 告警阈值是多少？
- 出问题时的回滚命令是什么？
- 应急联系人是谁？

### 示例问题
> "建议上线步骤：(1) 先在 staging 环境验证 30 分钟，确认无异常；(2) 灰度 5% 流量观察 1 小时，关注登录成功率（告警阈值：<95%）和 P99 延迟（告警阈值：>1s）；(3) 全量发布。回滚命令：`git revert <commit-hash> && npm run deploy`。请确认：这个上线方案是否可行？是否需要调整灰度比例或观察时长？"

### 警告
这个维度的确认最容易被用户跳过（"上线的事我们运维负责"），但恰恰是上线和监控的缺失导致了最多的事故。如果用户表示"不用确认"，至少坚持确认回滚方案是否明确。

---

## 问题质量检查

Agent 在生成每个确认问题后，应自查：
- [ ] 问题是否基于本次改动的具体内容（而非泛泛的模板套话）？
- [ ] 问题是否说明了"为什么问这个"（关联到具体的风险点）？
- [ ] 问题是否给出了"如果答案为否"的下一步动作？
- [ ] 问题的答案是否能帮助用户做出更好的决策（而不是走过场）？
- [ ] 六个维度是否每个至少有一个具体问题？
