# 人工确认 6 维度检查清单

> 用于 ask-change-questions 技能 · 每条必须显式确认 · 不可跳过

## 1. 功能正确性 ✅
- [ ] 改动是否实现了需求中描述的全部功能？
- [ ] 是否有遗漏的边缘情况？（空输入、超长输入、特殊字符、并发）
- [ ] 是否与现有功能产生冲突？

## 2. 代码质量 ✅
- [ ] 是否通过 code-review 的全部闸门（正确性/安全性/兼容性/可维护性/性能/可观测性/可访问性）？
- [ ] 是否有硬编码、魔法数字、未处理异常？
- [ ] 命名是否清晰、符合项目规范？

## 3. 测试充分性 ✅
- [ ] 是否通过 examine-quality 的全部闸门？
- [ ] 是否有回归测试？覆盖率是否达标？
- [ ] 是否有失败用例？失败原因是否已分析？

## 4. 文档完整性 ✅
- [ ] 是否有变更说明（CHANGELOG / commit message）？
- [ ] API 变更是否有文档更新？
- [ ] 是否有迁移指南（如有破坏性变更）？

## 5. 部署安全性 ✅
- [ ] 是否有回滚方案？
- [ ] 是否需要数据库迁移？迁移是否可逆？
- [ ] 是否涉及敏感数据/权限变更？

## 6. 团队对齐 ✅
- [ ] 相关人员是否已知晓变更？
- [ ] Code Review 是否已完成且通过？
- [ ] 是否有阻塞项（依赖其他团队/服务）？

---

> **铁律**: 以上 6 维度任何一项未确认 → 不允许进入部署流程。
