# Examine Quality（测试执行）

## 前置判断

**必须执行**：CODE 阶段产生了逻辑代码变更，且项目/语言具备可运行的测试环境。

**可以跳过**：纯文档/配置/文案变更、项目完全没有测试框架且用户同意跳过、纯理解任务。

**强烈不建议跳过**：Bug 修复、功能开发、重构——没有测试验证的改动就是盲飞。

## 执行流程

### 第 1 步：搜索测试命令

搜索以下位置，找到可执行的测试入口：

- `package.json` → `scripts` 中的 `test`/`test:unit`/`test:ci`
- `Makefile` → `test` 目标
- CI 配置 → `.github/workflows/*.yml`、`.gitlab-ci.yml`
- 测试文件目录 → `__tests__/`、`tests/`、`*.test.*`、`*.spec.*`

**关键**：不用跑全量（太慢）。找到与本次改动最相关的测试子集命令。如 Jest 的 `--testPathPattern`。

### 第 2 步：无测试则补充

如果本次改动涉及核心逻辑但没有任何测试覆盖，**必须补**。补充原则：

- **最小覆盖**：正常输入（Happy Path）+ 关键边界（空值/极值）+ Bug 复现场景（如果是修 Bug）
- **用项目现有框架**：不要引入新的测试框架
- **放对位置**：遵循项目的测试目录约定

**什么情况可以不补**：改动纯属配置/CSS/文案。

### 第 3 步：运行测试

在终端执行测试命令，完整保存输出。关注以下信号：

- ✅ All tests passed / 绿色 → 通过
- ❌ FAILED / 红色 → 进入第 4 步
- ⚠️ 卡住超过 5 分钟无输出 → 可能死循环，终止并向用户报告

### 第 4 步：失败则修复，重跑到全绿

分析失败原因 → 区分"代码 bug"还是"测试写错了" → 修复 → 重跑。

```
循环：
  跑测试 → 有失败？
    → 是：分析失败 → 修代码/修测试 → 回到跑测试
    → 否：✅ 全绿通过，记录结果
```

**循环超过 3 次还未全绿** → 挂起向用户报告，可能涉及更深层的设计问题。

> **详细检查清单**：测试命令搜索、补充测试质量要求、失败分析流程、常见陷阱，见 `references/quality-checklist.md`。

## 测试环境损坏时

如果测试框架本身跑不起来（Jest 版本冲突、依赖缺失、配置损坏），停止并报告：

```
【EXAMINE 阻断 - 测试环境不可用】
问题：{具体描述——如"Jest 27 与 Node 20 不兼容，运行时报 worker 错误"}
这与本次代码修改无关，属于项目基础设施问题。
选项：
  A) 跳过测试，仅凭 REVIEW 静态审查继续
  B) 先修复测试环境，再继续工作流
  C) 你手动运行测试，我基于你提供的结果继续
→ 请选择。
```

## 🆕 v2.8 交互式应用真实体验验证（铁律#26）

当任务涉及 CLI / TUI / GUI / Web 前端 / 游戏 / 交互式输入输出时，EXAMINE 必须在 standard/detailed 模式下**额外**覆盖以下验证。纯后端 API / 库 / 脚本 / 配置变更标注 `[交互式应用 EXAMINE 补充: N/A]`。

### 触发判断

任务分类时 Agent 检查改动是否涉及以下任一类型：
- CLI 工具（命令行交互、参数解析、stdin 读取）
- TUI 应用（终端 UI，如 ncurses/bubbletea/rich 等）
- GUI 桌面应用（Electron/Qt/WPF/SwiftUI 等）
- Web 前端（HTML/CSS/JS/React/Vue 等，涉及用户可见界面）
- 游戏（Unity/Unreal/WebGame 等）
- 交互式输入输出（readline/prompt/交互式命令行等）

### 验证清单

在 standard/detailed 模式下，执行以下验证并产出 `interactive_ux_validation` 字段：

| # | 验证项 | 验证方法 | 通过标准 |
|---|--------|---------|---------|
| 1 | **首屏稳定性** | 冷启动 ≥3 次，观察首屏渲染 | 无闪烁、无白屏 >1s、无未加载占位符 |
| 2 | **无输入时行为** | 启动后等待 10s，不做任何操作 | 界面安静、无异常日志刷屏（>10行/s）、CPU 空闲 <10% |
| 3 | **输入反馈** | 执行主要操作路径，观察反馈延迟 | 每次操作 200ms 内有可见反馈（按钮态/加载态/结果展示） |
| 4 | **退出路径** | 尝试所有退出方式（Esc/关闭按钮/Ctrl+C/`:q`/返回键） | 所有路径可退出，退出后无进程残留，数据不丢失 |
| 5 | **输出/渲染频率** | 如有自动刷新，运行 30s 观察输出量 | 刷新频率 ≤10Hz，输出量可控（分页/限流），肉眼可读 |
| 6 | **环境一致性** | 在主要运行环境各验证一次 | 行为一致或已知差异已记录 |

### CLI/TUI 额外验证

| # | 验证项 | 验证方法 | 通过标准 |
|---|--------|---------|---------|
| 7 | **stdin 关闭** | 运行中 Ctrl+D（Unix）/ Ctrl+Z 后 Enter（Win）关闭 stdin | 程序优雅退出或提示 stdin 不可用，不 crash |
| 8 | **空输入** | 所有 prompt/readline 处直接按 Enter | 有默认值或给出提示，不 crash，不进入异常状态 |
| 9 | **非法输入** | 输入特殊字符（`!@#$%^&*()`）、超长字符串（>1000字符）、emoji、二进制数据 | 有错误提示或过滤，不 crash，不出现乱码 |
| 10 | **退出指令** | 输入 Ctrl+C / `:q` / exit / quit | 程序立即退出，退出码合理（0 或 130），无资源泄漏 |

### ANSI 控制字符兼容性说明

如果应用使用了 ANSI 转义序列（清屏 `\033[2J`、着色 `\033[31m`、光标控制 `\033[H` 等），**必须在报告中说明**：

```
【ANSI 控制字符兼容性】
本应用使用以下 ANSI 控制字符：
- 清屏：\033[2J \033[H
- 着色：\033[31m \033[32m \033[0m
- 光标：\033[?25l \033[?25h

已知兼容性风险：
- VS Code 集成终端（Windows）：清屏序列可能导致输出折叠为一行，需用终端设置中"Terminal > Integrated: Right Click Behavior"调整
- WebStorm 内置终端：不支持光标隐藏 \033[?25l，进度条会残留光标闪烁
- 重定向到文件（> output.txt）：控制字符会被原样写入，导致文件不可读。建议提供 --no-color 选项
- Windows CMD（非 PowerShell）：需 Windows 10 16257+ 才支持 24-bit 颜色
- CI/CD 环境（GitHub Actions/GitLab CI）：TERM=dumb 时大部分 ANSI 序列失效
```

### 自动刷新验证

如果应用存在自动刷新/实时更新/TUI 重绘机制：

```
【自动刷新验证】
刷新机制：{描述——如 setInterval/requestAnimationFrame/stream listener}
刷新频率：{实测频率 Hz}
单次刷新耗时：{实测 ms}

验证结论：
✅ / ⚠️ / ❌
- 刷新频率是否 ≤10Hz（人眼可感知上限）？
- 是否增量更新（只更新变化部分）而非全量重绘？
- 30s 连续运行后输出行数是否可控（<300行）？
- 日志文件增长速度是否合理（<1MB/分钟）？
```

## 🆕 v2.12 第 5 步：测试充分性评估（铁律#34）

运行测试后，必须评估测试是否覆盖了风险。详见 `skills/examine-quality/references/test-adequacy-gate.md`。

**五维覆盖检查**：
- Happy Path：正常输入→预期输出
- Boundary：空值/极值/越界
- Error Path：异常/超时/资源不足
- Concurrency：并发/竞态（如适用）
- Security：注入/越权/敏感数据泄露（如适用）

**Bug修复回归测试强制**：必须新增≥1个回归测试，在老代码上FAIL、新代码上PASS。无回归测试=不可声称修复完成。

**Gate Check**：detailed→HARD GATE（Bug修复无回归测试→阻断；L2+三维未覆盖→阻断或豁免），standard→SOFT GATE

## 🆕 v2.12 第 6 步：环境保真度检查（铁律#32）

L3+交互式应用必须至少一次近真实环境冒烟测试。详见 `skills/examine-quality/references/environment-fidelity-gate.md`。

**检查内容**：环境分类→环境矩阵→近真实环境冒烟测试（≥1端到端场景）→环境差异记录

**Gate Check**：detailed→HARD GATE（L3+交互式应用未执行冒烟测试→阻断），L0-L2或纯后端→跳过标注N/A

## 🆕 v2.9 MCR+ 质量要求

| # | MCR+ 要求 | ❌ 反例（禁止） | ✅ 合格例 |
|---|----------|--------------|---------|
| 1 | 测试命令完整可复制 + 说明为什么这个命令能覆盖本次改动 | "go test 通过" | "`go test -v -run 'TestAuth\|TestToken' ./auth/...` — 只跑 auth 包而非全量(全量12min)" |
| 2 | 失败前/修复后对比（修Bug时）：修复前什么表现→修复后什么表现 | "问题已修复，测试通过" | "修复前：TestScannerEOF→FAIL(panic)。修复后：TestScannerEOF→PASS(退出码0,200ms)" |
| 3 | 测试覆盖矩阵：标注哪些是本次新增的测试 | 只有通过数无矩阵 | "TestLoginHappy(passed)/TestLoginExpired(passed,🆕新增)/TestLoginWeakNet(passed,🆕新增)" |
| 4 | 关键输出摘录作为测试凭据（非"全部通过"占位） | "go vet 通过" | "`=== RUN TestLoginExpired --- PASS: TestLoginExpired (0.32s)`" |

## 反模式

- ❌ 写一个 `expect(true).toBe(true)` 充数——这是自欺欺人
- ❌ 测试只跑 Happy Path，边界完全不测
- ❌ 🆕 v2.12 Bug修复不添加回归测试就声称"修复完成"——违反铁律#34
- ❌ 🆕 v2.12 回归测试在老代码和新代码上都是PASS——没有验证"老代码FAIL"
- ❌ 🆕 v2.12 五维全标"covered"但实际只有Happy Path测试——虚报覆盖率
- ❌ 🆕 v2.12 L3 Web应用只在localhost Chrome测试→声称"已验证"——缺少真实环境冒烟
- ❌ 🆕 v2.12 环境矩阵全是"相同"没有差异分析
- ❌ 看到已有的测试失败（不是本次改动导致的）假装没看到
- ❌ 觉得"改动太小不用测"——一行代码也能引发线上事故
- ❌ 不保存测试输出凭据——后面的 SUM 阶段需要引用
- ❌ v2.8：CLI/TUI 工具不测 stdin 关闭和空输入——用户第一个操作可能就是按 Ctrl+C 或直接回车
- ❌ v2.8：ANSI 控制字符不说明兼容性风险——IDE 控制台和标准终端行为不同是高频踩坑点
- ❌ v2.8：交互式应用只跑单元测试不验证真实体验——首屏闪烁、按钮没反馈、退出不了——这些单测覆盖不了
- ❌ v2.9："go test 通过""go vet 通过"——无命令、无输出、无覆盖矩阵——水文
