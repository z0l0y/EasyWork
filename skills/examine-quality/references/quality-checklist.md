# 质量验证检查清单

本文档提供 EXAMINE 阶段的详细检查指导，确保测试执行过程完整、规范。

## 测试命令搜索检查

### 搜索位置清单
- [ ] `package.json` 的 `scripts` 字段 — 最常见位置
- [ ] `Makefile` / `Makefile.in` — C/C++/Go 项目
- [ ] `build.gradle` / `pom.xml` — Java 项目
- [ ] `Cargo.toml` — Rust 项目
- [ ] `.github/workflows/*.yml` — GitHub Actions 中的测试命令
- [ ] `.gitlab-ci.yml` — GitLab CI 中的测试命令
- [ ] `Jenkinsfile` — Jenkins 中的测试命令
- [ ] `docker-compose.yml` 中的测试容器

### 搜索技巧
- 使用 Grep 搜索关键词：`test`、`spec`、`jest`、`mocha`、`pytest`、`go test`
- 如果项目有多个测试命令，优先运行与本次改动最相关的测试子集
- 不确定时，先运行 `--help` 查看测试命令支持哪些参数

## 测试执行检查

### 执行前检查
- [ ] 已确认测试命令的正确性（能在终端中执行）
- [ ] 已确认测试环境依赖已安装（node_modules / pip packages / go modules 等）
- [ ] 已确认数据库/缓存等外部依赖是否可用（部分测试需要）
- [ ] 如果全量测试太慢（>5 分钟），已确定只运行相关测试子集的命令

### 执行中检查
- [ ] 测试正在正常运行（没有卡住或死循环）
- [ ] 关注测试输出的实时信息（不只是等最终结果）
- [ ] 如果测试超时（>5 分钟无新输出），检查是否需要终止并调整

### 执行后检查
- [ ] 已检查全部测试输出，不只是最后的 Summary 行
- [ ] 已区分"本次改动导致的失败"和"项目中已有的失败"
- [ ] 如果存在已有的失败（不归因于本次改动），已在报告中标注
- [ ] 已保存完整的测试输出作为证据

## 补充测试检查

### 判断是否需要补充
- [ ] 本次改动涉及核心业务逻辑 → 必须补充测试
- [ ] 本次改动修复了一个 bug → 必须补充回归测试
- [ ] 本次改动新增了一个功能 → 必须补充功能测试
- [ ] 本次改动只是修改配置/文案/CSS → 可以不补充测试（但需在报告中说明理由）

### 补充测试的质量要求
- [ ] 测试用例覆盖了正常输入路径（Happy Path）
- [ ] 测试用例覆盖了关键边界条件（空值、极值、特殊字符等）
- [ ] 如果是 bug 修复，测试用例覆盖了该 bug 的复现场景
- [ ] 测试用例使用项目已有的测试框架和风格
- [ ] 测试用例文件放置在项目约定的测试目录中
- [ ] 测试用例可以独立运行，不依赖特定的执行顺序

### 补充测试的常见陷阱
- ❌ 写了一个恒为真的测试（如 `expect(true).toBe(true)`）——这不是测试，这是自欺欺人
- ❌ 测试只覆盖了正常路径，完全没有边界和异常场景
- ❌ 测试依赖了不稳定的外部资源（网络请求、当前时间等），导致偶发性失败
- ❌ 测试用例之间共享可变状态，导致执行顺序不同时结果不同

## 修复与重跑检查

### 失败分析流程
- [ ] 已逐条阅读失败信息，理解每个失败的根因
- [ ] 已判断失败是在"代码本身"还是"测试用例写错了"
- [ ] 对于代码本身的问题：回退到源文件修复
- [ ] 对于测试用例的问题：修正测试用例

### 重跑流程
- [ ] 修复后已重新运行全部相关测试（不只重跑刚才失败的）
- [ ] 重跑结果全部通过（绿色）
- [ ] 如果重跑仍有失败，继续循环直到全绿
- [ ] 循环超过 3 次仍未全绿 → 挂起并向用户报告，可能涉及更深层的设计问题

## 质量红线

如果以下任一条不满足，EXAMINE 阶段视为未完成：
1. 没有实际运行任何测试（也没有说明原因）
2. 存在归因于本次改动的测试失败但未修复
3. 需要补充测试但未补充（也没有说明原因）
4. 测试结果无法证明代码的正确性
