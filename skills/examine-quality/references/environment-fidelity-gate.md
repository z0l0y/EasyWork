# Environment Fidelity Gate（环境保真闸门）

> **铁律 #32**：根据任务类型识别运行环境。交互式应用（CLI/TUI/GUI/Web/Game）L3+ 必须在 EXAMINE 阶段至少执行一次近真实环境的冒烟测试。产出环境矩阵：OS/浏览器/设备/网络条件。纯后端/库/脚本标注 N/A。
> 此闸门与 v2.8 铁律 #26（交互式应用 EXAMINE 增强）互补——v2.8 关注用户体验，v2.12 关注运行环境匹配。

---

## 1. 环境分类

### 1.1 交互式应用类型（必须执行环境保真检查）

| 类型 | 关键环境维度 | 最少环境数 |
|------|------------|-----------|
| **CLI**（命令行工具） | OS (Windows/Linux/macOS)、终端模拟器、字符编码 | 2 个 OS |
| **TUI**（终端 UI） | 终端模拟器 (iTerm/Windows Terminal/xterm)、ANSI 兼容性、终端宽度 | 2 个终端 |
| **GUI**（桌面窗口） | OS 版本、窗口管理器、DPI 缩放 | 2 个 OS |
| **Web 前端** | 浏览器 (Chrome/Firefox/Safari/Edge)、屏幕尺寸 (Desktop/Mobile/Tablet)、网络 (WiFi/4G/3G) | 3 个浏览器 |
| **Game**（游戏） | GPU 驱动、OS、输入设备、帧率 | 2 个设备 |
| **Mobile App** | iOS/Android 版本、设备型号、网络条件 | 2 个设备 |

### 1.2 非交互式类型（标注 N/A）

| 类型 | 理由 |
|------|------|
| 纯后端 API | 无用户可见 UI——通过 API 测试验证即可 |
| 库/SDK | 被其他代码调用——通过单元测试/集成测试验证 |
| 脚本/批处理 | 在固定环境中运行——通过脚本执行验证 |
| CLI 纯数据处理 | 非交互式——输入文件→输出文件，无终端交互 |

### 1.3 环境配置说明

本文件第 1.1 节为默认矩阵定义。团队可在 `references/team-policy.md` 中声明 **team_environments** 字段来覆盖默认矩阵——列出团队有测试设备/权限的真实环境（而非泛泛的"需要测 Chrome"）。Agent 在检查环境保真度时应优先使用团队声明的环境列表。

## 2. 环境矩阵

### 矩阵模板

```markdown
### 环境矩阵

| 环境 | OS | 浏览器 | 设备 | 网络 | 状态 | 差异说明 |
|------|-----|--------|------|------|------|---------|
| **开发环境** | macOS 14 | Chrome 125 | MacBook Pro | localhost | ✅ 主要开发环境 | — |
| **近真实环境 1** | Windows 11 | Chrome 125 | Desktop 1920×1080 | localhost | ✅ 通过 | 字体渲染略有差异（Windows ClearType） |
| **近真实环境 2** | Ubuntu 22.04 | Firefox 125 | Desktop 1920×1080 | localhost | ⚠️ 未测 | 缺少 Ubuntu 测试环境 |
| **生产目标环境** | — | Chrome 120+ | Desktop/Mobile | 公网 (HTTPS) | ⚠️ 未测 | 生产环境差异：HTTPS、CDN、实际网络延迟 |

### 环境差异总结

| 差异项 | 开发环境 | 生产环境 | 影响评估 | 缓解措施 |
|--------|---------|---------|---------|---------|
| HTTPS | ❌ (localhost) | ✅ | 低：Mixed content 问题 | 本地已启用 HTTPS 自签证书测试 |
| CDN | ❌ | ✅ (CloudFront) | 中：静态资源加载路径差异 | 使用 CDN 绝对路径，本地开发时 mock |
| 网络延迟 | 0ms | 10-200ms | 低：已添加 loading 状态 | — |
```

## 3. 冒烟测试要求

### 3.1 定义

冒烟测试 = 一个端到端用户场景在近真实环境中完整执行并验证核心功能正常。

### 3.2 最低要求

| 应用类型 | 最少冒烟场景 | 场景示例 |
|---------|------------|---------|
| CLI | 1 个场景 | `tool --input file.txt --output result.json` → 验证输出文件内容正确 |
| TUI | 1 个场景 | 启动→导航到主菜单→选择一个操作→确认→退出 |
| GUI | 1 个场景 | 启动→打开文件→编辑→保存→关闭 |
| Web | 2 个场景 | ① 登录→浏览→操作→登出（Desktop Chrome）  ② 登录→浏览→操作→登出（Mobile Safari） |
| Game | 1 个场景 | 启动→主菜单→开始游戏→操作 30 秒→退出 |
| Mobile App | 1 个场景 | 启动→登录→主屏幕→核心操作→退出 |

### 3.3 冒烟测试凭据

```markdown
### 近真实环境冒烟测试

**场景**：用户登录→浏览首页→点击文章→发表评论→登出
**环境**：Windows 11, Chrome 125, Desktop 1920×1080, localhost
**命令**：`npx playwright test tests/smoke/user_journey.spec.ts --project=chrome-desktop`
**结果**：✅ 4/4 步骤通过（登录→浏览→评论→登出）
**输出凭据**：
```
Running 1 test using 1 worker
  ✓  smoke - user journey (12.3s)
    登录: OK (POST /api/login → 200, 320ms)
    浏览: OK (GET /api/articles → 200, 45ms)
    评论: OK (POST /api/comments → 201, 80ms)
    登出: OK (POST /api/logout → 200, 15ms)
  1 passed (12.3s)
```
```

## 4. 与 v2.8 交互式应用验证的关系

| 闸门 | 版本 | 关注点 | 检查内容 |
|------|------|--------|---------|
| 交互式应用 EXAMINE 增强 | v2.8 (#26) | **用户体验** | 首屏稳定性/输入反馈/退出路径/渲染频率/stdin边界/ANSI兼容 |
| 环境保真闸门 | v2.12 (#32) | **运行环境** | OS/浏览器/设备/网络条件匹配生产环境 |

两个闸门互补：
- v2.8 问："用户感觉对吗？操作流畅吗？"
- v2.12 问："环境对吗？在目标平台上测了吗？"

L3+ 交互式应用必须**两者都通过**。

## 5. Gate Judgment Rules

| 深度 | 闸门类型 | 命中处理 |
|------|---------|---------|
| **detailed** | **HARD GATE** | L3+ 交互式应用未执行近真实环境冒烟测试 → **拒绝放行** → 补充冒烟测试 |
| **standard** | **SOFT GATE** | 警告但允许继续 |
| **brief** / **L0-L2** / **纯后端** | **SKIP** | 跳过，标注 N/A + 理由 |

### 阻断解除条件

```
L3+ 交互式应用无冒烟测试：
  → 至少执行 1 次近真实环境冒烟测试
  → 记录测试环境、命令、输出凭据
  → 重新检测通过后解除

环境矩阵不完整：
  → 标注缺失环境
  → 记录环境差异和影响评估
  → 如无法补充环境→记录为"已知差异"
```

## 6. 反模式

- ❌ **L3 Web 应用只在 localhost Chrome 测试就声称"已验证"**：缺少 Firefox/Safari 验证，缺少移动端验证，缺少 HTTPS 环境
- ❌ **环境矩阵全是"相同"**：开发环境 = 生产环境？几乎不可能。至少 CDN/HTTPS/网络延迟/数据量级有差异
- ❌ **冒烟测试只在 dev 环境跑**：没有换一个 OS/浏览器/设备验证
- ❌ **把单元测试当冒烟测试**：`npm test` 全部通过 ≠ 在真实浏览器中完整走了一遍用户流程
- ❌ **纯后端 API 也标注 N/A 但不说明理由**：N/A 必须附理由——"本项目为纯 REST API，无用户可见 UI，功能通过 API 集成测试验证"
