# Test Adequacy Gate（测试充分性闸门）

> **铁律 #34**：EXAMINE 从"跑测试"升级为"判断测试是否覆盖风险"。五维覆盖检查——Happy Path/边界/错误路径/并发/安全。Bug 修复必须新增至少一个在老代码上 FAIL、在新代码上 PASS 的回归测试。无回归测试 = 不可声称修复完成。
> 此闸门不可跳过。L1+ 任务强制执行。L0 任务跳过。

---

## 1. 五维覆盖检查

### 1.1 Happy Path — 正常路径

**检查内容**：正常输入是否产生预期输出。

**典型缺失**：
- 没有测试正常输入路径——只测了错误处理
- 正常路径测试只有一个——未覆盖不同输入组合
- 正常路径测试依赖外部服务（数据库/API）且无 mock

**覆盖判定**：
| 状态 | 条件 |
|------|------|
| **covered** | ≥1 个 Happy Path 测试，覆盖主要输入组合 |
| **partial** | 有 Happy Path 测试但未覆盖主要输入变体 |
| **uncovered** | 无任何 Happy Path 测试 |

### 1.2 Boundary — 边界值

**检查内容**：边界值/空值/极值/最大长度/零值/负值。

**典型缺失**：
- 没有测空字符串/null/undefined
- 没有测最大长度和最大长度+1
- 没有测零值和负值（如适用）
- 没有测并发边界（如适用）

**覆盖判定**：
| 状态 | 条件 |
|------|------|
| **covered** | ≥1 个边界测试覆盖空值+极值+越界 |
| **partial** | 有边界测试但遗漏关键边界（如只测了空值没测最大值） |
| **uncovered** | 无任何边界测试 |

### 1.3 Error Path — 错误路径

**检查内容**：异常/超时/资源不足/网络失败/数据库连接失败/第三方 API 错误。

**典型缺失**：
- 没有测 try-catch 分支（只测了 try）
- 没有测超时场景
- 没有测外部依赖失败时的降级行为
- 没有测错误信息的可用性（用户能不能看懂）

**覆盖判定**：
| 状态 | 条件 |
|------|------|
| **covered** | ≥1 个错误路径测试覆盖主要异常类型 |
| **partial** | 有错误测试但遗漏关键异常（如只测了 null 没测超时） |
| **uncovered** | 无任何错误路径测试 |

### 1.4 Concurrency — 并发（如适用）

**检查内容**：并发访问/竞态条件/死锁/数据一致性。

**适用条件**：
- 代码涉及共享状态（全局变量/缓存/数据库）
- 代码使用异步操作（Promise/async-await/回调/事件）
- 代码涉及文件系统操作

**典型缺失**：
- 没有测并发写入（两个请求同时写同一资源）
- 没有测竞态条件（异步操作完成顺序不确定）
- 没有测锁/信号量的正确释放

**覆盖判定**：
| 状态 | 条件 |
|------|------|
| **covered** | ≥1 个并发测试（如适用）或标注"本项目为单线程/无共享状态，不适用" |
| **uncovered** | 适用但无任何并发测试 |
| **n/a** | 纯同步代码/无共享状态/无异步操作 |

### 1.5 Security — 安全（如适用）

**检查内容**：注入/越权/敏感数据泄露/输入验证绕过。

**适用条件**：
- 代码接收外部输入（HTTP 请求/CLI 参数/文件上传）
- 代码涉及认证/授权逻辑
- 代码处理敏感数据（密码/Token/PII）

**典型缺失**：
- 没有测 SQL/NoSQL 注入
- 没有测 XSS（如输出 HTML）
- 没有测越权访问（用户 A 访问用户 B 的数据）
- 没有测敏感数据的日志泄露

**覆盖判定**：
| 状态 | 条件 |
|------|------|
| **covered** | ≥1 个安全测试（如适用）或标注"不处理外部输入/敏感数据，不适用" |
| **uncovered** | 适用但无任何安全测试 |
| **n/a** | 纯内部计算/无外部输入/无敏感数据 |

## 2. 回归测试强制要求（Bug 修复铁律）

### 规则

> **Bug 修复必须新增至少一个回归测试。该测试必须：在老代码上 FAIL → 在新代码上 PASS。无回归测试 = 不可声称修复完成。**

### 回归测试三步验证

```
Step 1 — 在老代码上确认失败：
  git stash 新代码 → 运行回归测试 → 确认 FAIL
  记录: "回归测试在 commit {baseline_revision} 上 FAIL，输出: {关键错误信息}"

Step 2 — 在新代码上确认通过：
  git stash pop → 运行回归测试 → 确认 PASS
  记录: "回归测试在 commit {latest_revision} 上 PASS"

Step 3 — 确保测试测试的是根因，非表象：
  检查: 如果只 mock 了错误但没验证根本原因 → 测试不够
  反例: 程序崩溃因为 null pointer → 回归测试只检查"不崩溃" → 不够
  正例: 回归测试检查"函数在输入为 null 时返回默认值而非崩溃"
```

### 回归测试格式

```markdown
### 回归测试: {test_name}

| 属性 | 值 |
|------|-----|
| **测试名称** | `should_return_default_when_input_is_null` |
| **测试位置** | `tests/utils/parseConfig.test.ts:42-58` |
| **复现场景** | 用户输入空配置文件 → parseConfig 崩溃 |
| **老代码 FAIL** | `TypeError: Cannot read properties of null` on abc1234 |
| **新代码 PASS** | `parseConfig(null)` 返回 `DEFAULT_CONFIG`，5 assertions pass |
| **根因覆盖** | 测试验证了 null/undefined/空对象三种空值场景 |
```

### 何种修复不需要回归测试

| 修复类型 | 理由 | 替代验证 |
|---------|------|---------|
| 纯文案修正 | 不改逻辑 | 目视确认 |
| 配置变更（如端口号） | 不改逻辑 | 配置文件 diff |
| 依赖版本升级（patch） | 外部变更 | 现有测试通过即可 |
| 注释修正 | 不改逻辑 | 目视确认 |

## 3. 测试覆盖盲区分析

### 输出格式

```markdown
### 测试覆盖盲区

| # | 未覆盖场景 | 原因 | 风险 | 缓解措施 |
|---|-----------|------|------|---------|
| 1 | 并发登录请求（>100 QPS） | 项目无并发测试基础设施 | 中：高负载下可能出现 token 重复签发 | 添加限流中间件（已实现），生产环境监控 |
| 2 | Safari 15 兼容性 | 无 Safari 15 测试设备 | 低：Safari 15 市场份额 < 0.5% | 标注为已知差异，用户反馈后再处理 |

### 覆盖率评估

| 维度 | 覆盖状态 | 证据 |
|------|---------|------|
| Happy Path | ✅ covered | `login.test.ts`: 正常登录/记住我/重定向 3 个场景 |
| Boundary | ✅ covered | `login.test.ts`: 空密码/超长用户名/特殊字符 3 个边界 |
| Error Path | ✅ covered | `login.test.ts`: 错误密码/账户锁定/网络超时 3 个错误 |
| Concurrency | ⚠️ n/a | 本项目为纯同步代码，无共享状态 |
| Security | ✅ covered | `login.test.ts`: SQL注入/XSS/越权访问 3 个安全测试 |
```

## 4. Gate Judgment Rules

| 深度 | 闸门类型 | 命中处理 |
|------|---------|---------|
| **detailed** | **HARD GATE** | Bug修复且无回归测试→阻断；L2+且五维中三维未覆盖→阻断或要求声明豁免 |
| **standard** | **SOFT GATE** | 警告但允许继续（标注"测试充分性未完整检查"） |
| **brief** | **SKIP** | L0/L1 跳过 |

### 阻断解除条件

```
Bug修复无回归测试：
  → 补充回归测试 + 老代码FAIL记录 + 新代码PASS记录
  → 重新检测通过后解除

L2+三维未覆盖：
  → 补充测试覆盖缺失维度 或
  → 声明豁免（说明为何该维度不适用 + 用户确认）
  → 重新检测通过后解除
```

## 5. 反模式

- ❌ **expect(true).toBe(true) 充数**：写一个永远通过的测试声称"回归测试已添加"——欺诈
- ❌ **声称"不需要回归测试因为这个 bug 很简单"**：简单的 bug 也可能被再次引入——回归测试的成本极低，收益永久
- ❌ **五维全标 "covered" 但实际只有 Happy Path 测试**：虚报覆盖率——每维必须有具体测试用例名称和位置
- ❌ **回归测试没复现原始 bug**：写了一个测试但它无论在老代码还是新代码都 PASS——没有验证"老代码上 FAIL"
- ❌ **用"n/a" 绕过检查**：所有维度都标 "n/a"——事实上不可能一个维度都不适用
