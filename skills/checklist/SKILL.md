# Checklist（清单审计与就绪检查）

## 定位

**开发全流程的"飞行员检查清单"。不是又一个闸门——而是把分散在 40 条铁律、10+ 闸门、7 个技能中的关键检查点，聚合成一张可逐行核对的清单。**

为什么需要这个 skill：

- **开发前**：不清楚哪里会遗漏 → Pre-flight 清单把需求细化为可核对条目
- **开发后**：不清楚要 Check 哪些地方 → Audit 清单逐点核对，不遗漏
- **Agent 执行**：有清单后 Agent 执行更流畅——每做一项勾一项，不跳步

**跟现有闸门系统的关系**：

| | 闸门系统 | checklist skill |
|---|---------|----------------|
| 角色 | **执行机制**——深度验证某一步骤的产出 | **聚合视图**——全局可见的核对清单 |
| 粒度 | 每个闸门覆盖 1-2 个维度 | 一张清单覆盖全部 10 个类别 |
| 时机 | 分散在各步骤中触发 | Pre-flight（CODE 前）/ Audit（ASK 前） |
| 格式 | 闸门检查逻辑（硬阻断/软警告） | 可逐行核对的清单（阻塞/警告/提示） |

**参考来源**：

| 来源 | 吸收要点 |
|------|---------|
| **《清单革命》**（Atul Gawande） | READ-DO（执行前）vs DO-CONFIRM（执行后）两种格式；每节 5-9 条；只列"常被遗忘+遗漏代价高" |
| **GitHub PR Template**（Kubernetes/React/Rust） | 分类 checklist 格式；Type of Change / Testing / Checklist 三段式 |
| **Google PRR**（Production Readiness Review） | 多维度发布前检查；Security / Reliability / Observability / Rollback |
| **Amazon Working Backwards** | 5 问引导法（客户/问题/方案/描述/度量）→ Pre-flight 就绪检查 |
| **Cohen 的 6 条清单规则** | ≤10 条/节；不列显而易见的；不列可自动化的；聚焦常被遗忘的；缺陷数据驱动；定期淘汰 |

## 前置判断

### 什么时候用 Pre-flight（执行前就绪检查）

**触发**：READ 阶段完成，准备进入 CODE 阶段。
**格式**：READ-DO（读一步 → 做一步 → 勾一项）
**行为**：从完成定义（delivery_definition）自动生成清单，逐条核对就绪状态。

```
什么时候必须用：
  - 功能开发/重构任务（涉及多文件改动）
  - 风险等级 L2+
  - 用户明确说"开发前检查"、"就绪检查"、"预检"

什么时候跳过：
  - 微调任务（< 20 行改动，1 个文件）
  - 纯文档任务
  - 用户说"不用检查，直接做"
```

### 什么时候用 Audit（交付审计清单）

**触发**：SUM 阶段完成，准备进入 ASK 阶段。
**格式**：DO-CONFIRM（凭记忆做完所有步骤 → 停顿 60-90 秒 → 逐条核对验证）
**行为**：聚合所有闸门结果、测试结果、证据账本，生成交付审计清单。

```
什么时候必须用：
  - 所有有代码改动的任务
  - L1+ 风险等级
  - 用户明确说"交付检查"、"核对清单"、"交付出清"

什么时候跳过：
  - L0 纯理解/纯文档任务
  - 用户说"跳过审计"
```

### 任务类型 → 清单深度

| 任务类型 | Pre-flight | Audit | 类别数 | 条目数 |
|---------|-----------|-------|--------|--------|
| 🚀 功能开发 | ✅ 完整 | ✅ 完整 | 10 类 | 40-60 条 |
| 🔧 重构 | ✅ 完整 | ✅ 完整 | 8 类（跳过部署/合规） | 30-50 条 |
| 🐛 Bug修复 | ⏭️ 跳过 | ✅ 标准 | 6 类 | 20-30 条 |
| 📝 纯文档 | ⏭️ 跳过 | ✅ 标准 | 4 类 | 10-20 条 |
| ⚡ 微调 | ⏭️ 跳过 | ✅ 轻量 | 3 类 | 5-10 条 |
| 🎯 单步调用 | — | 按需 | 用户指定 | 按需 |

**深度自动升级**：
- 安全相关改动（auth/权限/加密/输入验证）→ 自动升级安全类别到完整
- 数据库 schema 变更 → 自动升级回滚/恢复类别到完整
- 外部 API 新增 → 自动升级性能/可靠性类别到完整

## 10 大检查类别

### 类别一览

| # | 类别 | 适用场景 | 最低风险等级 | 完整条目数 |
|---|------|---------|------------|-----------|
| 1 | 📋 **需求完整性** | 所有类型 | L0+ | 7 |
| 2 | 🔒 **安全性** | 功能开发/重构/Bug修复 | L2+ | 7 |
| 3 | ⚡ **性能/可靠性** | 功能开发/重构 | L2+ | 5 |
| 4 | 🧪 **测试覆盖** | 所有有代码改动的 | L1+ | 7 |
| 5 | 📝 **文档/注释** | 所有类型 | L0+ | 5 |
| 6 | 🚀 **部署/运维** | 功能开发/Bug修复 | L2+ | 5 |
| 7 | 🔄 **回滚/恢复** | 功能开发/重构 | L3+ | 5 |
| 8 | ♿ **可访问性** | 前端/交互式应用 | L3+ | 5 |
| 9 | 📐 **代码质量** | 所有有代码改动的 | L0+ | 7 |
| 10 | ⚖️ **合规/法律** | 功能开发 | L3+ | 5 |

## 执行流程

### 阶段 0：模式判定

```
用户输入分析:
  ├── 含 "开发前" / "就绪检查" / "预检" / "preflight" / "pre-flight"
  │   → 🔍 Pre-flight 模式（执行前就绪检查）
  │
  ├── 含 "交付" / "核对" / "审计" / "交付出清" / "检查清单"
  │   → ✅ Audit 模式（交付审计清单）
  │
  └── 无明确模式关键词
      → 按上下文字段推断:
          - 在 READ 之后、CODE 之前 → Pre-flight
          - 在 SUM 之后、ASK 之前 → Audit
          - 独立调用 → 询问用户要哪种模式
```

### 阶段 1：确定清单范围

**根据任务类型和风险等级裁剪类别**：

```
功能开发 + L3:  全 10 类，完整条目数（40-60 条）
功能开发 + L2:  9 类（跳过回滚/恢复），完整条目数（35-50 条）
重构 + L2:     8 类（跳过部署+合规），完整条目数（30-45 条）
Bug修复 + L1:   6 类（需求完整性+安全+测试+文档+部署+代码质量），标准条目数（20-30 条）
纯文档 + L0:    4 类（需求完整性+文档+代码质量+合规），标准条目数（10-20 条）
微调 + L0:      3 类（测试+文档+代码质量），轻量条目数（5-10 条）

特殊触发:
  - 安全相关改动（认证/权限/加密/输入验证）→ 安全类别强制完整
  - 数据库 schema 变更 → 回滚/恢复类别强制完整
  - 新增外部 API → 性能类别强制完整
  - 前端/UI 改动 → 可访问性类别强制完整
```

### 阶段 2：生成清单

**Pre-flight 模式**（从完成定义预填充）：

```
对每个适用类别:
  1. 读取 delivery_definition 中的相关字段
  2. 将已完成项自动勾选（✅），未完成项留空（☐）
  3. 对空项标注"需要做什么"和"找谁确认"
  4. 在每个类别标题旁显示进度（如 "📋 需求完整性 5/7"）
```

**Audit 模式**（从闸门结果聚合）：

```
对每个适用类别:
  1. 读取对应步骤的产出物（EXAMINE → 测试结果；SUM → 证据账本；...）
  2. 逐条核对：有证据→✅ 通过；无证据→❌ 失败；N/A→⏭️ 跳过
  3. 对失败项标注严重程度 + 修复建议 + 来源闸门
  4. 汇总所有失败项到 gaps 列表
```

**Standalone 模式**（独立触发，无上下文）：

```
Agent 输出：
"请告诉我你想检查什么类型的任务？
A) 功能开发（完整清单）
B) Bug 修复（标准清单）
C) 微调（轻量清单）
D) 自定义（选择要检查的类别）"
```

### 阶段 3：输出清单 + 交互

Agent 输出清单报告，用户逐行核对：

```
用户可交互:
  - "C1.2 通过" → 勾选该条
  - "C2.1 不适用" → 标记跳过(注明原因)
  - "展开 C4 类别" → 查看该类的详细说明
  - "失败项汇总" → 只看失败的条目
  - "全部通过" → 一键确认所有剩余条目
```

## 反模式

- ❌ **清单变成走形式**——每条必须真实核对，不能凭记忆批量勾选。"差不多都做了"= 没做清单
- ❌ **清单项超过 10 条/类**——工作记忆上限 7±2。超过 10 条 → 拆分到更细的类别
- ❌ **列显而易见的检查项**——"代码能编译"、"文件已保存" 这种不需要清单。只列常被遗忘+遗漏代价高的
- ❌ **列 CI 已经能检查的项**——"代码格式正确"、"lint 通过"——这些 CI 已经做了。清单聚焦 CI 覆盖不到的地方
- ❌ **Audit 模式不读取步骤产出**——清单不是凭空打勾。每个 ✅ 必须指向具体的证据位置
- ❌ **阻塞项和警告项混淆**——安全红线（阻塞）不能降级；注释不全（提示）不能升级。保持三类分明
- ❌ **跳过的条目不写原因**——每个 ⏭️ 必须标注为什么 N/A（如"非 Web 应用，跳过 CORS 检查"）
- ❌ **清单不随任务类型裁剪**——功能开发的完整清单不能直接用于微调。必须有按任务类型的裁剪逻辑
- ❌ **Pre-flight 模式不区分"可自动判定"vs"需人工确认"**——C1.2（完成定义是否填写）可自动判定，但 C1.1（需求理解是否正确）必须人工确认。两类标记清楚
- ❌ **把清单当成闸门的替代品**——清单是聚合视图，闸门是执行机制。清单告诉你"要检查什么"，闸门执行具体的检查逻辑。两者共存互补
- ❌ **失败项没有修复建议**——只标失败不给修复方向 = 没用。每条 ❌ 必须附"怎么修"
- ❌ **清单是一次性的**——每次 Audit 的结果应该反馈到下一次 Pre-flight。连续 3 次某条都通过 → 可考虑降为提示或移除。连续 3 次某条都失败 → 应该是流程问题，不是清单能解决的
