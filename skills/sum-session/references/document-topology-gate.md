# Document Topology Gate（文档拓扑闸门）

> **铁律 #28**：写入文档之前必须检查拓扑结构。文档不是变更日志堆，而是按流程节点持续演进的知识库。
> 每次更新应进入对应流程节点形成可追溯的版本树，而非在文档尾部堆积新块。

---

## 1. 什么是文档拓扑

文档拓扑指的是文档的**顶层结构骨架**——一级标题（`#`）和二级标题（`##`）的组织方式。拓扑合法意味着：

- 顶层节点类型一致（全部是 Run 块，或全部是流程节点）
- 每种流程节点只出现一次（Mode B）
- 顶部有版本索引
- 历史版本与当前内容通过版本标签区分

拓扑混乱的文档像"一本没有目录、章节编号重复、新内容贴在旧内容末尾的书"——读起来是流水账。

---

## 2. 七项拓扑检查

### Check 1 — 重复顶层流程节点

**检测方法**：
扫描所有 `## \d+\. (READ|CODE|REVIEW|EXAMINE|GIT|GRAPH|SUM|TALK|SELFCHECK|ASK)` 标题，检查是否有同名节点出现 ≥2 次。

**违规示例**：
```
## 1. READ（需求理解）    ← 第一次
...
## 2. CODE（代码实现）
...
## 1. READ（需求理解）    ← 重复！第二次出现
```

**判定**：
| 模式 | 处理 |
|------|------|
| Mode B | **HARD GATE** — 阻断写入，合并重复节点 |
| Mode A | 允许（每个 Run 下独立拥有流程节点） |

**修复**：将两个同名节点的内容合并为一个节点，按版本号排序子节。

---

### Check 2 — 模式混用

**检测方法**：
扫描顶层标题，检查是否同时存在 `## Run` 风格标题（`## Run \d{4}-\d{2}-\d{2}`）和 `## \d+\. (READ|CODE|...)` 风格标题。

**违规示例**：
```
## Run 2026-06-20 v1       ← Mode A 风格
### 1. READ
...
## 1. READ（需求理解）       ← Mode B 风格
...
```

**判定**：
| 模式 | 处理 |
|------|------|
| 任一 | **HARD GATE** — 阻断写入，强制统一为一种模式 |

**修复策略**：
1. 检测用户意图（是否有"保留审计日志"指示）
2. 默认统一为 Mode B（将 Run 块内容拆入节点）
3. 如用户明确选 Mode A → 将游离节点归入最近的 Run 块

---

### Check 3 — 版本索引缺失

**检测方法**：
在 Mode B 文档中，检查是否存在 `## 0. 版本索引` 或 `## 版本索引` 标题。检查索引中的版本是否与节点内版本一致。

**违规示例**（Mode B 文档）：
```
# 登录优化报告
## 1. READ（需求理解）      ← 直接开始，缺少版本索引
...
```

**判定**：
| 模式 | 处理 |
|------|------|
| Mode B | **HARD GATE** — 阻断写入，补建版本索引 |
| Mode A | N/A（Run 标题即版本标识，但顶部应有版本索引表） |

**修复**：扫描所有节点内 `### v{N}` 子节，自动生成版本索引表插入顶部。

---

### Check 4 — 本轮内容未入节点

**检测方法**：
检查本轮（当前版本）的改动内容是否出现在对应流程节点下，而非仅出现在文档级的"更新记录"块中。

**违规示例**（Mode B 文档，本轮 v3）：
```
## 📝 更新记录              ← 本轮改动全在这里
### 更新 v3 — 2026-06-21
本次修改了 auth.go 的 Login 函数...
增加了 TestLoginExpired 测试...
审查通过，七维度无阻断...
```

而 `## 2. CODE` 和 `## 4. EXAMINE` 节点下没有 v3 子节。

**判定**：
| 模式 | 处理 |
|------|------|
| Mode B | **HARD GATE** — 阻断写入，将更新记录内容拆入对应流程节点 |
| Mode A | 允许（Run 块包含本轮全部产出） |

---

### Check 5 — 孤儿更新记录块

**检测方法**：
在 Mode B 文档中，检查顶层是否存在 `## 📝 更新记录` 或 `## 更新记录` 或 `## 变更日志` 等独立块。

**违规示例**（Mode B 文档）：
```
## 1. READ（需求理解）
...
## 2. CODE（代码实现）
...
## 📝 更新记录              ← 孤儿块！应拆入节点内
### 更新 v1.1 — 2026-06-25
...
```

**判定**：
| 模式 | 处理 |
|------|------|
| Mode B | **HARD GATE** — 阻断写入，将更新记录内容拆入对应流程节点 |
| Mode A | 允许（更新记录是 Run 级别的，Mode A 下合法） |

**修复**：解析每个更新条目中的内容，识别涉及的流程节点（READ/CODE/REVIEW 等），将内容迁移到对应节点下的版本子节。

---

### Check 6 — 历史与当前混淆

**检测方法**：
检查过时内容是否标注了版本标记（`[已过时 — v{N} 起]`），当前有效内容是否与历史版本明确区分（通过版本小节标题 `### v{N}` 区分）。

**违规示例**：
```
## 2. CODE（代码实现）

修改了 auth.go 的 Login 函数，增加了重试逻辑。       ← 这是 v1？v2？无法判断
另外在 v2 中把 costFactor 从 14 改成了 10。           ← 混在一起
```

**判定**：
| 模式 | 处理 |
|------|------|
| Mode B | **WARN** — 标注混淆区域，要求格式化（不阻断） |
| Mode A | **WARN** — 同上 |

**修复**：按版本拆分为独立的 `### v{N}` 子节，旧版本前加 `[已过时]` 标记。

---

### Check 7 — 顶层索引与实际不一致

**检测方法**：
对比 `## 0. 版本索引` 表格中"影响节点"列与实际节点内版本子节。索引声明的 v3 影响了 CODE 和 EXAMINE，但 `## 4. EXAMINE` 下没有 `### v3` 子节 → 不一致。

**判定**：
| 模式 | 处理 |
|------|------|
| Mode B | **HARD GATE** — 阻断写入，修复索引 |
| Mode A | N/A |

**修复**：以节点内实际版本为准，重新生成版本索引。

## 3. 拓扑解析算法

### Step 1 — 识别文档模式

```
扫描顶层（## 级别）标题列表：
  if 所有标题匹配 "## Run \d{4}-\d{2}-\d{2}" 或 "## 第\d+次":
    → Mode A（审计日志）
  else if 所有标题匹配 "## \d+\. (READ|CODE|REVIEW|...)" 或 "## 0. 版本索引":
    → Mode B（持续维护）
  else if 混合:
    → 混乱态（需修复）
  else:
    → 新文档（默认为 Mode B）
```

### Step 2 — 提取节点树（Mode B）

```
nodes = {}
for each ## heading:
  key = heading_number  (e.g. "1", "2", "0")
  name = heading_text   (e.g. "READ（需求理解）", "版本索引")
  versions = []  # 扫描 ### v{N} 子标题
  for each ### heading under this node:
    if matches "### v(\d+) — (\d{4}-\d{2}-\d{2})":
      versions.append({version: N, date: date, type: type_tag})
  nodes[key] = {name, versions}
```

### Step 3 — 提取版本映射

```
从版本索引表提取：version → {date, summary, affected_nodes}
从节点内提取：node → [versions]
交叉比对 → 发现不一致
```

## 4. Gate 判定规则

| 深度 | 闸门类型 | 命中处理 |
|------|---------|---------|
| **detailed** | **HARD GATE** | 任一 Check 1-5,7 命中 → 拒绝写入 → 自动执行 structural merge repair → 重新检测（最多 2 轮） |
| **standard** | **SOFT GATE** | 展示违规清单 → 用户确认是否修复（默认修复） |
| **brief** | 跳过 | 标注 "⚠️ 简要摘要 — 文档拓扑未经检查，不适合工程追溯" |

### 自动修复 vs 需用户确认

| 修复操作 | 自动 | 需确认 |
|---------|------|--------|
| 重复节点合并 | ✅ | — |
| 孤儿更新记录拆入节点 | ✅ | — |
| 版本索引补建/修复 | ✅ | — |
| 历史/当前区分标注 | ✅ | — |
| 模式混用统一 | — | ✅（需确认目标模式） |
| Mode A→B 转换 | — | ✅（不可逆操作） |

## 5. 结构修复算法

### 算法 1 — 孤儿更新记录→节点迁移

```
输入：Mode B 文档，顶层存在 "## 📝 更新记录"
输出：更新内容已迁入对应流程节点

for each "### 更新 v{N}" in 更新记录块:
  提取变更原因文本 → 识别关键词:
    "需求" / "背景" / "验收" → READ
    "修改" / "实现" / "代码" / "重构" → CODE
    "审查" / "检查" → REVIEW
    "测试" / "验证" / "运行" → EXAMINE
    "提交" / "拆分" → GIT
    "总结" / "报告" → SUM
    "反思" / "复盘" / "根因" → TALK
    "拷打" / "自检" / "问答" → SELFCHECK
  在对应节点下创建 "### v{N} — {date}（变更）" 子节
  将更新内容填入子节

删除原 "## 📝 更新记录" 块
更新版本索引
```

### 算法 2 — 重复节点合并

```
输入：两个或多个同名顶层节点（如两个 "## 1. READ"）
输出：一个合并后的节点

合并节点 = 第一个节点
for 后续同名节点:
  for each "### v{N}" in 后续节点:
    如合并节点中无此版本 → 追加到合并节点末尾
    如合并节点中已有此版本 → 合并内容（追加不重复的内容）
  删除后续节点
按版本号升序排列子节
```

### 算法 3 — 版本索引重建

```
输入：节点树
输出：版本索引表

收集所有节点中出现的版本号 → 去重排序
for each version:
  affected = 包含此版本的节点列表
  summary = 从该版本子节第一段提取（或标记 "待补充"）
生成索引表 Markdown
插入到文档顶部（紧跟 # 标题之后）
```

## 7. 与 v2.8 的关系

v2.8 铁律 #25（文档迭代增量更新）的核心意图——文档可追溯完整演进历史——在 v2.10 中**保留但升级实现方式**：

| 维度 | v2.8（旧） | v2.10（新） |
|------|-----------|------------|
| 新增内容放哪里 | 文档末尾 `## 📝 更新记录` | 对应流程节点内 `### v{N}` |
| 过时信息处理 | 段落前 `[已过时]` 标记 | 节点内版本级标注 + 旧版本保留 |
| 版本标识 | 更新记录中的 `v1.1` | 全局版本号 `v1, v2, v3` + 节点内子节 |
| 可追溯性 | 更新记录堆（需阅读全部记录） | 节点内版本链（只看一个节点即知演进） |
| 结构性约束 | 无（"不新增顶层标题"） | 严格（拓扑闸门 + 7 项检查） |

v2.8 的 `### 更新 v1.1 — date` 格式在 Mode A 中仍可使用（Run 级别的更新记录），但在 Mode B 中已被节点内 `### v{N}` 替代。

## 8. 反模式

- ❌ Mode B 文档在顶层出现 `## 📝 更新记录`——应拆入流程节点
- ❌ Mode B 文档出现两个 `## 1. READ` 节点——应合并
- ❌ 新版本内容全在更新记录块里，节点内无新版本子节——是流水账
- ❌ 历史内容和当前内容混在同一段落无版本区分——后人不知道什么是当前有效的
- ❌ 跳过 Topology Gate 直接写入——detailed 模式下此为 HARD GATE
- ❌ 混合使用 Mode A 和 Mode B 顶层结构——必须统一
- ❌ 版本索引有 v3 但节点内找不到——索引与实际不一致
