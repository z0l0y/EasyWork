# Document Modes（双模文档结构）

> EasyWork 产物支持两种文档模式：**审计日志模式（Mode A）** 和 **持续维护模式（Mode B）**。
> Mode B 是默认模式，适合所有交付文档。Mode A 仅在用户明确要求保留完整执行轨迹时使用。

---

## 1. Mode A — 审计日志模式（Audit Log）

### 设计意图

保留每次工作流执行的**完整轨迹**。每个 Run 是一份独立的、自包含的工程记录，包含该次执行的全部步骤产出。适合合规审计、事故复盘、对外交接等需要"看到每一次完整过程"的场景。

### 结构模板

```
# {项目/功能} 工程报告

## 0. 版本索引

| Run | 日期 | 版本 | 变更摘要 |
|-----|------|------|---------|
| Run 3 | 2026-06-21 | v3 | 修复 Safari 17 兼容性 |
| Run 2 | 2026-06-20 | v2 | 降低 bcrypt cost factor |
| Run 1 | 2026-06-18 | v1 | 初始实现：登录性能优化 |

当前版本：v3
文档模式：审计日志（Mode A）

---

## Run 2026-06-21 v3 — 修复 Safari 17 兼容性

### 1. READ（需求理解）
{完整的 READ 产出}

### 2. CODE（代码实现）
{完整的 CODE 产出}

### 3. REVIEW（代码审查）
{完整的 REVIEW 产出}

### 4. EXAMINE（测试验证）
{完整的 EXAMINE 产出}

### 5. SELFCHECK（CTO 拷打）
{完整的 SELFCHECK 产出}

---

## Run 2026-06-20 v2 — 降低 bcrypt cost factor

### 1. READ（需求理解）
{完整的 READ 产出}
...

---

## Run 2026-06-18 v1 — 初始实现

### 1. READ（需求理解）
{完整的 READ 产出}
...
```

### 关键特征

- 每个 Run 块是**自包含的**：从 READ 到 SELFCHECK 完整链路
- 版本索引在顶部提供快速导航
- Run 之间按时间倒序排列（最新在前）
- 每次新执行 → 新 Run 块（追加在版本索引之后、上一 Run 之前）
- 历史 Run **不修改**——保留当时产出的完整快照

### 适用场景

- 用户说"保留完整审计日志""每次完整记录执行轨迹""审计模式"
- 合规性要求每次变更有独立可追溯记录
- 对外交接（给外部团队每次完整上下文）
- 事故复盘（需要完整时间线）

---

## 2. Mode B — 持续维护模式（Continuous Maintenance）

### 设计意图

文档是**持续演进的知识库**。每次变更不产生新的完整报告，而是增量更新到对应流程节点下，形成节点内的版本树。适合日常开发、持续迭代、团队内部文档维护。

### 结构模板

```
# {项目/功能} 工程报告

## 0. 版本索引

| 版本 | 日期 | 变更摘要 | 影响节点 |
|------|------|---------|---------|
| v3 | 2026-06-21 | 修复 Safari 17 兼容性（CSS Grid fallback） | READ, CODE, REVIEW, EXAMINE |
| v2 | 2026-06-20 | 降低 bcrypt cost factor 14→10 | CODE, EXAMINE |
| v1 | 2026-06-18 | 初始实现：登录性能优化 | 全部 |

当前版本：v3
文档模式：持续维护（Mode B）

---

## 1. READ（需求理解）

### v3 — 2026-06-21（补充：Safari 兼容性需求）
{本轮新增/变更的 READ 内容}

### v1 — 2026-06-18（初始：登录性能优化需求）
{原始 READ 内容}

---

## 2. CODE（代码实现）

### v3 — 2026-06-21（变更：Safari CSS fallback）
{本轮 CODE 变更内容}

### v2 — 2026-06-20（变更：bcrypt cost factor）
{上次 CODE 变更内容}

### v1 — 2026-06-18（初始：登录页布局 + 密码验证）
{原始 CODE 内容}

---

## 3. REVIEW（代码审查）

### v3 — 2026-06-21（变更后审查）
{本轮 REVIEW 产出}

### v1 — 2026-06-18（初始审查）
{原始 REVIEW 产出}

---

## 4. EXAMINE（测试验证）

### v3 — 2026-06-21（变更后测试）
{本轮 EXAMINE 产出}

### v2 — 2026-06-20（变更后测试）
{上次 EXAMINE 产出}

### v1 — 2026-06-18（初始测试）
{原始 EXAMINE 产出}

---

## 5. SELFCHECK（CTO 拷打）

### v3 — 2026-06-21
{本轮 SELFCHECK 产出}
...
```

### 关键特征

- 每个流程节点只出现**一次**（如 `## 1. READ` 只出现一次）
- 节点内按版本分层：`### v{N} — {date}（{变更类型}）`
- 版本号是**全局递增**的（v1, v2, v3...），但在各节点内**只出现有变更的版本**
- 旧版本内容保留在节点内，不删除，但当前版本在最前
- 版本索引标注每个版本影响了哪些节点
- **不允许**出现独立的 `## 📝 更新记录` 块

### 适用场景

- **默认模式**——除非用户明确要求 Mode A
- 日常功能开发、Bug 修复、重构
- 持续迭代的产品文档
- 团队内部 Code Review 参考文档
- PR 描述和 Release Notes 的基础材料

---

## 3. 模式选择决策树

```
新任务 → 是否有已有产物文档？
  ├── 否 → 新文档 → Mode B（默认）
  └── 是 → Fetch 已有文档 → 解析拓扑 →
        ├── 现有 Mode A → 保持 Mode A（追加新 Run）
        │     └── 除非用户说"改为维护模式" → 确认后转 Mode B
        ├── 现有 Mode B → 保持 Mode B（节点内版本追加）
        └── 拓扑混乱 → 先修复 → 用户确认目标模式 → 统一

用户显式指示：
  - "保留完整审计日志""审计模式""每次完整记录" → Mode A
  - "持续维护""更新文档""追加到上次报告""维护模式" → Mode B
  - 未说明 → Mode B（默认）
```

---

## 4. Mode A → Mode B 转换

**触发条件**：用户明确说"把审计日志转为维护模式""合并历史 Run"

**转换步骤**：
1. 列出所有 Run → 用户确认转换（**不可逆**）
2. 提取所有 Run 中出现的流程节点类型
3. 对每个流程节点：
   - 遍历所有 Run，提取该节点的产出
   - 按 Run 日期标注版本号
   - 合并到节点下（去重 + 排序）
4. 重建版本索引
5. 删除原 Run 级标题
6. 标注文档模式为 Mode B

**风险提示**：转换后每个 Run 的"完整时间线上下文"会丢失——各步骤产出分散到节点中，不再能按 Run 一次性看到完整链路。转换前必须告知用户此损失。

---

## 5. 版本号规则

### 全局版本号递增

- v1, v2, v3... 全局递增，不按节点独立编号
- 版本号由 SUM 阶段确定：已有文档最大版本号 + 1
- 新文档从 v1 开始

### 节点内版本出现规则（Mode B）

- 只有该节点的内容在本轮发生了变更，才在节点下追加 `### v{N}` 子节
- 没有变更的节点 → 不追加（不写"### v3 — 无变更"）
- 版本索引的"影响节点"列记录实际变更的节点

### 版本变更类型标签

| 标签 | 含义 | 示例 |
|------|------|------|
| 初始 | 该节点首次创建 | `### v1 — 2026-06-18（初始：登录性能优化需求）` |
| 变更 | 修改了已有内容 | `### v2 — 2026-06-20（变更：bcrypt cost factor）` |
| 补充 | 追加了新内容但未修改旧内容 | `### v3 — 2026-06-21（补充：Safari 兼容性需求）` |
| 修复 | 修复了问题 | `### v3 — 2026-06-21（修复：Safari 白屏问题）` |

## 6. Structured Merge 流程

Mode B 下，每次更新执行以下 6 步结构化合并：

### Step 1 — Fetch 现有文档

- 通过后端接口读取已有文档完整内容
- 本地文件（local_html/markdown）→ `Read` 文件
- 飞书文档（lark_doc）→ MCP lark-cli 读取

### Step 2 — 解析拓扑结构

- 识别文档模式（Mode A / Mode B / 混乱）
- 提取所有顶层节点及其子版本
- 提取版本索引表
- 记录当前最大版本号

### Step 3 — Topology Gate 预检

- 执行 7 项拓扑检查（见 `document-topology-gate.md`）
- 如命中违规 → 先执行结构修复（自动或用户确认）
- 修复后重新解析拓扑

### Step 4 — 按节点分配本轮内容

- 从 `all_step_outputs` 提取各步骤产出
- 将每个步骤的产出分配到对应流程节点下：
  - `read_output` → `## 1. READ` 下新增 `### v{N} — {date}（{变更类型}）`
  - `code_output` → `## 2. CODE` 下
  - `review_output` → `## 3. REVIEW` 下
  - `examine_output` → `## 4. EXAMINE` 下
  - `git_output` → `## 5. GIT` 下
  - `graph_output` → `## 6. GRAPH` 下
  - `sum_output` → `## 7. SUM` 下
  - `talk_output` → `## 8. TALK` 下
  - `selfcheck_output` → `## 9. SELFCHECK` 下
  - `ask_output` → `## 10. ASK` 下
- 被跳过的步骤不追加
- 未变更的步骤不追加（不写"无变更"占位）
- 首次创建该节点时使用"初始"标签

### Step 5 — 更新版本索引

- 在 `## 0. 版本索引` 表格追加新行（最新版本在最前）
- 标注本轮实际影响的节点
- 更新"当前版本"字段

### Step 6 — 标注过时内容

- 在节点内，旧版本小节前方追加 `[已过时 — v{N} 起]` 标记
- 不删除旧内容——保留演进历史
- 标记格式：

```
[已过时 — v3 起] 以下 v1 结论已在 v3 中更新——Safari 17 以下版本需额外 fallback。

### v1 — 2026-06-18（初始）
原内容...
```

## 8. Mode A 新 Run 追加

Mode A 下，每次新任务在现有 Run 块**之前**插入新 Run：

1. 读取已有文档
2. 新 Run 块插入在版本索引之后、第一个已有 Run 之前
3. 更新版本索引表（新增 Run 行）
4. 历史 Run 块**不修改**

## 9. 自检清单

在 structured merge 之后、write 之前：

- [ ] v2.10 文档模式已确定（Mode A / Mode B）并记录到 `document_mode`
- [ ] Mode B：不存在重复顶层流程节点
- [ ] Mode B：不存在 `## 📝 更新记录` 孤儿块
- [ ] Mode B：本轮内容已按节点分配（非集中在更新记录块）
- [ ] 版本索引已更新，影响节点标注正确
- [ ] 版本索引版本号与节点内实际版本一致
- [ ] 过时内容已标注版本标记（`[已过时 — v{N} 起]`）
- [ ] 历史内容保留（未删除），与当前内容通过版本小节区分
- [ ] Mode A：新 Run 块格式正确，在版本索引之后
- [ ] 🆕 v2.11 write_mode 已确定（quick_fix / normal / full_archive）并记录
- [ ] 🆕 v2.11 document_scope 已确定（round_report / engineering_active）并记录
- [ ] 🆕 v2.11 已有文档未使用 overwrite（除非 full_archive 三合法场景）
- [ ] 🆕 v2.11 Quick Fix 仅追加版本索引行+节点内版本小节，未覆写已有内容
- [ ] 🆕 v2.11 round_report 未试图 overwrite engineering_active

## 10. 反模式

- ❌ Mode B 下追加 `## 📝 更新记录` 块而非节点内版本子节
- ❌ Mode B 下出现两个 `## 1. READ` 节点
- ❌ 新版本内容全部堆在"更新记录"里，节点内无新版本
- ❌ 不 fetch 现有文档直接覆盖写入——破坏已有结构
- ❌ 版本号跳跃或重复（v1, v3, v5 但缺 v2, v4 需说明原因）
- ❌ Mode B 下所有节点都追加 v{N}（包括无变更的节点）——应只追加有变更的节点
- ❌ 旧内容直接删除不标注——丢失演进历史
- ❌ 🆕 v2.11 Quick Fix 时 overwrite 整篇文档——只能追加版本记录
- ❌ 🆕 v2.11 把 round_report 当 engineering_active 写入——作用域混淆
- ❌ 🆕 v2.11 已有文档不 fetch 直接 overwrite——无基准覆写
- ❌ 🆕 v2.11 Normal 模式产出摘要级内容——内容退化

## 11. Write Mode Classification（写入模式三级分类）🆕 v2.11

### 模式总览

| 模式 | 名称 | 适用场景 | 写入行为 | 可否 Overwrite |
|------|------|---------|---------|---------------|
| **quick_fix** | 快速记录 | 微调（typo/config/样式）、<5 行改动、无新增逻辑 | 仅追加：版本索引 +1 行 + 受影响节点下 `### v{N}` 小节 | **NEVER** |
| **normal** | 正常更新 | 标准功能开发/Bug 修复/重构（**DEFAULT**） | Full structured merge：Fetch→Parse→Pre-check→Distribute→Index→Mark→Verify | 仅限完整 merge 后 |
| **full_archive** | 全量重建 | 首次创建 / 用户明确要求重写 / 已有全文完整 merge | 全量写入，附带 preservation checklist + post-write fetch-compare | 仅限三合法场景 |

### 模式选择决策树

```
新任务 → 是否有已有产物文档？
  ├── 否 → 新文档 → normal（首次创建，实质等同 full_archive 合法场景①）
  └── 是 → Fetch 已有文档 → 判断任务规模 →
        ├── 微调任务（task_type=微调 或 report_depth=brief）→ quick_fix
        │     └── 仅追加版本记录，不覆写已有内容
        ├── 标准任务 → normal（DEFAULT）
        │     └── 执行完整 structured merge
        └── 用户明确说"重写整篇""全量重建""完整重写" →
              └── 已 fetch 当前全文 + 完成 merge → full_archive
              └── 未 fetch → 先执行 fetch + merge，再决定
```

### Quick Fix 详细行为

**允许**：
- 版本索引表追加一行新版本记录
- 受影响节点下新增 `### v{N} — {date}（{变更类型}）` 小节
- 更新版本索引"当前版本"字段
- 如涉及修改已有结论，旧版本小节前加 `[已过时 — v{N} 起]` 标记

**禁止**：
- 删除/修改任何已有版本小节的内容
- 删除/合并已有流程节点
- 重建版本索引表（只能追加行）
- 修改文档标题或元信息

> 完整写入模式定义、决策树、行为矩阵详见 `skills/sum-session/references/document-preservation-gate.md` §4。

## 12. Document Scope（文档作用域）🆕 v2.11

### 两种作用域

| 作用域 | 用途 | 可短？ | 可覆写正式文档？ |
|--------|------|--------|-----------------|
| **round_report** | 当前轮摘要——聊天回复、快速沟通、敏捷站会 | YES（1-2 页） | **NEVER** |
| **engineering_active** | 长期工程档案——Code Review/交接/审计 | NO（保留全部历史） | 仅限 full_archive 三合法场景 |

### 混用阻断规则

- round_report 内容（< 500 字 + 无代码摘录 + 无版本小节）→ 不得写入 engineering_active 路径
- engineering_active 不得因本轮改动小而被缩减（Quick Fix → 追加版本记录，已有内容不变）
- 用户说"记录一下"→ Agent **必须追问**：记录为 round_report 还是更新 engineering_active

### round_report 格式

简短但必须包含：
1. 本轮做了什么（1-2 句）
2. 关键代码位置（至少 1 处含路径+行号）
3. 结果如何（测试通过/验收确认）
4. 标注 `⚠️ 本报告为轮次摘要（round_report），非正式工程文档。`

> 完整作用域定义、混用阻断规则详见 `skills/sum-session/references/document-preservation-gate.md` §5。

## 13. Anti-Overwrite Rules（禁止无基准覆写）🆕 v2.11

| # | 规则 | 说明 |
|---|------|------|
| R1 | **已有文档默认局部更新** | 对已有正式工程文档，默认 Normal（structured merge），禁止把本轮总结当整篇 overwrite |
| R2 | **Overwrite 必须附带 Preservation Checklist** | 执行 overwrite 前必须完成 7 项 checklist，逐项打勾 |
| R3 | **Quick Fix 禁止覆写正式文档** | 小修小补只允许追加版本索引行 + 节点内新增版本小节 |
| R4 | **round_report 不得 overwrite engineering_active** | 当轮摘要只能独立记录或聊天回复 |
| R5 | **Overwrite 仅限三种场景** | ①首次创建 ②用户明确要求重写整篇 ③已拿当前全文做完整 merge |
| R6 | **Overwrite 后必须 fetch-compare** | overwrite 后立即 fetch 回读，对比退化→回滚+阻断 |

> 完整规则、检测方法、修复策略详见 `skills/sum-session/references/document-preservation-gate.md`。
