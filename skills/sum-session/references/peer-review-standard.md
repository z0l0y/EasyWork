# 同行审查就绪标准（Peer Review Readiness）

> **🆕 v2.9**：最终报告写入前，Agent 必须自问 6 个问题。任一答案为否 → 报告不允许写入（detailed 模式 HARD GATE）。
> 核心原则：**另一个工程师不看聊天记录、不读源码、不问你，就能理解、复盘、审查、接手。**

---

## 1. 六问自检

### 问题 1：另一个工程师能否根据报告复现问题？

**通过标准**（必须全部满足）：
- [ ] 报告中有问题发生的具体现象描述（错误信息/日志/行为异常）
- [ ] 报告中有触发条件（什么输入/什么操作/什么环境会导致问题）
- [ ] 报告中有复现步骤（具体到命令/操作序列/输入数据）
- [ ] 如果问题不可稳定复现，说明了概率/条件和已知模式

**不通过示例**：
- ❌ "用户反馈登录慢"——没有时间/数量/延迟数据，无法复现
- ❌ "偶发 500 错误"——没有说什么时候偶发、什么条件下偶发

**通过示例**：
- ✅ "6月15日起 APM 显示 login API P99 延迟从 300ms 升至 2.8s。复现：在高峰期（12:00-14:00）POST /api/auth/login，body 含有效用户名+密码，响应时间 >2s 的概率约 40%。非高峰期（凌晨 3:00-5:00）正常（<500ms）。定位到 bcrypt cost factor=14 在单次验证上消耗 800ms × 3次调用 = 2.4s。"

### 问题 2：能否定位到改动代码？

**通过标准**（必须全部满足）：
- [ ] 每个变更文件有完整路径（非仅文件名）
- [ ] 每个变更文件有改动的函数/方法名
- [ ] 每个变更文件有行号范围
- [ ] 代码摘录标注了路径和行号（格式：`auth/auth.go:192-198`）

**不通过示例**：
- ❌ "修改了 auth.go"——哪个 auth.go？哪个函数？哪几行？
- ❌ "在登录逻辑中增加了重试"——没有位置，无法定位

**通过示例**：
- ✅ "auth/auth.go:192-198 — Login() 函数：将 `close(ch)` 改为发送 `io.EOF` 信号，由 `mainLoop()` 统一关闭 channel。auth/auth_test.go:45-67 — 新增 `TestScannerEOF` 测试。"

### 问题 3：能否知道为什么这么改？

**通过标准**（必须全部满足）：
- [ ] 有改动原因说明（不是"按需求修改"——是"为什么这里需要改"）
- [ ] 有关键逻辑解释（3-6 句话，读得懂核心思路）
- [ ] 有替代方案说明（≥1 个未被采用的方案+不被采用的技术理由）
- [ ] 方案选择有技术推理链（非"这样最简单"）

**不通过示例**：
- ❌ "按需求将超时时间从 5s 改为 10s"——不知道为什么是 10s 不是 15s
- ❌ "采用方案 A 因为简单"——"简单"不是技术理由

**通过示例**：
- ✅ "为什么在这里改不在别处改：token 过期判断在整个认证链路中有 3 处——网关层 `gateway/auth.go`、服务层 `auth/auth.go`、客户端 `client/token.go`。修改 auth/auth.go 而非 gateway/auth.go 是因为：1) 这是 token 验证的单一数据源（Single Source of Truth），2) 网关层改需要同步更新 3 个微服务的调用方。为什么用方案 A（bcrypt factor=10）不用方案 B（SHA-256）或 C（加缓存）：见 CODE §4 方案对比表。"

### 问题 4：能否知道测试覆盖了什么？

**通过标准**（必须全部满足）：
- [ ] 有完整可复制的测试命令
- [ ] 有测试覆盖矩阵（测试用例 → 覆盖场景 → 结果）
- [ ] 有测试输出凭据（关键片段，非仅"通过"）
- [ ] 新增的测试标注了"为本次改动新增"
- [ ] 如果有失败前/修复后对比 → 两者都有

**不通过示例**：
- ❌ "测试全部通过"——无命令、无输出、无覆盖矩阵
- ❌ "go test ./..."——有命令但无输出凭据

**通过示例**：
- ✅ "测试命令：`go test -v -run 'TestAuth|TestToken' ./auth/...`"
- ✅ "覆盖矩阵：TestLoginHappy — 正常登录路径 — passed / TestLoginExpired — token 过期重试 — passed（本次新增）/ TestLoginWeakNet — 弱网降级 — passed（本次新增）"
- ✅ "关键输出：`=== RUN TestLoginExpired --- PASS: TestLoginExpired (0.32s) --- PASS: TestLoginWeakNet (1.05s)`"
- ✅ "失败前（v1.0）：TestLoginExpired 直接 401 → FAIL。修复后（v1.1）：重试3次后降级为匿名访问 → PASS。"

### 问题 5：能否知道还有什么没有覆盖？

**通过标准**（必须全部满足）：
- [ ] 有未覆盖风险列表（≥1 项）
- [ ] 每项说明了为什么未覆盖
- [ ] 每项有风险评估（影响面+概率）
- [ ] 有后续计划或诚实标注"暂无计划"

**不通过示例**：
- ❌ "后续持续优化"——没说优化什么
- ❌ 报告没有风险/限制章节

**通过示例**：
- ✅ "未覆盖：1) Safari < 17 CSS Grid fallback 打印布局（低频，<0.5%用户，v1.2评估）。2) 极弱网（<100KB/s）token 刷新重试3次可能不够（暂未模拟此环境，风险缓解：重试失败降级为匿名访问，用户可手动登录）。3) 并发刷新 token 的竞态条件（已知理论风险，当前 QPS <100 时概率 <0.001%，监控告警已配置）。"

### 问题 6：能否判断这次改动能不能合并？

**通过标准**（必须全部满足）：
- [ ] 有明确的验收通过/未通过结论
- [ ] 有审查结论（verdict + 关键发现）
- [ ] 有质量评分（v2.9+）
- [ ] 有已知风险对合并的影响评估
- [ ] 如果有"不合并"的理由 → 明确列出

**不通过示例**：
- ❌ "代码已就绪"——没有给出判断依据
- ❌ 审查结论缺失

**通过示例**：
- ✅ "合并建议：✅ 可以合并。审查结论 pass_with_fixes（发现并修复了 Scanner EOF 处理问题）。质量评分 87/100。已知风险 3 项均为低频/有缓解措施/监控已配置。建议在合并后观察 APM 中的 login P99 指标 24h。"

## 2. 判定规则

| 深度 | 闸门类型 | 不通过处理 |
|------|---------|-----------|
| **detailed** | **HARD GATE** | 任一问题答案为"否" → 报告不允许写入 → 标注不通过的问题编号 → 回退补充 |
| **standard** | **SOFT GATE** | 不通过时展示不通过清单 → 用户确认后写入 |
| **brief** | 跳过 | 6 问不适用——brief 报告本身定位为摘要，不做工程追溯 |

## 3. 自检输出格式

```
【同行审查就绪自检】

| # | 问题 | 结论 | 证据位置 |
|---|------|------|---------|
| 1 | 能否复现问题？ | ✅ | SUM §2 发现过程 — 含 APM 数据+复现步骤 |
| 2 | 能否定位代码？ | ✅ | CODE §1 文件变更表 — 含路径+函数+行号 |
| 3 | 能否知道为什么？ | ✅ | CODE §3 设计说明 + §4 方案对比表 |
| 4 | 能否知道测试覆盖？ | ✅ | EXAMINE §1-4 — 命令+矩阵+凭据+对比 |
| 5 | 能否知道未覆盖？ | ✅ | EXAMINE §5 未覆盖风险 |
| 6 | 能否判断合并？ | ✅ | SUM §5 效果 + §6 未来 + 质量评分 |

总体判定：✅ 6/6 通过 — 报告满足同行审查就绪标准
```

---

## 4. 反模式

- ❌ 跳过自检直接写入——detailed 模式此为 HARD GATE
- ❌ 问题答案含糊（"大概能""应该能"）——这不是"能"还是"不能"的判断
- ❌ 把"能"的标准降到"勉强能找到"——标准是"不读源码、不问你"就能理解
- ❌ brief 模式假装满足——brief 报告天然不满足同行审查就绪，必须标注免责声明
