# 总结报告检查清单

本文档提供 SUM 阶段六要素的详细检查指导，确保总结报告完整、准确、有用。

## 要素一：背景

- [ ] 已说明业务背景——这段代码在哪个系统中、服务于什么业务
- [ ] 已说明触发原因——是事故驱动、需求驱动还是技术债务驱动
- [ ] 已说明不修改的后果——让读者理解这件事的紧迫性和重要性
- [ ] 没有使用"优化"、"改进"等模糊词汇代替具体的业务问题描述

**反面示例**：
> "为了提升用户体验，我们对登录模块进行了优化。"
> — 为什么优化？用户遇到了什么问题？不优化会怎样？

**正面示例**：
> "线上监控发现登录接口 P99 延迟达到 2.8 秒，超过 SLA 阈值（1 秒）。每日约有 3000 名用户在高峰期遇到超过 3 秒的登录等待，导致客服收到大量投诉。根因定位到密码加密算法使用了同步的 bcrypt 高强度哈希（耗时 800ms/次），且未做缓存。"

## 要素二：发现过程

- [ ] 已说明最初的触发信号（谁在什么情况下发现了问题）
- [ ] 已描述排查路径——从现象到根因的推理链
- [ ] 排查路径是逻辑连贯的，不是跳跃式的
- [ ] 已说明排除了哪些可能性以及排除的原因
- [ ] 如果有关键证据（日志、截图、数据），已附上

**排查路径示例**：
> ```
> 用户反馈登录慢 → 检查 APM 发现 auth/login 接口 P99=2.8s
>   → 初步疑数据库慢 → 检查慢查询日志，auth 表查询仅 12ms，排除
>   → 疑网络延迟 → 检查同机房内部调用延迟，<5ms，排除
>   → 加入代码级耗时打点 → 发现 bcrypt.compare() 单次耗时 800ms
>   → 确认根因：使用 cost factor=14 的 bcrypt，安全性高但耗时过长
> ```

## 要素三：问题说明

- [ ] 已区分"直接原因"和"根本原因"
- [ ] 直接原因是技术层面的近因（哪行代码、什么条件触发了问题）
- [ ] 根本原因挖掘了更深层的原因（如设计缺陷、配置不当、流程缺失）
- [ ] 已说明影响范围（功能、用户、时长、数据）
- [ ] 问题描述是技术语言而非情绪语言

## 要素四：解决方案

- [ ] 已说明最终采用的方案及其技术细节
- [ ] 已说明为什么选这个方案（技术权衡、时间约束等）
- [ ] 如果考虑过其他方案，已说明不选的原因
- [ ] 方案描述与前面 CODE 阶段的变更清单一致

**方案选择理由示例**：
> "选择降低 bcrypt cost factor 从 14 到 10（而非换成更快的算法如 SHA-256）的理由：
> (1) bcrypt 是业界公认的密码哈希标准，安全性有保障
> (2) cost factor=10 在安全性和性能之间取得平衡（单次约 60ms，仍在暴力破解的安全阈值内）
> (3) 不引入新的哈希算法，兼容已有密码数据，无需密码重置"

## 要素五：最终效果

- [ ] 已引用 EXAMINE 阶段的测试结果
- [ ] 验收标准逐项对照，每项都有验证方式
- [ ] 如果有可量化数据（性能对比、错误率对比），已提供
- [ ] 没有使用"应该没问题了"、"大概修好了"等不确定表述

**量化效果示例**：
> | 指标 | 修改前 | 修改后 | 变化 |
> |------|--------|--------|------|
> | 登录接口 P99 延迟 | 2.8s | 0.32s | ↓ 88.6% |
> | 单次密码验证耗时 | 800ms | 60ms | ↓ 92.5% |
> | 登录成功率 | 97.2% | 99.8% | ↑ 2.6% |

## 要素六：未来展望

- [ ] 诚实地列出了本次的局限性
- [ ] 已标注临时方案和技术债务，并给出后续改进计划
- [ ] 后续建议是具体的、可执行的（有负责人、有时间线）
- [ ] 没有使用"我们会继续优化"等空泛的承诺

**具体建议示例**：
> "(1) 上线后观察登录延迟和成功率指标 1 周（责任人：张三，告警阈值：P99>500ms）
>  (2) 下个迭代评估引入 Redis 缓存登录失败计数，避免每次错误都查库（责任人：李四）
>  (3) 补充不同 cost factor 下的性能基准测试文档，供后续安全审计参考（本周五前完成）"

## 质量红线

如果以下任一条不满足，SUM 阶段视为未完成：
1. 六要素中有任一要素完全空白
2. 背景部分只描述了"做了什么"而没有"为什么做"
3. 方案说明没有解释"为什么选择这个方案"
4. 效果部分没有客观证据支撑
5. 未来展望完全是空话套话（如"继续优化"、"持续关注"），没有具体行动
