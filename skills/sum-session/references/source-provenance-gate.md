# Source Provenance Gate（来源出处闸门）

> **铁律 #37**：每个版本内容必须标注来源类型（完整记录/摘要恢复/重构推断/外部引用）。如为摘要恢复→必须声明"该版本为摘要恢复，非逐字原始记录"。禁止文档伪装完整却实际残缺。
> 此闸门与 Historical Version Coverage（铁律#36）互补——#36 检查"有没有内容"，#37 检查"内容从哪里来"。

---

## 1. 来源分类

### 完整记录（full_record）

**定义**：内容来自该版本的完整步骤产出（step_outputs），逐步骤保留，未经删减。

**标注格式**：
```markdown
### v3 — 2026-06-21（变更：修复登录超时）
> 📋 来源类型：完整记录
> 来源步骤：READ/CODE/REVIEW/EXAMINE 完整产出
```

**判定条件**：
- 内容 ≥ read_output + code_output + review_output + examine_output 等完整步骤产出的 80% 字数
- 包含代码摘录（文件路径+行号）、测试命令+输出、审查发现+修复记录
- 非从摘要/回忆/重构恢复

### 摘要恢复（summary_recovery）

**定义**：内容从轮次摘要（round_report）、SUM 概要、或对话摘要中恢复——不是完整的步骤产出。

**标注格式**：
```markdown
### v2 — 2026-05-10（修复：Token 刷新）
> ⚠️ 来源类型：摘要恢复
> ⚠️ 该版本为摘要恢复，非逐字原始记录。
> 原始会话日期：2026-05-10
> 恢复来源：round_report（.claude/easywork/round_reports/2026-05-10.md）
> 缺失内容：CODE 节点缺少代码摘录、EXAMINE 节点缺少测试命令输出
```

**判定条件**：
- 内容 < 该任务级别应有的 MCR 内容要求的 50%
- 缺少代码摘录（只有文字描述"修改了 X 文件"）
- 缺少测试命令输出（只有"测试通过"）
- 从 round_report/对话摘要/Agent 记忆中恢复

### 重构推断（reverse_engineered）

**定义**：内容从代码 diff/commit message 反向推断——当时没有记录，后来根据代码变更推测。

**标注格式**：
```markdown
### v1 — 2026-01-15（初始：WebSocket 实现）
> 📝 来源类型：重构推断
> ⚠️ 该版本内容从 git diff + commit message 反向推断，非当时的开发记录。
> 推断来源：commit abc1234（"feat: add WebSocket support"）
> 不确定项：CODE 节点的实现细节是从 diff 推断的，可能遗漏当时的讨论和取舍
```

**判定条件**：
- 内容来源于 git log/diff/PR description 而非当时的 step_outputs
- Agent 根据代码变更"还原"当时的实现过程
- 必然不完整——diff 只能看到代码变化，看不到决策过程

### 外部引用（external_ref）

**定义**：内容引用自外部文档（Issue/PR/Wiki/设计文档），非 EasyWork 自身产出。

**标注格式**：
```markdown
### v3 — 2026-06-21（补充：性能测试结果）
> 🔗 来源类型：外部引用
> 来源：性能测试报告（https://wiki.company.com/perf-test-20260621）
> 引用日期：2026-06-21
> 引用范围：仅 EXAMINE 节点的性能测试数据部分
```

**判定条件**：
- 内容来自外部 URL/文档/Issue
- 不是 EasyWork 工作流产出的直接产物

## 2. 出处标注格式

### 2.1 每个版本小节必须标注来源

在 Mode B（持续维护）下，每个节点内的 `### v{N}` 子节开头必须包含来源标注行：

```markdown
### v{N} — {date}（{变更类型}）

> {来源标注块}

{版本内容}
```

### 2.2 来源标注块格式

```markdown
> 📋 来源类型：{full_record | summary_recovery | reverse_engineered | external_ref}
> {如为 summary_recovery} ⚠️ 该版本为摘要恢复，非逐字原始记录。
> 来源{步骤/会话/URL}：{具体来源路径}
> {如有缺失内容} 缺失内容：{列出缺失的步骤/证据类型}
> {如有不确定项} 不确定项：{列出不确定的内容}
```

### 2.3 标注完整性检查

| 检查项 | 条件 |
|--------|------|
| 每个 `### v{N}` 有来源标注 | ✅ 必查 |
| 来源类型为四种之一 | ✅ 必查 |
| summary_recovery 必须含 "该版本为摘要恢复，非逐字原始记录" | ✅ 必查（HARD GATE） |
| 来源路径具体可追溯 | ✅ 必查（不能写"来源：历史记录"） |
| 缺失内容已列出（summary_recovery/reverse_engineered） | ✅ 必查 |

## 3. 全文来源声明

在文档开头的元信息区（版本索引下方），新增全文来源声明：

```markdown
### 文档来源声明

本文档各版本内容的来源类型分布：

| 来源类型 | 版本数 | 说明 |
|---------|--------|------|
| 完整记录 | v3, v4 | EasyWork 完整步骤产出 |
| 摘要恢复 | v2 | 从 round_report 恢复，非逐字记录 |
| 重构推断 | v1 | 从 git history 反向推断 |

> ⚠️ 标注为"摘要恢复"或"重构推断"的版本内容可能不完整。完整版本（v3+）由 EasyWork v2.12 生成。
```

## 4. Gate Judgment Rules

| 深度 | 闸门类型 | 命中处理 |
|------|---------|---------|
| **detailed** | **HARD GATE** | 任何版本小节无来源标注→阻断；summary_recovery 未声明"非逐字原始记录"→阻断 |
| **standard** | **SOFT GATE** | 警告但允许（标注"来源未经完整检查"） |
| **brief** | **SKIP** | 跳过 |

### 特殊处理

```
版本小节来源标注为 "完整记录" 但经检查实际为摘要恢复：
  → 升级为 HARD GATE——这是严重的来源造假
  → 纠正标注类型为 summary_recovery
  → 追加"非逐字原始记录"声明
  
无法确定来源（Agent 不知道该版本内容怎么来的）：
  → 标注为 "来源不明（unknown）"
  → 追加声明 "⚠️ 该版本来源无法确定。内容可能来自 Agent 记忆/对话摘要/或不完整恢复。"
```

## 5. 反模式

- ❌ **摘要恢复 2000 字的 CODE 章节但标注"完整记录"**：来源造假——摘要恢复不可能有完整的代码摘录和审查记录
- ❌ **所有版本都标注"完整记录"但实际来源混杂**：v1 明显是旧文档（EasyWork v1.0 时代），不可能有完整记录
- ❌ **来源路径写"见上文"**：不可追溯——必须写具体的文件路径/会话日期/URL
- ❌ **summary_recovery 不声明"非逐字原始记录"**：让读者误以为内容完整——这是最严重的反模式
- ❌ **外部引用不标注引用日期**：外部文档可能已更新或删除——标注日期可追溯当时的版本
