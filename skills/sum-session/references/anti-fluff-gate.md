# 水文检测闸门（Anti-Fluff Gate）

> **🆕 v2.9**：在 write_final_report 之前强制执行。检测报告中的"水文"——有结构无内容、有结论无推理、有标题无证据。
> 核心原则：**产物不是过程打卡表，而是可供他人复盘、审查、接手的工程记录。**

---

## 1. 水文定义

"水文"是指满足以下任一条件的报告内容：
- 有形式（标题、分段、表格）但无实质（具体数据、推理过程、证据引用）
- 可以被复制到任何项目的任何报告中而不违和
- 删掉后不影响读者对本次改动内容的理解

---

## 2. 六类水文信号

### 类别 1：空泛结论（Fluffy Conclusions）

**检测模式**：扫描全文，匹配以下关键词或语义等价表述

| 水文信号 | 为什么是水文 | 必须替换为 |
|---------|-------------|-----------|
| "功能正常" | 没有说怎么验证的、验证了什么路径 | 列出测试命令+关键输出片段+覆盖了哪些场景 |
| "逻辑清晰" | 没有说怎么判断的、清晰在哪 | 列出具体审查点+代码位置+为什么认为逻辑正确 |
| "代码质量较好" | 无标准、无证据、无对比 | 引用具体审查维度的检查结果 |
| "提升用户体验" | 无数据、无对比 | 量化前后差异（P99延迟从X降到Y）或说明具体改善点 |
| "增强稳定性" | 无具体措施、无风险评估 | 列出具体修复的边界条件+异常处理路径 |
| "符合最佳实践" | 没有说是哪个最佳实践、怎么符合的 | 引用具体规范/模式名+说明代码中如何体现 |
| "后续可以优化" | 没有说优化什么、什么时候、谁来做 | 列出具体遗留问题+建议方案+优先级 |
| "持续改进"/"不断优化" | 空洞无物的万能结尾 | 诚实列出："本次未覆盖X，因为Y，后续Z版本中处理" |
| "经过严谨分析" | 套话——删掉不影响理解 | 直接说分析过程和结论 |
| "综上所述" | 套话模板 | 删除或替换为具体结论 |

**检测示例**：
- ❌ "正确性：通过，逻辑正常，功能符合预期。"
- ✅ "正确性：main.go:192-198 — 确认 `Scanner.EOF` 时不再直接 `close(ch)`，改为发送 `io.EOF` 到 channel 后由读取方关闭——避免主循环在 IDE 非交互模式下读到 closed channel 后退出。对 `io.EOF`/`context.Canceled`/`os.Signal` 三种退出路径逐一验证。回归测试：`TestScannerEOF` — 输入 `\x04` (EOT) 后程序 200ms 内退出且退出码为 0。"

### 类别 2：无证据通过（Evidence-Free Pass）

**检测模式**：出现"通过""pass""正常""无问题""OK"等词，但前后 3 行内无命令/输出/代码位置/数量统计

| 水文信号 | 为什么是水文 | 必须替换为 |
|---------|-------------|-----------|
| "测试通过"（无命令+输出） | 不知道跑了什么、怎么过的 | 测试命令 + 关键输出片段 + 总数/通过/失败统计 |
| "审查通过"（无检查点） | 不知道看了哪里、怎么判断的 | 每维度≥2条具体检查点 + 代码位置 + 审查依据 |
| "问题已修复"（无复现+回归） | 不知道修复前后对比 | 修复前表现 + 修复后表现 + 回归测试结果 |
| "go vet 通过"（无输出） | 不知道是否有 warning、怎么确认的 | 完整命令 + 输出（即使是空的也标注"输出为空=通过"） |
| "review 通过"（无检查点） | 没有说看了什么 | 同上——必须列出具体检查点 |

**检测示例**：
- ❌ "测试全部通过，go vet 无报错，代码质量符合要求。"
- ✅ "测试命令：`go test -v -run TestAuth ./...`，耗时 2.3s，12 passed / 0 failed / 0 skipped。`go vet ./...` 输出为空（=通过）。测试覆盖矩阵见下表。"

### 类别 3：无位置描述（Location-Free Claims）

**检测模式**：出现文件名/模块名/代码实体，但无文件路径+函数名+行号（三要素缺一不可）

| 水文信号 | 为什么是水文 | 必须替换为 |
|---------|-------------|-----------|
| "修改了 auth.go" | 不知道哪个 auth.go、改了哪个函数、哪几行 | `auth/auth.go:192-198 — Login() 函数` |
| "某处有问题" | 无法定位 | 文件名+函数名+行号范围 |
| "错误处理有遗漏" | 不知道在哪遗漏的 | `handler.go:45 — 未检查 json.Unmarshal 的 err 返回值` |
| "边界情况未覆盖" | 不知道哪个边界、在哪 | "空输入在 `parseInput()` 中未处理：`input=""` 时 `strings.Split()` 返回 `[""]` 导致后续逻辑把空字符串当有效值" |

**检测规则**：正文中提到代码实体（文件/函数/模块/组件/接口）但首次出现时未附带位置信息 → 命中

### 类别 4：无取舍（No Trade-off Analysis）

**检测模式**：描述了采用的方案，但未提及任何替代方案或未解释选择理由

| 水文信号 | 为什么是水文 | 必须替换为 |
|---------|-------------|-----------|
| "采用了方案A" | 不知道为什么选A不选B/C | 列出≥1个替代方案 + 每个方案不被采用的具体技术理由 |
| "这样改最简单" | "简单"不是技术理由 | 说明为什么简单在技术上是合理的（而非偷懒） |
| "按需求实现" | 没有独立判断，只是执行 | 说明为什么这种实现方式匹配需求 |

**检测示例**：
- ❌ "采用 bcrypt cost factor=10，因为简单且安全。"
- ✅ "方案对比——A) bcrypt factor=10：单次 60ms，安全性在暴力破解安全阈值内。B) SHA-256 加盐：单次 0.5ms 更快，但不符合 OWASP 密码存储推荐。C) 保持 factor=14 但加缓存：引入缓存一致性风险。选择 A——在性能和安全间取平衡点，且兼容已有 bcrypt 哈希格式，用户无需重置密码。未选 B 因为 OWASP 明确推荐 bcrypt/argon2 而非通用哈希；未选 C 因为缓存层增加运维复杂度且无法解决首次登录慢的问题。"

### 类别 5：无风险（Risk-Free Claims）

**检测模式**：报告未提及任何未覆盖场景、副作用、假设前提、已知限制

| 水文信号 | 为什么是水文 | 必须替换为 |
|---------|-------------|-----------|
| "没有问题" / "无风险" | 没有改动是真正零风险的 | 说明为什么风险低（具体理由）或列出已知限制 |
| 报告末尾无风险/限制章节 | 遗漏了"不知道的事" | 至少1条未覆盖风险或已知限制 |
| "后续持续关注" | 万能结尾——没说关注什么 | 列出具体遗留问题+建议方案+优先级 |
| "测试已全覆盖" | 没有测试能真正全覆盖 | 诚实地列出已知未覆盖的边界/场景 |

**检测示例**：
- ❌ （报告末尾）"后续持续优化和完善。"
- ✅ "已知限制：1) Safari < 17 版本的 CSS Grid fallback 在打印时布局错位——因为 `@page` 伪类对 Grid 的支持在 WebKit 中不完整。低频场景（<0.5% 用户打印表单）。暂不修复，v1.2 中评估。2) 弱网环境（<100KB/s）下 token 刷新重试 3 次可能不够——当前未模拟极弱网场景测试。风险缓解：重试失败后降级为匿名访问（非 401 踢出），用户可手动重新登录。"

### 类别 6：SelfCheck 放水（SelfCheck Rubber-Stamp）

**检测模式**：SelfCheck 产出不满足 MCR+ 硬性要求

| 水文信号 | 为什么是水文 | 必须替换为 |
|---------|-------------|-----------|
| 问答 < 8 个 | 不够深入——形式主义 | ≥8 个具体追问（含跟进追问） |
| 无反事实问题 | 没有压力测试方案的边界 | ≥2 个"如果不用/如果失败/如果复现"问题 |
| 无替代方案质询 | CTO 放过了解释 | ≥2 个"为什么不用 X/Y"的质询 |
| 回答含"大概""应该""可能""比较好" | 模糊——未追问 | 每个模糊回答必须有跟进追问 |
| 无认知缺口 | 假装什么都懂 | ≥1 个认知缺口+补救动作 |
| "因为这样更合理" | 不是技术理由 | 替换为具体技术分析 |
| "改动小所以没问题" | 改动大小与风险无关 | 替换为具体的风险评估 |
| "测试通过所以没问题" | 测试通过≠没有未知问题 | 诚实地分析测试覆盖的边界 |

## 3. 执行流程

### 3.1 触发时机

SUM 步骤在调用 `write_final_report` **之前**，必须执行水文检测。

### 3.2 检测步骤

```
1. 加载本文档（anti-fluff-gate.md）
2. 获取待写入的完整报告内容（all_step_outputs 拼装后的全文）
3. 逐类扫描：
   a. 类别1（空泛结论）：全文关键词匹配
   b. 类别2（无证据通过）：语义扫描——"通过/正常"前后3行无命令/输出/位置
   c. 类别3（无位置描述）：语义扫描——代码实体出现但无位置
   d. 类别4（无取舍）：方案描述后无替代方案
   e. 类别5（无风险）：报告末尾无风险章节
   f. 类别6（SelfCheck放水）：selfcheck_output MCR+ 逐项检查
4. 汇总命中项 → 输出检测报告
5. 按判定规则处理
```

### 3.3 判定规则

| 深度 | 闸门类型 | 命中处理 |
|------|---------|---------|
| **detailed** | **HARD GATE** | 任一类别命中 → **拒绝写入** → 输出命中清单 + 定位到具体段落 → 回退对应步骤补充 → 重新检测（最多2轮）。2轮后仍未通过 → 挂起用户，列出已清除项和仍命中项 |
| **standard** | **SOFT GATE** | 命中时记录警告 → 展示命中清单 → 要求用户确认是否继续写入。用户可逐类豁免 |
| **brief** | 跳过 | 不执行水文检测，但报告头部标注 "⚠️ 简要摘要 — 未经水文检测，不适合工程追溯" |

### 3.4 豁免机制

用户可对特定类别声明豁免（仅在 standard 模式）：
- "允许空泛结论" → 类别1 SOFT GATE 通过
- "允许跳过替代方案" → 类别4 SOFT GATE 通过
- 豁免声明记录到 `sum_output.anti_fluff_result.waivers`

**detailed 模式不支持豁免**——所有 6 类必须通过。

## 4. 检测报告输出格式

```
【水文检测闸门】— report_depth={detailed/standard}

逐类扫描结果：

| 类别 | 结果 | 命中位置 | 处理 |
|------|------|---------|------|
| 1. 空泛结论 | ❌ 命中 | SUM §4 解决方案："功能正常，逻辑清晰" / §5 效果："提升用户体验" | 回退补充 |
| 2. 无证据通过 | ✅ 通过 | — | — |
| 3. 无位置描述 | ❌ 命中 | CODE §2 "修改了 auth.go" — 无函数名+行号 | 回退补充 |
| 4. 无取舍 | ✅ 通过 | — | — |
| 5. 无风险 | ❌ 命中 | 报告末尾无未覆盖风险章节 | 回退补充 |
| 6. SelfCheck放水 | ✅ 通过 | — | — |

总体判定：❌ fail — 3 类命中，回退补充后重新检测
```

## 5. 与 MCR+ 的关系

- **MCR（v2.7）**：定义"至少有什么"——数量要求
- **MCR+（v2.9）**：定义"质量有多好"——内容深度要求
- **Anti-Fluff Gate**：定义"什么是不可接受的"——质量底线

三者关系：MCR 是入场条件 → MCR+ 是质量标准 → Anti-Fluff Gate 是底线闸门

---

## 6. 反模式

- ❌ 跳过水文检测直接写入——detailed 模式此为 HARD GATE
- ❌ 命中后不补充内容，只是换一种说法重写——水文不是措辞问题，是内容缺失问题
- ❌ 在 brief 模式下不标注免责声明——读者不知道这是未经检测的摘要
- ❌ 把水文检测当成"措辞检查"——它检查的是"有没有证据/推理/风险"，不是"用词好不好"
