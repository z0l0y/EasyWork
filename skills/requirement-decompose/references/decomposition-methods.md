# 拆解方法指南（Decomposition Methods）

> 5 种拆解方法 × 决策树 × 完整示例 × 组合使用规则

## 核心原则

1. **纵向切片优先**——永远不按技术层横切
2. **方法不是互斥的**——一个需求可以先用方法 A 拆大块，再用方法 B 拆小块
3. **先 Spike 再拆**——如果有技术不确定性，第一个任务永远是探针

---

## 方法选择决策树

```
需求分析：
├── 团队用过这项技术吗？
│   └── 没有 → 🧪 方法 5「风险优先」——先做 Spike 探针
│
├── 需求有多个端到端用户场景吗？
│   └── 是 → 🔪 方法 1「纵向切片」——按用户价值线程切
│
├── 业务规则是逐层叠加的吗？
│   └── 是 → 📏 方法 4「规则拆解」——先最简版本→逐层叠规则
│
├── 需要处理多种数据格式/类型吗？
│   └── 是 → 📊 方法 3「数据拆解」——按数据子集逐步支持
│
├── 用户有多种操作路径吗？
│   └── 是 → 🛤️ 方法 2「路径拆解」——按不同路径切
│
└── 以上都不适用
    └── 默认 → 🔪 方法 1「纵向切片」
```

---

## 方法 1：纵向切片 🔪

### 什么时候用
- 需求有多个端到端用户场景（最常用）
- 每个场景能独立交付用户价值
- 例：支付功能 = 微信支付 + 支付宝 + 银行卡

### 怎么切
按用户价值线程，每片从 Controller → Service → DB 一层到底。

### 示例：支付功能

```
❌ 横向拆分（错误）：
  任务 1: 写支付 Model
  任务 2: 写支付 Service
  任务 3: 写支付 Controller
  → 每个任务都没有独立用户价值，必须全部做完才能用

✅ 纵向拆分（正确）：
  任务 1: 微信支付端到端（前端→后端→微信API→回调→DB）
  任务 2: 支付宝支付端到端
  任务 3: 银行卡支付端到端
  → 做完任务 1 用户就能微信支付了，任务 2/3 可并行
```

### 拆解规则
- 第一片 = 最重要的用户场景（80% 用户用到的）
- 每片必须包含：UI/FE → API → 业务逻辑 → 数据持久化 → 外部调用（如果涉及）
- 片与片之间：共享的抽象（支付接口）在第一片中定义，后续片只需实现接口

## 方法 2：路径拆解 🛤️

### 什么时候用
- 一个功能有多个不同的操作路径
- 每个路径的用户行为和系统响应不同
- 例：登录 = 密码登录 + 短信验证码登录 + 扫码登录 + 第三方登录

### 怎么切
按用户执行的不同操作路径，每条路径端到端。

### 示例：用户登录

```
任务 1: 密码登录（输入账号→验证密码→返回 token）
任务 2: 短信验证码登录（输入手机→发验证码→验证→返回 token）
任务 3: 扫码登录（生成二维码→手机扫码确认→轮询→返回 token）
任务 4: 第三方登录 OAuth（跳转→授权回调→创建/绑定账号→返回 token）
```

### 拆解规则
- 第一个任务 = 主路径（最常用、最简的路径）
- 后续任务 = 替代路径
- 共享基础设施（token 生成、session 管理）放在第一个任务中

## 方法 3：数据拆解 📊

### 什么时候用
- 功能需要处理多种数据格式/类型
- 每种数据类型可以独立支持
- 例：数据导入 = Excel 导入 + CSV 导入 + XML 导入 + PDF 解析

### 怎么切
按数据类型子集，先支持最常见/最简单的格式。

### 示例：报表导出

```
任务 1: 导出 CSV（最简格式——不需要样式，纯数据）
任务 2: 导出 Excel（加样式/多 sheet/公式）
任务 3: 导出 PDF（固定布局/分页/水印）
任务 4: 导出 HTML（可嵌入邮件/网页）
```

### 拆解规则
- 第一个数据格式 = 最常见/最简单的格式
- 提取共用抽象：Convert() 接口 → CSV实现 → Excel实现 → PDF实现
- 每个任务完成后，用户就能导出那种格式

## 方法 4：规则拆解 📏

### 什么时候用
- 业务规则复杂且是逐层叠加的
- 最简版本本身就有用户价值
- 例：购票 = 基础购票 → 加优惠券 → 加会员折扣 → 加限购 → 加选座

### 怎么切
先做最简版本（MVP），然后逐层叠规则。每一次叠加都是可独立验证的。

### 示例：促销活动创建

```
任务 1: 基础促销（创建活动→设置折扣—固定金额/百分比→应用范围→生效时间）
         ✅ 做完就能创建最简单的"全场 8 折"活动
任务 2: 叠加条件规则（满减/满折/阶梯价、商品/品类/用户级别限定）
         ✅ 创建"满 200 减 50，仅 VIP 用户"的活动
任务 3: 叠加互斥规则（同一商品不可同时享受两个促销、优先级策略）
         ✅ 防止"一个商品同时打 8 折又满 200 减 50"
任务 4: 叠加时间规则（预热/抢购/限时秒杀、库存扣减策略）
         ✅ 支持"前 100 件 5 折，100 件后恢复原价"
```

### 拆解规则
- 第一个版本 = 最简可用的版本（哪怕只有一种优惠类型）
- 每个后续版本 = 新增一种规则 + 验证不与已有规则冲突
- 每次叠加后：已有功能不能退化（回归测试必须 PASS）

## 方法 5：风险优先 🧪

### 什么时候用
- 团队没用过这项技术（新技术栈/新框架/新第三方服务）
- 需求中有重大技术不确定性
- 需求涉及多个高风险领域（安全/支付/合规/大数据量）

### 怎么切
第一个任务永远是探针（Spike）——不计入正常开发时间。
探针目标是"消除不确定性"而非"完成功能"。

### 示例：用 WebSocket 实现实时协作编辑

```
任务 0: 🧪 Spike 探针——WebSocket 技术验证（！不计入正常开发时间）
  目标：回答以下问题：
  - WebSocket 在现有架构下能稳定维持 1000 并发连接吗？
  - 消息延迟在可接受范围内吗（<100ms）？
  - OT/CRDT 哪种算法更适合我们的场景？
  - 移动端 WebSocket 连接稳定性如何？
  产出：技术验证报告 + 最小 PoC（两人同时编辑一个文本框）
  AC：上述 4 个问题全部有明确答案 + PoC 可运行

任务 1: 基础协作——实时文本同步（基于 Spike 结论选 OT/CRDT）
任务 2: 用户光标/选区同步
任务 3: 冲突解决 + 撤销重做
任务 4: 离线编辑 + 重连同步
```

### 拆解规则
- 探针任务有时限——约定 N 小时/天，时间到还不确定 → 暂停，汇报，决策
- 探针任务的 AC 不是"功能完成"而是"不确定性消除"
- 如果探针发现技术不可行 → 提前终止，避免在不可能的需求上浪费时间

## 方法组合使用

实际项目中很少只用一种方法。通常是组合：

```
例：电商秒杀功能
  Step 1: 🧪 风险优先 — Spike 验证 Redis 能否扛住 10万 QPS
  Step 2: 🔪 纵向切片 — 切出"普通商品下单"和"秒杀商品下单"两个端到端线程
  Step 3: 📏 规则拆解 — "秒杀下单"内部：基础秒杀→加限购→加防刷→加排队

例：B端报表系统
  Step 1: 📊 数据拆解 — Excel 报表 → PDF 报表 → 在线看板
  Step 2: 📏 规则拆解 — Excel 报表内部：基础数据导出→加样式→加多Sheet→加公式
```

## 横向拆解为什么是反模式

```
❌ 横切（按技术层）：
  任务 1: 数据库表设计
  任务 2: 后端 API
  任务 3: 前端页面
  问题：
    - 做完任务 1 无法验证——不知道表设计对不对（没写 API 验证不了）
    - 做完任务 2 无法验证——前端没写，不知道 API 返回的数据前端能不能用
    - 任务间强依赖——必须串行，不能并行

✅ 纵切（按用户价值）：
  任务 1: 用户列表页面（FE→API→DB 全栈）——做完就能看
  任务 2: 用户详情页面
  任务 3: 用户编辑功能
  优势：
    - 每个任务做完都能独立验证
    - 任务之间软依赖或可并行
    - 第一片就能给用户看 → 快速反馈
```
