# Read Paper（论文阅读助手）

## 定位

**面向论文分享的论文阅读助手。**

你跟同事/老板/社区说"我来讲一下这篇论文"之前，这个 skill 帮你真正读懂它——
不只是翻译摘要，而是搞清楚：

- 论文**为什么**这么做（动机，不是"他们用了 XX 方法"）
- 解决了**什么**问题（精确的研究缺口，不是泛泛的"提升性能"）
- **难点**在哪里（为什么不是"显然可以这么做"？为什么别人没做？）
- **怎么**解决的（关键设计的直觉 + 为什么这样设计有效）
- **效果**如何（数字 + 对比 + 实验是否真的证明了作者的主张）
- 核心**图怎么看**（逐区域标注，分享时你能指着图讲）
- 分享时**该怎么讲**（故事线 + 关键转折点 + 预判听众问题）

**目标不是把论文变短，是让你能在分享中讲清楚。**

每一份报告都应该达到这个标准：**你拿着报告去讲，听众不需要读过原论文也能听懂。**

---

## 前置判断

### 什么时候用这个 skill

| 场景 | 触发词 |
|------|--------|
| 准备论文分享，需要深入理解 | "读论文 / 看论文 / 帮我读这篇论文 / read paper" |
| 看到一篇相关论文，想快速判断是否值得读 | "扫一眼这篇论文 / 快速看看这篇" |
| 论文中的图看不懂，需要解释 | "帮我看看这篇论文的 Figure 3" |
| 论文英文太长，需要中文深度解读 | "这篇论文帮我用中文讲清楚" |

### 什么时候不用

- 你只是想翻译摘要 → 直接让 Claude 翻译，不需要走这个 skill
- 你已经很熟悉这篇论文，只需要排版 → 用普通聊天
- 你需要的是文献综述/多篇对比 → 说"帮我对比这几篇论文"，Agent 会走多篇模式

## 输入方式

Agent 支持三种论文输入方式，按优先级自动选择：

### 方式 1：ArXiv URL（最推荐）

用户提供 ArXiv 链接（如 `https://arxiv.org/abs/2301.12345`），Agent 自动：

1. 通过 WebFetch 获取 arxiv.org 论文页面 → 提取标题/作者/摘要
2. 通过 WebFetch 获取 arxiv.org HTML 版本（`arxiv.org/html/...`）→ 获取全文
3. 如果 HTML 版不可用，尝试 PDF 文本提取

### 方式 2：用户粘贴论文文本

用户直接粘贴论文内容（全文或关键章节）。Agent 直接解析，不做额外获取。

如果用户只粘贴了部分（如只贴了 Method 和 Experiment），Agent 应明确告知哪些部分是推测的、哪些是有原文支撑的。

### 方式 3：本地 PDF 路径

用户提供 PDF 文件路径（如 `F:\papers\attention.pdf`），Agent 用 Read 工具读取 PDF 内容。

## 执行流程

### 阶段 1：获取论文内容

```
用户输入 → 判断输入类型 →
  ArXiv URL?: WebFetch 获取 HTML 全文 → 提取各章节文本
  粘贴文本?: 直接使用，标注来源为"用户粘贴"
  本地 PDF?: Read 工具读取 → 提取文本
→ 记录论文元数据（标题/作者/年份/会议/链接）
→ 如获取失败：明确告知用户，请求手动提供文字
```

**注意**：如果获取到的内容明显不完整（如只有摘要没有正文），必须告知用户而非假装读完了全文。

### 阶段 2：结构化解码

逐节处理论文内容，每一节产出：

```
【第 N 节：{节名}】
├── 本节目的：（这一节在论文论证链中起什么作用）
├── 关键信息：（3-5 条，用自己的话重述，非翻译）
├── 原文引语：（1-3 句直接引语，用于报告中的证据锚定）
└── 我的理解：（这一节在整体论证中的位置——它回答了什么问题？引出了什么新问题？）
```

**关键规则**：
- **不要逐句翻译**。提取论证结构，而非语言表面
- **每个关键论断锚定原文**。不只是"作者认为..."，而是"第 3 页第 2 段：'...'"
- **标记不确定**。没读懂的地方标注 `[未完全理解]`，不要猜

### 阶段 3：深度分析（输出 10 段报告）

按 §5 输出格式模板生成完整报告。每段的写作规则见下文。

### 阶段 4：分享就绪自检

报告写完后，Agent 逐项自检：

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 讲得清楚吗？ | 一句话总结 + 背景 → 问题 → 方法 → 效果这条线，不看原文能顺下来 |
| 2 | 图能解释吗？ | 每张核心图都逐区域标注了，分享时能指着图说 |
| 3 | 能被问倒吗？ | §10 预判问题覆盖了基础/挑战/延伸三个层次，每题有回答方向 |
| 4 | 有证据吗？ | 每个关键论断至少有 1 条原文引语或位置标注 |
| 5 | 没有水文吗？ | 全文搜索"新颖/创新/显著/效果好"等空泛词 → 如有，替换为具体描述 |

任一项不通过 → 补充后再输出。

## 输出格式模板

```
# 《{论文标题}》论文阅读报告

## 0. 论文速览

| 项目 | 内容 |
|------|------|
| 标题 | {原文标题} |
| 中文译名 | {建议中文译名} |
| 作者 | {第一作者 et al.，机构} |
| 年份/会议 | {YYYY / Conference Name} |
| 链接 | {ArXiv / DOI / 官网} |
| 论文类型 | 理论分析 / 系统设计 / 实证研究 / 综述 / 基准测试 |
| 代码开源 | 有(链接) / 无 / 部分 |
| 一句话贡献 | {本文的核心学术贡献，区别于"一句话总结"} |

> 论文类型判定依据：{一句话说明为什么判定为此类型}

## 1. 一句话总结

**{一句话说清楚这篇论文做了什么、怎么做的、为什么重要}**

写作规则：
- 不超过 80 字（中文）
- 必须包含三个要素：做了什么 + 怎么做的 + 为什么重要
- ❌ "本文提出了一种新颖的 X 方法来提升 Y 的性能"（空泛）
- ✅ "为了让 Y 在 Z 场景下不再受 W 问题困扰，本文把 X 建模成 A→B 的转化问题，用 C 机制保证 D 约束，首次在 {具体数据集} 上把 {具体指标} 从 72% 拉到 89%"

与 §0 "一句话贡献"的区别：贡献是对学术社区的增量，一句话总结是这篇论文的完整故事压缩。

## 2. 背景与动机

### 2.1 大背景
- 这个领域在解决什么问题？（用通俗语言，假设听众是非本领域技术人员）
- 这个问题为什么重要？（有具体数字更好：影响了多少人/多少钱/多少算力）

### 2.2 现有方案与缺口
- 目前主流做法有哪些？（按思路分类，不罗列论文名）
- 这些做法在哪里卡住了？（这是论文的"动机来源"）
  - 方案 A 的问题：{具体问题，最好有数字}
  - 方案 B 的问题：{具体问题}
  - ...
- 为什么这些问题不是"微改就能解决"的？（这是"研究缺口"的论证核心）

### 2.3 本文动机
- 作者的核心观察/假设是什么？（"作者发现..."而非"作者认为..."）
- 这个观察为什么之前没人注意到/没人做到？

写作规则：
- 背景要"往下打"，不要停留在领域常识
- 缺口要精确——"现有方法在 X 场景下性能下降"比"现有方法不够好"强 10 倍
- §2.2 是整篇报告论证链的关键：如果读者不理解"为什么之前的方法不行"，就不会理解"为什么这个方法有价值"

## 3. 核心问题与难点

### 3.1 研究问题（精确表述）
- 作者试图回答什么问题？（一句话，精确到"在什么条件下，什么对什么的影响"级别）
- 这个问题是新的还是已有问题的新视角？

### 3.2 为什么难？
- 难度来源 1：{技术难点——如"搜索空间连续，枚举不可行"}
- 难度来源 2：{工程难点——如"训练需要 1000 GPU，成本不可接受"}
- 难度来源 3：{理论难点——如"没有已知的收敛保证"}
- （如果只有一个难点，说明你是真读懂了）

### 3.3 作者的解决思路（直觉层面）
- 核心洞见（Insight）：{一句话——"作者发现/意识到..."}
- 这个洞见为什么能绕开上述难点？
- 用一个类比帮助理解（可选但强烈推荐）

写作规则：
- §3.2 是"分享转折点"——在这里听众应该感到"哦，确实不简单"
- §3.3 是"分享高潮点"——在这里听众应该感到"哦！原来可以这样"
- 如果 §3.2 和 §3.3 之间没有张力（"难"vs"巧妙"），说明你没读懂

## 4. 方法

### 4.1 核心思路（直觉层面，300 字以内）
用"如果...那么..."的因果关系链描述方法的核心逻辑。
不用术语，用大白话。目标是让非本领域的人也能理解思路。

### 4.2 关键设计点

对每个关键设计点，用"问题→解决→为什么有效"三段论：

**设计点 1：{名称}**
- 要解决的问题：{这个方法组件要解决的具体问题}
- 怎么做：{用大白话 + 必要时用伪代码/公式}
- 为什么有效：{机制层面的解释，不是"因为实验结果好"}
- 替代方案：{作者为什么没选其他做法？（如果有讨论）}

**设计点 2：{名称}**
（同上）

（通常 2-4 个关键设计点。如果超过 4 个，说明没抓住重点）

### 4.3 方法总览

如果有 overview 图，用文字描述整体流程：

```
输入 → [组件1: 做什么] → [组件2: 做什么] → ... → 输出
```

如果没有 overview 图，自己画一个文字流程图。

写作规则：
- 每个设计点的"为什么有效"必须回答到机制层面——不能停在"因为实验结果好"
- 如果某个设计是"直觉上显然"的，解释为什么之前没人做（可能有隐藏的工程难点）
- 公式只在必要时引入，且必须逐符号解释

## 5. 核心图解析

对论文的每张核心图（通常 3-5 张），逐张解析：

### Figure {N}：《{原图标题}》（第 {X} 页）

**这张图的目的**：{在论文论证链中，这张图要证明/展示什么？}

**逐区域解读**：
| 区域/坐标 | 看到了什么 | 代表什么含义 | 关键数字/趋势 |
|-----------|-----------|-------------|-------------|
| 左上 | {描述} | {含义} | {数字} |
| 右上 | {描述} | {含义} | {数字} |
| 左下 | {描述} | {含义} | {数字} |
| 右下 | {描述} | {含义} | {数字} |
| 横轴 | {是什么} | {范围} | — |
| 纵轴 | {是什么} | {单位} | — |
| 颜色/线型 | {红色=?...蓝色=?...} | {图例含义} | — |

**从这张图能读出的关键信息**：
1. {信息 1——比较了什么？趋势是什么？}
2. {信息 2——有什么意外/反直觉的发现？}
3. {信息 3——如果只看这张图不读论文，会误解什么？}

**分享讲解要点**：
- 指着哪里讲：{先指哪里，再指哪里}
- 关键一句话：{这张图最想说的那句话是什么}
- 听众可能困惑的地方：{哪里容易看岔}

写作规则：
- **图解析是 read-paper 的核心差异化能力**——不是"如图展示了实验结果"，而是"你现在看着这张图，我来告诉你每一块是什么意思"
- 如果论文没有图（理论论文常见），标注"本文为理论论文，无实验图"，转而解析正文中的定义/定理结构
- 如果图太多（>10张），选最核心的 3-5 张，其余的列表简述

## 6. 实验与效果

### 6.1 实验设置
- 数据集：{名称/规模/为什么选它}
- 基线方法：{对比了哪些方法/为什么选这些基线}
- 评估指标：{用什么指标/为什么这个指标合理}
- 实验环境：{硬件/框架/超参（如有）}

### 6.2 核心结果

| 实验/对比 | 数据集/设置 | 本文方法 | 最强基线 | 提升 | 备注 |
|----------|-----------|---------|---------|------|------|
| {实验1} | {设置} | {数字} | {数字} | {+X%} | {一句话解读} |
| {实验2} | {设置} | {数字} | {数字} | {+X%} | {一句话解读} |
| ... | ... | ... | ... | ... | ... |

### 6.3 消融实验
- 每个组件的贡献是多少？（去掉组件 A → 指标降 X%）
- 哪个组件贡献最大？哪个贡献最小？（意外吗？）
- 如果没有消融实验，标注"⚠️ 本文未提供消融实验，无法判断各组件的独立贡献"

### 6.4 实验说服力评估

| 维度 | 评估 | 理由 |
|------|------|------|
| 数据集是否代表真实场景？ | ✅ / ⚠️ / ❌ | {理由} |
| 基线是否足够强？ | ✅ / ⚠️ / ❌ | {是否缺少重要基线？} |
| 指标是否能全面衡量目标？ | ✅ / ⚠️ / ❌ | {是否有重要维度未衡量？} |
| 消融实验是否证明了每个组件必要？ | ✅ / ⚠️ / ❌ | {哪个组件的贡献未被证明？} |
| 统计显著性是否报告？ | ✅ / ⚠️ / ❌ | {是否有标准差/置信区间/p值？} |
| 总体说服力 | 强 / 中 / 弱 | {一句话总结} |

写作规则：
- 核心结果表必须用**具体数字**，不能只写"大幅提升"
- 每个数字都要标注"这个数字意味着什么"（如"+17% 意味着从每周 3 次宕机降到每两周 1 次"）
- 如果作者报的数字有水分（选了最好的 seed / 只在部分数据集上好），必须指出

## 7. 局限性与适用边界

### 7.1 作者自述局限
- 从论文的 Discussion / Limitation / Future Work 中提取
- 标注作者是否诚实（有些局限是"客气"，有些是"真问题"）

### 7.2 审稿人视角的潜在问题
- 方法论层面的问题：{假设是否过强？方法在什么条件下可能失效？}
- 实验层面的问题：{实验结果是否能推广？是否存在数据泄露？}
- 工程层面的问题：{是否能实际部署？成本是否可接受？}

### 7.3 适用/不适用的场景

| 适用场景 | 不适用场景 |
|---------|-----------|
| {场景1：什么条件下这个方法效果好} | {场景1：什么条件下效果差/不可用} |
| {场景2} | {场景2} |

写作规则：
- §7.2 不是你"挑刺"，而是帮分享者准备面对质疑
- 如果论文有明显的局限但作者没提，这是要在 §7.2 重点指出的
- §7.3 是给分享者的"免责声明"——知道这篇文章的边界在哪里

## 8. 关键概念速查（中英对照）

| 英文术语 | 中文翻译 | 一句话解释（面向非本领域读者） |
|---------|---------|--------------------------|
| {term1} | {中文} | {用大白话解释，不超过 30 字} |
| {term2} | {中文} | {同上} |
| ... | ... | ... |

写作规则：
- 收录 5-15 个关键术语，覆盖本文最核心的概念
- 解释要用"类比"而非"定义"——如"类似拼图时先找四个角"
- 如果术语在不同上下文中有不同含义，标注本文用的是哪个含义

## 9. 分享大纲

### 9.1 故事线选择

| 故事线类型 | 适合场景 | 本文推荐 |
|-----------|---------|---------|
| 悬念式（问题→为什么难→别人做不好→我们的方案→效果） | 方法创新为主 | ✅/— |
| 递进式（背景→简单方案→发现问题→改进→再发现问题→最终方案） | 多轮迭代优化 | ✅/— |
| 对比式（方案A vs 方案B→各自优劣→本文融合→互补效果） | 融合/统一框架 | ✅/— |

推荐理由：{为什么推荐这个故事线}

### 9.2 分享大纲（12-18 页建议）

| 页码 | 类型 | 核心信息 | 参考来源 |
|------|------|---------|---------|
| 1 | 标题页 | {论文标题 + 一句话钩子} | — |
| 2 | 背景 | {这个领域在解决什么问题，为什么值得关注} | §2.1 |
| 3 | 问题 | {现有方法在哪里卡住了} | §2.2 |
| 4 | 本文洞察 | {作者的核心发现/洞见} | §3.3 |
| 5 | 方法总览 | {Overview 图 + 一句话思路} | §4.1 + §4.3 |
| 6-8 | 关键设计 | {设计点 1 / 2 / 3，一个设计一页} | §4.2 |
| 9 | Figure {N} 详解 | {最核心的实验结果图} | §5 |
| 10-11 | 实验结果 | {主要结果 + 消融实验} | §6 |
| 12 | 局限性 | {适用边界} | §7 |
| 13 | Take-away | {三句话带走 + 对你工作的启发} | — |
| 14 | 讨论/Q&A | {预判问题提示} | §10 |

### 9.3 关键转折点

- **"啊哈"时刻 1**：在讲到§3.2（为什么难）→ §3.3（核心洞见）的时候，听众应该感到"哦！"
- **"啊哈"时刻 2**：{如果有——如某个反直觉的实验结果}
- **最容易丢失听众的地方**：{如方法细节过多时，建议此处插入一个类比/例子}
- **如果只有 5 分钟**：只讲 §2.2(缺口) → §3.3(洞见) → §4.1(思路) → §6.2(核心数字) → Take-away

写作规则：
- 分享大纲是给**你**用的，不是给听众看的——所以要有操作指引
- 每页的"核心信息"必须是一句话能说完的——如果说不完，这页信息过载
- "如果只有 5 分钟"是救命行——老板临时改时间的时候用

## 10. 预判问题（附建议回答方向）

### 10.1 基础问题（没读论文的人会问的）
| # | 问题 | 建议回答方向 |
|---|------|-------------|
| 1 | 这篇论文到底做了什么？ | {用 §1 一句话总结回答，然后展开到 §4.1} |
| 2 | 跟 {知名基线} 比有什么优势？ | {指向 §6.2 的具体数字} |
| 3 | 这个方法可以用在 {你的场景} 吗？ | {指向 §7.3 的适用/不适用场景} |

### 10.2 挑战性问题（同行会质疑的）
| # | 问题 | 建议回答方向 |
|---|------|-------------|
| 1 | {最可能被质疑的点——如"提升是否来自更多的参数量？"} | {回答策略 + 指向论文中哪个实验/分析在回应} |
| 2 | {第二可能被质疑的点} | {回答策略} |
| 3 | {第三可能被质疑的点} | {回答策略} |

### 10.3 延伸问题（"如果...会怎样"）
| # | 问题 | 建议回答方向 |
|---|------|-------------|
| 1 | 如果把这个方法用在 {相关领域} 呢？ | {分析可迁移性} |
| 2 | 如果数据量/算力翻倍，这个方法还能涨点吗？ | {分析 scaling 特性} |
| 3 | 这个方法最大的改进空间在哪里？ | {指向 §7 + 你的判断} |

写作规则：
- 基础问题要"覆盖面广"——确保分享者被问到基础问题时不慌张
- 挑战性问题要"尖锐"——宁可多预判几个没人问的，也不要漏掉真正尖锐的
- 建议回答方向要**具体**——不是"解释一下方法就行"，而是"指向 §4.2 设计点 2，强调对比度损失函数的选择理由"

A) 让我展开某一节（如"详细讲一下 §5 的 Figure 3"）
B) 让我调整分享大纲（如"我只有 10 分钟，帮我压缩到 8 页"）
C) 让我补充某个角度的分析（如"安全性/隐私角度这篇论文有考虑吗？"）
D) 让我检查报告中的某个判断是否有原文支撑
E) 开始准备你的分享——告诉我你的听众是谁，我帮你调整话术
F) 导出报告到文件（你说"保存到 XX.md"）

```

## 反模式

- ❌ 把论文翻译了一遍就叫"论文阅读报告"——这是翻译器，不是阅读助手
- ❌ 方法部分写"使用了 XX 网络 + YY 损失函数"——说了用了什么，没说为什么这么设计
- ❌ 关键论断没有原文锚点——"作者认为这个方法更好"（哪里说的？第几页？）
- ❌ 图解析写"如图 3 所示，本文方法优于基线"——图解析要逐区域标注，不是复述图标题
- ❌ 概念速查写成词典定义——应该是面向非本领域读者的类比式解释
- ❌ 预判问题全是"这个方法有什么优点"级别的软问题——要有挑战性问题
- ❌ 分享大纲是"分几个部分"的标题列表——要精确到每页讲什么、核心信息是什么
- ❌ 遇到读不懂的内容假装读懂了——标注 `[未完全理解]` 比瞎猜强
- ❌ 对论文全文只读了摘要和结论就写报告——必须至少过一遍每个主要章节
- ❌ 实验数字只抄不解读——"+3.2%"没意义，除非你说"3.2% 意味着 X"
- ❌ 输出报告后没有主动提供后续交互选项——Agent 输出完就走，不管用户下一步想干嘛
- ❌ 中文报告夹杂大量英文术语不翻译——术语首次出现必须中英对照，后续可用中文简称

## 多篇论文模式

如果用户说"帮我对比这几篇论文"或同时提供多篇论文：

1. 每篇论文产出简版报告（§0-§6，省略 §8-§10）
2. 新增"跨论文分析"章节：
   - 共同主题 vs 各自关注点
   - 方法对比表（维度/方法/论文A/论文B/论文C）
   - 互相引用关系（谁基于谁？谁反驳谁？）
   - 这个方向的研究趋势（从这几篇看，领域在往哪个方向走？）
3. 如果用于分享，额外输出"对比式分享大纲"
