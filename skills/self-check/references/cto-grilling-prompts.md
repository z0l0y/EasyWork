# CTO 拷打提示词模板

> 用于 self-check 技能 · 四阶段拷打 · 可直接复用

## 阶段 1：业务背景拷打

```
1. 这个问题的业务影响是什么？用数据说话（影响用户数/金额/转化率）。
2. 为什么这个问题值得现在修，而不是下个迭代？
3. 如果不修，最坏情况是什么？
4. 有没有更简单的替代方案（workaround）？
5. 这个问题是"真正的问题"还是"问题的症状"？
```

## 阶段 2：问题发现拷打

```
1. 你是怎么发现这个问题的？（监控告警/用户反馈/代码审查/测试失败）
2. 为什么之前没发现？是监控盲区还是测试覆盖不足？
3. 这个问题存在多久了？有历史数据佐证吗？
4. 同类问题是否在其他模块也存在？（横向排查）
```

## 阶段 3：解决方案拷打

```
1. 为什么选方案A而不是方案B？有对比数据吗？
2. 这个方案的代价是什么？（性能/复杂度/维护成本）
3. 如果方案失败，回滚路径是什么？
4. 有没有更简单的方案被你忽略了？
5. 至少想一个反事实（counterfactual）："如果反过来做会怎样？"
```

## 阶段 4：实现过程拷打

```
1. 这个改动如何验证"问题确实被修复了"？（具体测试用例）
2. 改动是否引入了新的风险？（回归测试覆盖）
3. 是否有监控/告警来捕获未来的同类问题？
4. 团队成员能否在 5 分钟内理解你的改动？
5. 如果 6 个月后有人问"为什么这样写"，你能从 commit 中找到答案吗？
```

## 反事实追问模板

```
"假设你不做这个改动，而是做 ______，会有什么不同？"
"如果你只有一半的时间，你会砍掉哪部分？"
"如果你有双倍时间，你会额外做什么？"
```
