# 提交拆分策略参考

## 拆分原则

### 核心原则：让人类审查者能逐个理解每个提交

拆分的目的是降低审查难度，而非追求拆分的"纯粹性"。一个好的拆分应该让审查者能在 5 分钟内理解一个提交单元的内容，而不需要同时在大脑中加载 20 个文件的上下文。

### 拆分粒度指南

- **最小粒度**：一个提交单元至少包含一个完整的功能点。不能把一个功能的实现和它的测试拆成两个提交（因为审查实现时需要看测试来理解预期行为）。
- **最大粒度**：一个提交单元不应超过 10 个文件。超过 10 个文件的提交会让审查者产生疲劳。
- **黄金法则**：每个提交单元必须能**独立编译通过**（不一定能独立运行，但至少不能因为缺少其他提交的代码而编译失败）。

## 常见拆分场景

### 场景一：新功能开发

典型的拆分结构：
1. **数据层**（如果涉及）：新增数据库 migration、Model 定义
2. **核心逻辑**：Service 层、业务规则、API 实现
3. **UI 层**：页面/组件、样式
4. **配置与路由**：路由注册、菜单配置、权限配置
5. **测试**：单元测试 + 集成测试

### 场景二：Bug 修复

典型的拆分结构：
1. **修复代码**：核心逻辑修复（通常只有 1-2 个文件，不需要再拆分）
2. **回归测试**：针对 bug 场景新增的测试用例
3. **如果修复涉及多个模块**：按模块拆分为多个提交

### 场景三：重构

典型的拆分结构：
1. **准备工作**：新增抽象层、抽取公共函数（不影响现有行为）
2. **迁移**：逐个模块迁移到新实现
3. **清理**：删除旧代码、移除废弃的依赖
4. **测试更新**：更新测试以匹配新实现

### 场景四：依赖升级

1. **依赖版本变更**：package.json / go.mod 等
2. **API 适配**：因依赖 API 变更而做的代码适配（如有）
3. **配置调整**：构建配置适配新版本

## 不推荐的拆分方式

### ❌ 按文件类型机械拆分
```
提交1: 所有 .ts 文件
提交2: 所有 .tsx 文件
提交3: 所有 .css 文件
```
**问题**：审查者无法从单个提交中理解完整的功能变更。

### ❌ 每个文件一个提交
```
提交1: src/services/user.ts
提交2: src/controllers/user.ts
提交3: src/routes/user.ts
...
```
**问题**：过于碎片化，审查者在提交之间来回跳转，消耗更多精力。

### ❌ 不拆分（All in One）
```
提交1: 所有 37 个文件的改动
```
**问题**：人类无法有效审查，大概率直接 Approve。

## 拆分决策流程

```
获取所有改动文件列表
    │
    ▼
判断：文件数是否 > 10？
    │
    ├── 否 → 判断：是否涉及多个维度？
    │         ├── 否 → 单提交，附详细说明
    │         └── 是 → 按维度拆分为 2-3 个提交
    │
    └── 是 → 按维度归类
              │
              ▼
         判断：核心逻辑文件 > 5？
              │
              ├── 是 → 按功能子模块进一步拆分核心逻辑
              └── 否 → 保持当前拆分
              │
              ▼
         检查：每个提交单元是否能独立编译？
              │
              ├── 否 → 合并有依赖关系的单元
              └── 是 → 拆分完成
```

## 实际 Git 操作命令

### 场景：当前工作区有 12 个文件改动，需要拆为 3 个提交

```bash
# 第1步：先将所有文件从暂存区移除（保留工作区改动）
git reset HEAD

# 第2步：只添加"配置/构建"维度的文件
git add package.json docker-compose.yml .env.example
git commit -m "chore: 更新依赖和部署配置

- package.json: 升级 axios 到 1.6.0
- docker-compose.yml: 新增 healthcheck 配置
- .env.example: 新增 PAYMENT_TIMEOUT 环境变量

风险：低（配置变更，不影响业务逻辑）
验证：docker-compose up 成功启动"

# 第3步：添加"核心逻辑"维度的文件（重点审查的提交）
git add src/services/auth.service.ts src/middleware/validator.ts
git commit -m "feat: 登录接口增加短信验证码登录

- auth.service.ts: 新增 loginBySms() 方法，支持手机号+验证码登录
- validator.ts: 新增短信验证码格式校验（6位数字）

⚠️ 重点审查：涉及核心登录流程，影响所有用户的登录行为
风险：中（新增登录方式，需确保与现有密码登录互不干扰）
验证：12个测试全绿，含新增的短信登录测试"

# 第4步：添加"测试/文档"维度的文件
git add tests/auth.service.test.ts README.md
git commit -m "test: 补充短信登录测试和接口文档

- auth.service.test.ts: 新增短信验证码登录的3个测试用例
- README.md: 更新登录接口文档，增加短信登录说明

风险：低（仅测试和文档）
验证：npm test -- --testPathPattern='auth' 全部通过"
```

### 实用技巧

**用 `git add -p` 实现同一文件的不同改动拆到不同提交**：
```bash
git add -p src/services/auth.service.ts
# Git 会逐个 hunk 询问是否暂存：
# Stage this hunk [y,n,q,a,d,e,?]?
#   y → 暂存此 hunk（放到当前提交）
#   n → 跳过此 hunk（留给后续提交）
#   e → 手动编辑此 hunk（精细拆分）
```

**如果已经做了一个大提交，想拆开**：
```bash
git reset --soft HEAD~1    # 撤销最近1个提交，改动回到暂存区
git reset HEAD              # 全部 unstaged
# 然后按上面流程重新分批 add + commit
```

**拆分完成后检查各提交是否独立可编译**：
```bash
git rebase -i --exec "npm run build" HEAD~3
# 对最近3个提交逐个执行 npm run build，确保每个都能编译通过
```

## HITL 交互指南

拆分方案生成后，在向用户展示时需要说明：
1. **拆分逻辑**：解释为什么这么拆分（让用户理解你的分类思路）
2. **审查指引**：标注哪些提交需要重点审查（通常是核心逻辑部分）
3. **灵活调整**：明确告知用户"如果你觉得拆分方式不合理，告诉我你期望的拆分维度，我来调整"
4. **混合文件警示**：如果有无法完美拆分的文件，必须醒目地标注出来
