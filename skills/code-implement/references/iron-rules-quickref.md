# 代码实现铁律速查

> 用于 code-implement 技能 · 来自 fullchain-dev-workflow 铁律系统

## 核心铁律（CODE 场景适用）

| # | 铁律 | 检查方式 |
|---|------|---------|
| 1 | **禁止过度设计** —— 只实现需求明确要求的功能 | 代码行数/新增文件 = 需求复杂度匹配？ |
| 2 | **禁止触碰范围外代码** —— 不改动需求未涉及的模块 | git diff 是否包含非目标文件？ |
| 3 | **保持现有风格** —— 命名/缩进/注释密度 = 已有代码 | 新代码是否"像"周围代码？ |
| 4 | **先写注释后写代码** —— 关键逻辑必须有注释先行 | 每个函数是否有 docstring？ |
| 5 | **零魔法数字** —— 所有字面量必须是命名常量 | grep `\d+` 是否全为常量引用？ |
| 6 | **向前兼容** —— 不破坏已有 API/接口签名 | 是否有 breaking change？ |
| 7 | **错误处理完备** —— 无裸 `except: pass` | 每个 except 是否至少记录日志？ |
| 8 | **无死代码** —— 不保留注释掉的代码块 | git diff 是否有被注释的代码？ |

## 代码产物门禁（提交前自查）

```
□ 需求追溯：每条改动可以追溯到具体的需求条目
□ 风格一致：命名/缩进/注释与周围代码一致
□ 无副作用：未改动需求范围外的文件
□ 错误处理：所有异常路径有明确处理（非静默吞掉）
□ 可测试：关键逻辑抽取为可单独测试的单元
□ 无硬编码：路径/URL/密钥使用配置或环境变量
```

## 反模式

- ❌ "顺便重构一下" → 应单独提重构 PR
- ❌ "先跑通再说" → 跑通 ≠ 完成，需过门禁
- ❌ "这个以后再优化" → 记录 TODO 并关联 issue，不可口头承诺
- ❌ "测试后面补" → TDD 原则：先写测试，再写实现
