# 代码实现指导原则

本文档提供 CODE 阶段的详细实施指导，帮助 Agent 在写代码时做出正确的技术决策。

## 核心原则

### 原则一：最小变更原则
- 优先选择改动最少文件、最少行数的方案
- 能用配置解决的，不改代码
- 能在现有函数里加一个参数的，不新建函数
- 能在现有文件里加一个函数的，不新建文件
- 能用一个 if 条件解决的，不引入新的控制流框架

### 原则二：一致性优先于"最佳实践"
- 团队习惯用 class 组件就继续用 class 组件，不要因为"函数组件更现代"就切换风格
- 团队习惯用回调风格就继续用回调，不要因为"async/await 更优雅"就切换
- 一致性比先进性更重要——降低团队的认知负担和维护成本

### 原则三：可读性优先于简洁性
- 三行直白的 if-else 比一行精妙的 reduce+三元运算符更好
- 有意义的中间变量比链式调用更好（方便断点调试）
- 代码是写给人看的，顺便能在机器上运行

## 代码风格适配指南

### 如何在现有项目中找到风格参照
1. **找到最近修改的文件**：`git log --oneline -5` 看最近几次提交，查看那些被修改的文件，它们代表了团队当前的风格偏好
2. **找到同类型文件**：如果要新增一个 Service，先看项目中其他 Service 的写法
3. **查看配置文件**：ESLint、Prettier、EditorConfig 等配置文件会告诉你团队的形式规范

### 常见风格点（到达新项目时逐一检查）
- [ ] 缩进方式（空格还是 Tab，几个空格）
- [ ] 引号风格（单引号还是双引号）
- [ ] 分号（加还是不加）
- [ ] 函数定义方式（function 声明 vs 箭头函数 vs 类方法）
- [ ] 导入方式（ES Module import vs CommonJS require）
- [ ] 文件命名（kebab-case vs camelCase vs PascalCase）
- [ ] 变量命名（camelCase vs snake_case）
- [ ] 错误处理方式（try-catch vs .catch() vs error-first callback）
- [ ] 日志方式（console.log vs 专用 logger vs debug 包）

## 中文注释规范

### 什么情况下必须写注释
1. **业务逻辑**：任何涉及业务规则的判断和处理，必须注释说明业务背景
2. **复杂算法**：非直观的算法逻辑，必须注释说明算法思路
3. **临时方案 / Hack**：明知不是最佳方案但因时间/兼容性等原因采用的临时修改，必须注释说明为什么这样做、计划什么时候改为正式方案
4. **边界条件处理**：对特殊情况的处理逻辑，必须注释说明处理的是什么特殊情况
5. **公共 API / 导出函数**：供其他模块调用的函数，必须有 JSDoc 风格的中文注释

### 什么情况下不需要写注释
1. 代码本身已经足够清晰的简单赋值和取值
2. 一眼就能看懂的简单判断（如 `if (user == null) return;`）
3. 和代码内容完全重复的"翻译型"注释（如 `// 设置 name 为 'John'` 对应 `name = 'John'`）

### 注释质量检查
- [ ] 注释是否解释了"为什么"而不只是"做了什么"？
- [ ] 如果用中文读出来是否通顺自然？
- [ ] 注释中的技术术语是否与项目其他部分一致？
- [ ] 是否有过时的、与代码实际行为不符的注释？

## 错误处理指导

### 什么时候需要处理错误
- 所有外部输入（API 调用、文件读取、用户输入）都需要错误处理
- 所有可能失败的操作（数据库操作、网络请求）都需要错误处理
- 内部函数调用如果可能抛出异常，调用方需要 catch

### 错误处理方式
- 使用项目已有的错误处理模式（错误码、异常类、Result 类型等）
- 错误日志必须包含足够的上下文信息（操作名称、关键参数值、错误堆栈）
- 面向用户的错误提示必须友好且有用（不暴露系统内部信息）
- 区分"可恢复的错误"（如网络超时，提示用户重试）和"不可恢复的错误"（如数据损坏，记录日志并通知开发团队）

## 性能意识

### 不需要过早优化的场景
- 数据量确定很小（<100 条）的循环操作
- 非高频触发的用户操作（如点击"保存"按钮）
- 管理后台的功能（用户量小，QPS 低）

### 需要关注性能的场景
- 循环内有 IO 操作（数据库查询、API 调用、文件读写）
- 高频触发事件（滚动、输入、拖拽）
- 大量数据渲染（超过 1000 条数据的列表）
- 面向 C 端用户的高并发接口
