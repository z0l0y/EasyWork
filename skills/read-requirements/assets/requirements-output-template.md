# 需求理解输出模板

## 1. 核心目标（Goal）

用 1-3 句话清晰描述本次任务要达成的最终结果。目标必须是可验证的，避免模糊表述。

> 示例：将用户登录接口的 P99 响应时间从当前的 800ms 优化到 300ms 以内，同时保持现有 12 个登录相关测试用例全部通过。

- 本次目标：

## 2. 修改范围（Scope）

| 文件/模块 | 修改类型 | 预计改动量 | 说明 |
|-----------|---------|-----------|------|
| path/to/file1.ts | 修改 | 小（<20行） | 修复 XX 逻辑的空指针判断 |
| path/to/file2.ts | 新增 | 中（50行） | 新增 YY 校验函数 |
| ... | ... | ... | ... |

## 3. 约束条件（Constraints）

### 技术约束
- {如：必须兼容 Node 14+，不能使用可选链操作符 ?.}
- {如：数据库查询必须使用项目已有的 ORM 封装，不能直接写 SQL}

### 业务约束
- {如：不能修改 User 模型的公共 API 签名，下游有 15 个模块依赖它}
- {如：本次改动不能影响现有的微信支付流程}

### 其他约束
- {如：必须在周五上线前完成}

## 4. 验收标准（Acceptance Criteria）

每条标准必须是二元的（满足 / 不满足），可以用测试、脚本或手动验证来判断。

- [ ] {可验证的标准 1：如"输入正确用户名和密码后，2 秒内跳转到首页"}
- [ ] {可验证的标准 2：如"输入错误密码 3 次后，账号锁定并提示'请 15 分钟后重试'"}
- [ ] {可验证的标准 3：如"现有的 12 个登录测试用例全部通过"}

## 5. 不做什么（Non-Goals）

明确排除以下内容，防止 AI 擅自扩大改动范围：

- {如：本次只修改登录逻辑，不碰注册、忘记密码、第三方登录等其他认证流程}
- {如：本次只优化后端接口，不动前端页面布局和样式}
- {如：不对数据库表结构做任何变更}

## 6. 澄清问题（如有）

如果对需求有任何不确定的地方，在此列出（不超过 3 个）：

1. {问题 1} — 当前理解：{你的推测}，请确认或纠正
2. {问题 2}
3. {问题 3}

> 注意：如果输入材料足够清晰，此部分可以为空。
