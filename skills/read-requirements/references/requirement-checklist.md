# 需求理解检查清单

本文档提供 READ 阶段的详细检查指导，帮助 Agent 确认需求理解的完整性和准确性。

## 输入解析检查

### 如果是文档类输入
- [ ] 已通读全文，识别了所有功能点
- [ ] 已区分"必须实现"、"建议实现"和"可选实现"
- [ ] 已提取非功能性需求（性能指标、安全要求、兼容性要求）
- [ ] 已识别文档中可能存在的自相矛盾之处

### 如果是图片类输入
- [ ] 已识别图片中的所有交互组件和状态变化
- [ ] 已将视觉信息转化为文字功能描述
- [ ] 已标注图片中模糊或不确定的部分（作为澄清问题提出）

### 如果是错误日志类输入
- [ ] 已定位报错的具体代码文件和行号
- [ ] 已阅读对应源代码，理解上下文
- [ ] 已分析报错的触发条件（什么输入导致了这个错误）
- [ ] 已将"修报错"转化为具体的代码修改目标

### 如果是现有代码类输入（"照这个风格重构"）
- [ ] 已提取参考代码的命名规范、代码风格、组织模式
- [ ] 已识别该代码中的设计模式和常用工具函数
- [ ] 这些风格信息将作为 CODE 阶段的约束

## 五要素完整性检查

### 核心目标
- [ ] 目标表述清晰、具体、可验证
- [ ] 没有使用"优化"、"改进"、"完善"等模糊动词（这些词缺乏可验证性）
- [ ] 如果用户没有给出明确验收标准，已基于常识推断并请用户确认

### 修改范围
- [ ] 涉及的文件/模块列表完整（允许有"待探索"的标注）
- [ ] 标注了修改类型（新增/修改/删除）
- [ ] 标注了预计改动量（小<20行/中50行/大100行+），帮助评估工作量

### 约束条件
- [ ] 技术约束已从项目配置（package.json、tsconfig.json 等）中提取
- [ ] 业务约束已从用户输入中提取
- [ ] 没有遗漏隐含约束（如"不能影响现有用户"这类文档未写明但显然存在的约束）

### 验收标准
- [ ] 每条标准是二元的（满足/不满足），不存在"部分满足"
- [ ] 每条标准有验证方式（测试、手动验证、监控指标等）
- [ ] 覆盖了正常路径和关键异常路径

### 不做什么
- [ ] 已将用户输入中"不在此次范围内"的内容明确记录
- [ ] 已将 Agent 容易"顺手"去做的关联改动明确排除
- [ ] 如果不确定某项是否在范围内，已作为澄清问题提出

## 常见遗漏检查

以下项目是 Agent 最容易在 READ 阶段遗漏的，需特别留意：

- [ ] **性能要求**：用户说"快点"但没有给出具体数字，需要追问
- [ ] **错误处理**：用户描述了正常流程但没有描述异常流程，需要补充推断
- [ ] **存量数据兼容**：新增字段/修改结构时，存量数据如何处理？
- [ ] **上下游影响**：修改公共模块时，是否考虑了所有调用方？
- [ ] **权限与安全**：新功能是否需要登录？不同角色的权限差异？
- [ ] **国际化/多语言**：新增文案是否需要多语言支持？

## 质量红线

如果以下任一条不满足，READ 阶段视为未完成：
1. 核心目标无法用一句话说清楚
2. 验收标准无法被客观判断是否满足
3. 修改范围完全不确定（全是指向整个项目的模糊范围）
4. 有超过 3 个无法自行解决的澄清问题
