# Requirement Traceability Matrix（需求可追溯矩阵）

> **铁律 #31**：READ 阶段必须生成需求可追溯矩阵（用户需求 → 验收标准 → 自动化测试 → 手动验证 → 状态）。禁止测试不覆盖真实用户痛点。矩阵必须机器可读且可追溯。

---

## 1. 矩阵定义

### 为什么需要

v2.11 之前，验收标准和测试之间存在断层：

| 问题 | 表现 |
|------|------|
| 写了测试但没覆盖痛点 | 用户抱怨"登录慢"，Agent 写了"登录按钮可点击"的测试 |
| 需求没有对应测试 | 需求说"支持 Safari"，但测试矩阵里没有 Safari 条目 |
| 测试没有对应需求 | 跑了一堆测试不知道验证了什么需求 |
| 状态不可追踪 | 不知道哪些需求已验证、哪些待验证、哪些无法验证 |

### 矩阵解决的问题

- **前向追溯**：每个需求 → 对应哪个测试在验证它
- **后向追溯**：每个测试 → 验证了哪个用户需求
- **覆盖缺口**：哪些需求还没有测试覆盖
- **状态追踪**：哪些已验证、哪些待验证、哪些不适用

---

## 2. 矩阵结构

### 五列定义

| 列 | 含义 | 示例 |
|----|------|------|
| **用户需求** | 用户真正要达成的业务目标（非技术实现） | "用户在 3 秒内完成登录" |
| **验收标准** | 可验证的标准（来自 delivery_definition） | "POST /api/login 响应时间 P99 < 500ms" |
| **自动化测试** | 自动化测试用例（路径或用例名） | "tests/login/perf.test.js: login_p99_under_500ms" |
| **手动验证** | 无法自动化的验证步骤 | "Chrome DevTools Performance 面板确认 LCP < 2.5s" |
| **状态** | 覆盖状态 | "covered" / "partial" / "uncovered" / "n/a" |

### 状态说明

| 状态 | 含义 | 允许继续？ |
|------|------|-----------|
| **covered** | 需求有对应测试且测试通过 | ✅ |
| **partial** | 需求部分覆盖——有测试但未覆盖全部验收标准 | ⚠️ 警告（需说明未覆盖部分） |
| **uncovered** | 需求无任何测试覆盖 | ❌ L2+→阻断，L0-L1→警告 |
| **n/a** | 此需求不需要此类验证（如纯视觉需求无需自动化测试） | ✅（需说明理由） |

## 3. 矩阵生成算法

### Step 1 — 从用户输入提取需求项

```
输入：用户原始消息 + READ 阶段的五要素（目标/范围/约束/验收/不做）
输出：需求项列表

提取规则：
  - 用户明确说的："支持 Safari 17" → 需求项
  - delivery_definition 中的验收标准 → 每条 = 至少一个需求项
  - 用户的业务目标（读业务目标，非技术目标）
  - 不要从技术实现倒推需求——需求是业务语言
```

### Step 2 — 为每项定义验收标准

```
给每个需求项分配至少 1 条可验证的验收标准：
  - 有明确阈值的：时间 < Xms, 失败率 < Y%, 覆盖 ≥ Z 个浏览器
  - 有明确行为的：输入 A → 输出 B
  - 有明确检查点的：文件 X 包含 Y，日志中出现 Z
```

### Step 3 — 为每项指定测试方法

```
对每条验收标准：
  - 可用命令/脚本自动验证的 → 填入"自动化测试"列
  - 需要人眼/手动操作的 → 填入"手动验证"列
  - 两者都需要的 → 两列都填
  - 无法验证的 → 标注"n/a"并说明原因
```

### Step 4 — 检查孤立项

```
检查：
  - 是否有需求行在"自动化测试"和"手动验证"两列都为空？→ 标记为"uncovered"
  - 是否有测试不关联任何需求？→ 多余测试，标注或移除
  - 是否有需求来源为"用户反馈"但未覆盖真实痛点？→ 追加
```

---

## 4. 用户痛点覆盖检查

### 痛点识别规则

每个需求项必须追溯到一个**真实的用户场景**：

| 需求来源 | 痛点识别方法 |
|---------|------------|
| 用户反馈 | 用户原话 → "用户无法完成 X 因为 Y" |
| 监控告警 | 告警内容 → "X 指标异常影响 Y 用户" |
| Bug Report | Issue 描述 → "用户遇到 X 导致 Y" |
| 测试失败 | 失败日志 → "场景 X 在条件 Y 下失败" |
| 需求文档 | PRD 用户故事 → "作为 X，我想 Y 以便 Z" |

### 反例检测

```
以下信号说明"需求不来自用户痛点"：
  - 需求原文 ="优化代码"、"重构模块"——没有用户场景
  - 需求只有技术术语无业务语言——"实现 Redis 缓存层" vs "用户查询结果在 1 秒内返回"
  - 验收标准 = "代码可运行"——没有用户可见的改善
```

## 5. 矩阵输出格式

Agent 在 READ 阶段末尾输出（在完成定义之后）：

```markdown
## 🔗 需求可追溯矩阵（Requirement Traceability Matrix）

| # | 用户需求 | 验收标准 | 自动化测试 | 手动验证 | 状态 |
|---|---------|---------|-----------|---------|------|
| R1 | 用户登录耗时 < 3 秒（P99） | POST /api/login 响应时间 P99 < 500ms | `tests/login/perf.test.js: login_p99_test` | — | covered |
| R2 | 密码错误时有明确提示 | 输入错误密码后页面显示"用户名或密码错误" | `tests/login/ui.test.js: wrong_password_message` | Safari/Chrome/Firefox 验证文案一致 | covered |
| R3 | Safari 17 上登录页正常显示 | Safari 17 上页面加载无白屏、无布局错位 | — | Safari 17 上手动验证登录流程 | uncovered ⚠️ |

### 覆盖统计

| 状态 | 计数 |
|------|------|
| covered | 2 |
| partial | 0 |
| uncovered | 1 (R3: Safari 17) |
| n/a | 0 |

### 未覆盖项说明

- **R3 (Safari 17)**：当前项目无 Safari 17 测试环境。需用户在 Safari 17 上手动验证。如不可用，降级为"已知差异"并记录。
```

## 6. Gate Judgment Rules

| 深度 | 闸门类型 | 命中处理 |
|------|---------|---------|
| **detailed** | **HARD GATE** | 有任何需求行状态为 "uncovered" 且风险等级 ≥ L2 → **拒绝进入 CODE** → 补充测试或声明豁免 |
| **standard** | **SOFT GATE** | 警告但允许继续（标注"有未覆盖需求"） |
| **brief** | **SKIP** | 跳过但标注"需求追溯矩阵未经完整检查" |

### 豁免规则

```
以下情况允许 "uncovered" 行存在：
  - 需求 = 纯人工验证类（如"UI 动画流畅"）
  - 已在"手动验证"列中标注具体验证步骤和负责人
  - 已说明无法自动化的原因
  - 用户确认接受此风险
```

---

## 7. 反模式

- ❌ **假矩阵**：所有行都标 "covered" 但没有写任何实际测试——矩阵只是摆设
- ❌ **需求全是技术术语**：R1 "实现缓存层"、R2 "重构认证模块"——不可追溯到用户价值
- ❌ **测试不覆盖痛点**：用户痛点 ="登录慢"，测试 ="登录按钮渲染正确"——测试与需求错位
- ❌ **矩阵 < 3 行**：一个功能开发任务只列了 1 行——明显的覆盖不足
- ❌ **状态不更新**：READ 阶段标注 "covered"，但 EXAMINE 阶段测试失败——状态未同步更新
- ❌ **"n/a" 滥用**：所有手动验证需求都标 "n/a" 以绕过 uncovered 阻断——必须说明每条 n/a 的理由
