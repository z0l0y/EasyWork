# Delivery Definition Gate（完成定义闸门）

> **铁律 #30**：READ 阶段必须输出结构化的"完成定义"——明确任务类型、可机器验证的完成标准、仅人工可验证的完成标准、需显式确认的高风险操作。不定义"做到什么程度算完成"就不可进入 CODE。
> 完成定义不是自由文本，是可逐项勾选的结构化清单。SUM 阶段交付验证清单从完成定义自动生成，逐项回溯。

---

## 1. 什么是"完成定义"

### 为什么需要

在 v2.11 之前，Agent 从用户需求直接跳到代码实现，中间缺少一个关键步骤——**明确定义"做到什么程度算完成"**。这导致：

| 问题 | 表现 |
|------|------|
| 理解偏差 | Agent 认为"功能能跑"就是完成，用户期望的是"带测试+文档+性能达标" |
| 验证断层 | CODE 和 EXAMINE 之间没有桥梁——测试跑了但不知道是否覆盖了用户真正关心的标准 |
| 风险盲区 | 没有人列出"哪些操作需要用户显式确认"——Agent 自主执行了高风险操作 |
| 交付争议 | 用户说"这不算完成"，Agent 说"代码已经写好了"——双方对"完成"的定义不一致 |

### 完成定义不是

- ❌ "功能正常"——不可验证
- ❌ "代码写好就行"——缺少测试、文档、部署等维度
- ❌ 自由文本描述——无法逐项勾选
- ❌ 用户需求原文照抄——Agent 必须做结构化提取

## 2. 六维完成定义结构

### 2.1 任务类型

```
task_type: "Bug修复" | "功能开发" | "重构" | "微调" | "纯文档" | "纯理解" | "纯审查"
```

从编排中枢的任务分类器继承。如分类器判断有误，此处可修正。

### 2.2 完成标准 — 机器可验证

这些标准可以通过运行命令、检查输出、解析结果来自动判定。

**格式要求**：每条标准必须包含——验证方法 + 预期结果 + 具体阈值（如有）。

**示例**：
```markdown
✅ 好：
- 运行 `npm test -- --testPathPattern=auth` → 全部 12 个测试通过，0 失败
- `curl -s -o /dev/null -w '%{time_total}' http://localhost:3000/api/login` → 响应时间 < 300ms
- `grep -r "TODO\|FIXME" src/` → 无新增 TODO/FIXME（与 git diff 对比）

❌ 差：
- 测试通过  ← 没说哪个命令、多少测试、什么算通过
- 性能达标  ← 没说哪个指标、阈值多少
- 代码没问题  ← 不可验证
```

### 2.3 完成标准 — 仅人工可验证

这些标准无法自动化，需要人来判断——如 UI 美观度、交互流畅度、文案准确性。

**格式要求**：每条标准必须包含——验证对象 + 判断标准 + 验证环境（如适用）。

**示例**：
```markdown
✅ 好：
- 登录页在 Chrome/Firefox/Safari 最新版上布局一致，无元素错位
- 错误提示文案清晰、无错别字、与产品文案风格一致
- 移动端（375px 宽度）下登录表单不被截断，按钮在首屏可见

❌ 差：
- UI 正常  ← 没说在什么浏览器、看什么
- 用户体验好  ← 不可判断
```

### 2.4 高风险操作需显式确认

这些操作不可由 Agent 自主执行——必须在执行前获得用户显式确认。

**识别规则**（继承铁律#11 + 风险等级 L3/L4）：
- 数据库 schema 变更、数据迁移、数据删除
- 权限模型变更
- 生产环境部署/发布
- 第三方 API 密钥配置
- 删除文件/资源
- 涉及 PII/敏感数据的处理

**输出格式**：
```markdown
| # | 高风险操作 | 风险说明 | 确认对象 | 执行时机 |
|---|-----------|---------|---------|---------|
| 1 | 修改 user 表索引 | 可能导致写锁，影响线上服务 | DBA + 值班负责人 | 部署前 1 天 |
| 2 | 删除废弃 API /v1/users | 需确认无外部调用方 | API 网关日志确认 | CODE 阶段前 |
```

### 2.5 不可交付条件

这些条件触发后，**即时阻断交付**——无论其他标准是否满足。

**示例**：
```markdown
- 任何安全漏洞（XSS/SQL注入/越权）未修复 → 阻断交付
- 生产环境测试失败 → 阻断交付
- 用户显式拒绝某项高风险操作但 Agent 仍执行了 → 阻断交付并回滚
- 数据迁移后数据条数不一致 → 阻断交付
```

### 2.6 证据要求

交付时必须附带的最低证据集。按风险等级递增。

| 风险等级 | 最低证据要求 |
|---------|------------|
| L0 | 1 条：代码位置引用或用户确认记录 |
| L1 | 2 条：测试输出 + 代码变更摘录 |
| L2 | 3 条：测试输出 + 代码变更摘录 + 验收对照 |
| L3 | 4 条：测试输出 + smoke test 凭据 + 环境矩阵 + 验收对照 |
| L4 | 5 条：全部上述 + dry-run 结果 + 备份方案 + 回滚方案 |

## 3. 完成定义输出格式

Agent 在 READ 阶段末尾输出（在需求理解确认之后）：

```markdown
## 📋 完成定义（Delivery Definition）

### 任务信息
| 字段 | 值 |
|------|-----|
| 任务类型 | Bug修复 |
| 风险等级 | L2 — 正常功能 |

### 机器可验证标准
- [ ] M1: 运行 `npm test -- --testPathPattern=login` → 全部 8 个测试通过
- [ ] M2: `curl /api/login` 返回 200 且响应时间 < 500ms
- [ ] M3: 新增回归测试 `login_timeout_spec.js` 在老代码上 FAIL、新代码上 PASS

### 仅人工可验证标准
- [ ] H1: 登录错误提示文案清晰无错别字
- [ ] H2: 密码输入框支持显示/隐藏切换

### 高风险操作（需显式确认）
- 无（本次改动不涉及高风险操作）

### 不可交付条件
- 现有 12 个测试中有任何失败 → 阻断交付
- 登录响应时间 > 800ms → 阻断交付

### 证据要求
最低 3 条证据：测试输出 + 代码变更摘录 + 验收标准对照
```

## 4. Gate Judgment Rules

| 深度 | 闸门类型 | 命中处理 |
|------|---------|---------|
| **detailed** | **HARD GATE** | delivery_definition 为空 / 只有"功能正常"等模糊表述 / 缺少机器可验证标准 → **拒绝进入 CODE** → 回退补全（最多2轮） |
| **standard** | **SOFT GATE** | 警告但允许继续（标注"完成定义不完整"） |
| **brief** | **SKIP** | 跳过但标注"完成定义未经完整检查，仅适用于低风险任务" |

### 空/无效定义检测

```
判定条件（任一命中即视为无效）：
  - machine_verifiable 为空（[] 或无 [ ] checklist）
  - machine_verifiable 中所有条目都含模糊词（"正常""没问题""通过""OK"且无具体命令/阈值）
  - 全部 6 个维度缺失 ≥3 个
  - high_risk_operations 非空但无具体确认对象
```

## 5. 交付验证清单（Delivery Verification Checklist）

### 5.1 生成时机

SUM 阶段，在质量评分之前，从 delivery_definition 自动生成。

### 5.2 生成算法

```
Algorithm: GENERATE_VERIFICATION_CHECKLIST

Input: read_output.delivery_definition
Output: 验证清单 Markdown

1. 初始化清单 = []
2. 对于 machine_verifiable 中每条标准:
     生成 checklist 项: "- [ ] M{N}: {标准原文} → 证据位置：{自动填入或待填}"
3. 对于 human_only 中每条标准:
     生成 checklist 项: "- [ ] H{N}: {标准原文} → 验证人：______ 验证日期：______"
4. 对于 high_risk_operations 中每条:
     生成 checklist 项: "- [ ] ⚠️ 高风险: {操作描述} → 确认人：______ 确认日期：______"
5. 追加不可交付条件检查:
     "- [ ] 不可交付条件未触发"
6. 追加证据样本数检查:
     "- [ ] 证据样本数 ≥ {evidence_requirement}（当前：{actual_count}）"
7. 输出验证清单
```

### 5.3 输出格式

```markdown
## ✅ 交付验证清单（Delivery Verification）

> 自动生成自 READ 阶段的完成定义。逐项回溯，确认每项"完成标准"是否真的完成了。

### 机器可验证标准
- [ ] M1: 运行 `npm test -- --testPathPattern=login` → 全部 8 个测试通过
  → 证据：`EXAMINE: test_output` 第 3-5 行，8 passed, 0 failed ✅
- [ ] M2: `curl /api/login` 返回 200 且响应时间 < 500ms
  → 证据：`EXAMINE: perf_test_output` avg=320ms ✅
- [ ] M3: 新增回归测试 `login_timeout_spec.js` 在老代码上 FAIL、新代码上 PASS
  → 证据：`EXAMINE: regression_test_log` FAIL on abc1234, PASS on def5678 ✅

### 仅人工可验证标准
- [ ] H1: 登录错误提示文案清晰无错别字 → 验证人：______ 日期：______
- [ ] H2: 密码输入框支持显示/隐藏切换 → 验证人：______ 日期：______

### 高风险操作确认
- 无高风险操作

### 不可交付条件
- [ ] 现有 12 个测试中有任何失败 → 未触发 ✅
- [ ] 登录响应时间 > 800ms → 未触发（实测 320ms）✅

### 证据样本数
- [ ] 证据样本数 ≥ 3 → 当前：5 条（TEST_OUTPUT:2, CODE_DIFF:2, METRIC:1）✅

### 总体判定
✅ 交付验证通过（5/5 机器标准通过，2 条人工标准待用户确认）
```

## 6. 反模式

- ❌ **"功能正常"作为完成标准**：最经典的反模式——不可验证、不可追溯、不可审计
- ❌ **只有机器验证项没有人工验证项**：UI 改动任务却只写"测试通过"——用户实际体验未覆盖
- ❌ **只有人工验证项没有机器验证项**：纯靠人眼看——没有自动化回归保护
- ❌ **交付验证与完成定义脱节**：定义了"登录 < 300ms"但验证时只写"性能测试通过"——没有对照原标准逐项勾选
- ❌ **高风险操作不标注**：Agent 在 CODE 阶段修改了数据库 schema 但 READ 阶段未标注为高风险——风险分类在后续步骤中被跳过
- ❌ **完成定义写完后不再回顾**：READ 阶段写完就扔——SUM 阶段不检查是否真的完成了
- ❌ **不可交付条件太宽泛**："有重大问题就阻断"——什么都没说。必须是具体的、可检测的条件
