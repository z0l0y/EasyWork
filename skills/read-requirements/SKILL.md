# Read Requirements（需求理解）

## 前置判断：什么时候需要这一步？

**必须执行**：
- 用户的需求描述超过一句话，存在多种理解的可能
- 输入包含多种材料（文档+代码+截图混合）
- 改动预计涉及 2 个以上文件
- 用户的核心目标不够明确（"优化一下"、"改改这个"）

**可以跳过**：
- 用户给出极其精确的指令（"把 `config.ts` 第 42 行的 `==` 改成 `===`"）
- 用户明确说"按我说的直接做，不用分析"

## 核心流程

### 第 1 步：收集所有输入材料

不要只看用户最后一条消息。回顾对话上下文，收集所有相关材料：

- **文档**：PRD、需求说明、API 文档、技术方案 → Read 全文
- **图片**：UI 设计稿、架构图 → 如模型有 Vision 能力，解析布局和交互
- **错误日志**：报错堆栈、监控告警 → 定位报错文件和行号，阅读对应源代码
- **现有代码**：用户指定的参考文件 → 提取风格、模式、命名规范
- **口语描述**：从对话中提取关键名词（模块名、功能名）和动词（新增/修改/删除/修复）

### 第 2 步：输出五要素

| 要素 | 核心问题 | 好坏对比 |
|------|---------|---------|
| **目标** | 这次到底要达成什么？ | ❌ "优化登录" → ✅ "将登录接口 P99 从 800ms 降到 300ms" |
| **范围** | 动哪些文件/模块？ | ❌ "改用户模块" → ✅ "仅改 `auth.service.ts` 的 `login()` 方法" |
| **约束** | 有什么不能碰的？ | ❌ 没列 → ✅ "必须兼容 Node14，不能改 API 签名" |
| **验收** | 怎么算做完了？ | ❌ "功能正常" → ✅ "输入 X → 输出 Y，现有 12 个测试全过" |
| **不做** | 哪些顺手的事要忍住？ | ❌ 没列 → ✅ "不修改注册流程，不调整数据库表结构" |

### 第 3 步：发现矛盾时挂起

如果输入材料之间有冲突（如 PRD 说用 Vue，代码库是 React），**立刻停止**：

```
【READ 阻断 - 材料矛盾】
发现：PRD 要求技术栈为 {X}，但实际代码库为 {Y}
影响：按 PRD 执行会导致技术栈不一致
请澄清：应该以哪个为准？
```

> **详细检查清单**：五要素的完整性和质量检查标准，见 `references/requirement-checklist.md`。

## 反模式

- ❌ 材料没读完就开始写需求分析
- ❌ 看到报错日志就跳过直接去改代码——先理解"这个报错意味着什么"
- ❌ 把用户的口语描述直接当验收标准——"快一点"不是验收标准，"P99 < 300ms"才是
- ❌ 发现矛盾不报告，自己猜一个就往下走
- ❌ 不列"不做什么"——这是防止后续步骤越界的关键约束

## 记住

理解偏差在这一步会被放大 9 倍。多花 2 分钟确认理解，比写完后推倒重来划算 100 倍。
