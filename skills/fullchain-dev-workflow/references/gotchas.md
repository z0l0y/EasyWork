# 项目踩坑知识库（Gotchas）

> **工业界共识**：Anthropic 内部工程团队将 Gotchas 定位为"信号密度最高的 Skill 内容"——
> 在 Claude Code 官方博客中明确建议每个 Skill 都应积累真实失败案例。Agent 的默认行为
> 是"用最常规的方式解决问题"，而 Gotchas 的作用是把它从常规路径上推开——
> 告诉它"在这个项目里，常规做法会踩坑"。

---

## 设计理念

1. **不重复平台已知信息**：Agent 已经会编程，不需要教它语法。Gotchas 记录的是**特定于本项目的反直觉事实**
2. **积累而非预设**：最好的 Gotchas 来自真实的踩坑记录。每次 Agent 在项目中犯错后，将根因追加到此文件
3. **Agent 可自动追加**：当 Agent 发现一个新陷阱并解决后，应主动在此文件中追加一条记录

---

## Gotchas 格式

每条 Gotchas 包含以下字段：

```markdown
### {标题（一句话描述陷阱）}

- **触发条件**：什么情况下会触发这个问题
- **错误表现**：触发后会看到什么（报错信息、异常行为）
- **根因**：为什么会这样（技术本质）
- **正确做法**：应该怎么做才能避免
- **关联模块**：涉及的文件/服务（可选）
- **首次发现**：日期 + 场景（可选）
```

## 项目 Gotchas

> **说明**：以下是模板条目。Agent 在项目中实际踩坑后，删除模板条目并追加真实记录。

### 模板：示例条目

- **触发条件**：在 Node 20 环境下运行 Jest 27 测试
- **错误表现**：`Worker process failed to initialize` 或测试卡住无输出
- **根因**：Jest 27 的 worker 实现与 Node 20 的 V8 引擎不兼容
- **正确做法**：升级到 Jest 29+，或在 `.npmrc` 中固定 `engine-strict=false`
- **关联模块**：`jest.config.js`、`package.json`
- **首次发现**：2026-06-20 — 初始化测试环境时

## Agent 追加规则（🆕 v2.4：候选-确认制）

Agent 在工作流结束时（SUM 步骤）检查是否满足以下 4 种触发条件。**满足条件时生成候选条目展示给用户，用户确认后才写入此文件。**

### 触发条件

1. **耗时超过 10 分钟才定位到的 bug**：你的排查过程本身就是宝贵知识
2. **一个"看起来没问题"的写法导致了实际 bug**：这类反直觉陷阱价值最高
3. **用户指出了你忽略的边界情况**：人类的经验不容浪费
4. **同一个文件/模式在不同的 session 中被反复问及**：说明它需要被记录下来

### 候选输出格式

满足任一条件时，Agent 在 SUM 产出后展示候选条目：

```
【Gotchas 候选】（以下为 Agent 建议追加的陷阱记录，请确认后写入）

---

### {标题}

- **触发条件**：{条件}
- **错误表现**：{表现}
- **根因**：{根因}
- **正确做法**：{做法}
- **关联模块**：{模块}
- **首次发现**：{日期} — {场景}

---

是否将以上 Gotchas 追加到 references/gotchas.md？
→ 回复"确认"或"修改 {具体调整}"。
```

### 追加指令（仅在用户确认后执行）

```
1. 打开 references/gotchas.md
2. 在"项目 Gotchas"段落下追加用户确认的条目
3. 如用户要求修改，按修改后的内容追加
4. 未确认的候选条目记录在 SUM 产出中，不写入文件
```

### 追加示例

```markdown
### users 表查询必须带 status 过滤

- **触发条件**：在 `users` 表上做任何查询（SELECT/UPDATE/DELETE）而未加 `status` 条件
- **错误表现**：查询结果包含已软删除的用户（`status=-1`），导致业务逻辑错误
- **根因**：`users` 表使用软删除机制，`deleted_at IS NULL` 不等于"活跃"——
  部分业务流程会将用户标记为 `status=-1` 但不设 `deleted_at`
- **正确做法**：所有 users 查询必须同时加 `WHERE status != -1 AND deleted_at IS NULL`
- **关联模块**：`src/services/user.service.ts`、`src/repositories/user.repo.ts`
- **首次发现**：2026-06-20 — 修登录 500 bug 时发现

```

## 维护原则

- **质量 > 数量**：一条有根因分析和正确做法的 Gotchas 胜过十条"这里要注意"
- **保持时效**：如果某条 Gotchas 对应的技术债务已修复，标注 `[已修复 — YYYY-MM-DD]` 而非删除
- **Agent 初次加载时扫描**：编排中枢在 READ 阶段指示 Agent 浏览本文件，了解已知陷阱
- **与 team-policy 的关系**：Gotchas 是"事实型知识"（这个地方会踩坑），team-policy 是"规则型知识"（我们要求这么做）。如果一条 Gotchas 反复出现，考虑将其升级为 team-policy 规则
