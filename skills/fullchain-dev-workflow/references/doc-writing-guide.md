# 文档写作规范（Document Writing Guide）

> **🆕 v2.5**：EasyWork 产物不再使用模板化套话。所有后端的最终报告必须遵守本规范。
> 核心原则：**像一位有经验的工程师在写项目复盘，不像 AI 在填模板。**

---

## 1. 核心原则

### 1.1 写作口吻

产品文档用中文业务复盘口吻撰写。读起来像是一个了解项目的人写的内部技术总结——有观点、有判断、有取舍说明，而不是一篇"放之四海而皆准"的通稿。

**好的复盘口吻**：
> 登录 P99 延迟从 2.8s 降到 0.32s，降幅 88.6%。主要改动在 auth.service.ts 第 88 行——原来的 bcrypt cost factor 14 在单次验证上消耗 800ms，调到 10 之后降到 60ms。这个调整不会影响已有密码的兼容性，因为 bcrypt 的哈希格式本身就包含了 cost factor 参数。

**坏的模板口吻**：
> 综上所述，经过严谨的分析和优化，我们显著提升了登录模块的性能表现，为用户带来了更加流畅的使用体验。后续我们将持续关注系统性能，不断优化用户体验。

套话清单——以下短语**禁止使用**：
- "综上所述"
- "经过严谨分析"
- "我们深刻认识到"
- "为用户带来了更好的体验"（没有数据支撑）
- "持续关注""不断优化""持续改进"（空洞无物）
- "在未来的工作中"
- "本项目采用了业界领先的"

### 1.2 信息密度

每一段都有信息增量。删掉任何一句话不会影响理解 → 这句话就该删。

## 2. 禁止事项

### 2.1 反引号（Markdown 代码标记）

**禁止使用反引号**包裹文件名、命令名、变量名、代码片段。

❌ 错误：
> 修改了 `auth.service.ts` 中的 `verifyPassword` 函数，将 `costFactor` 从 `14` 改为 `10`。

✅ 正确：
> 修改了 auth.service.ts 中的 verifyPassword 函数，将 costFactor 从 14 改为 10。

**例外**：以下场景可以使用反引号：
- 技术文档中需要与 Markdown 渲染器兼容的代码块（如 Mermaid 代码块）
- 用户明确要求 Markdown 格式且需要被 Markdown 渲染器正确解析时
- Git commit message（遵循 Conventional Commits 格式时）

### 2.2 Markdown 代码块标记

**禁止使用** fenced code block（三个反引号）包裹代码/命令。

✅ 正确做法——命令集中展示：
> 本次操作的 Git 命令：
>
>   git add auth.service.ts session.ts
>   git commit -m "fix(auth): 降低 bcrypt cost factor 从 14 到 10"
>
>   git add auth.service.test.ts
>   git commit -m "test(auth): 添加 bcrypt 性能基准测试"
>
> 执行顺序：先提交逻辑改动，再提交测试文件。两个 commit 分开是为了回滚粒度——如果性能问题复现，可以单独 revert 第一个 commit。

### 2.3 模板套话

禁止使用任何"一看就是 AI 写的"短语和句式。如果一段话可以被复制到任何项目的任何报告中而不违和，就重写。

### 2.4 标题数量

**一个标题下面最多 4 个子标题**。超过 4 个时，合并或拆分。

❌ 错误：
> ## 改动详情
> ### 文件 1
> ### 文件 2
> ### 文件 3
> ### 文件 4
> ### 文件 5
> ### 文件 6

✅ 正确：
> ## 改动详情
> ### 核心逻辑层（3 个文件）
> （auth.service.ts, session.ts, token.ts 的改动说明）
> ### 测试层（2 个文件）
> （auth.service.test.ts, session.test.ts 的改动说明）
> ### 配置变更（1 个文件）
> （package.json 的依赖版本更新）

## 3. 必须遵守

### 3.1 段落自然

段落有逻辑起承转合——不是"标题 + 列表"的机械堆叠。一个段落表达一个完整意思。

**好的段落**（有逻辑）：
> 这次问题的根因是密码哈希耗时过高。线上监控发现登录接口 P99 延迟达 2.8 秒，排查路径是：先在 APM 中确认瓶颈在 auth 模块，然后用 profiling 定位到 bcrypt.hash 调用——cost factor 设为 14，单次调用 800ms。正常登录流程中这个函数被调用了 3 次（密码验证、token 刷新、session 更新），累积耗时 2.4s。

**坏的段落**（机械列表）：
> - 问题：登录慢
> - 原因：bcrypt 耗时高
> - 定位：profile 分析
> - 解决：降低 factor

### 3.2 表格仅用于结构化信息

表格只用在需要对比、对照、多项指标罗列的场景。不用表格做布局、不用表格装"标题+内容"。

✅ 适合用表格的：
- 指标对比（修改前 vs 修改后）
- 多文件改动概要（文件名、改动类型、风险等级）
- 测试结果统计（总数、通过、失败、跳过）

❌ 不适合用表格的：
- 把"背景说明"拆成"项目|内容"两列
- 用表格做页面布局
- 把一段完整的话拆成多行表格

### 3.3 命令集中展示

涉及多个命令时，归类、加说明，集中在一个位置展示，不要散落在文章各处。

✅ 正确：
> 以下是本次改动相关的验证命令：
>
>   测试：
>     npm test -- --grep "auth"     # 只跑 auth 相关测试
>     npx jest --coverage           # 检查覆盖率
>
>   构建：
>     npm run build                 # 确认没有编译错误
>
>   Git（审查后手动执行）：
>     git add src/auth/auth.service.ts src/auth/session.ts
>     git commit -m "fix(auth): 降低 bcrypt cost factor"

### 3.4 改动说明结合业务

每个改动不只说"改了什么文件"，更要说明"为什么要改"以及"在业务上达成了什么效果"。

❌ 错误（只有技术描述）：
> 修改了 auth.service.ts，将 BCRYPT_COST_FACTOR 常量从 14 改为 10。

✅ 正确（结合业务）：
> auth.service.ts：将 BCRYPT_COST_FACTOR 从 14 降到 10。这个改动直接影响了用户登录速度——在日均 3000 次登录的场景下，每次登录等待从 2.8 秒降到 0.3 秒。14 和 10 的安全性差异在密码哈希场景下可以忽略（10 仍然是暴力破解的安全阈值），但用户体验差异显著。

## 4. 各步骤特定约束

### 4.1 READ（需求理解）

- 目标用业务语言描述，不用技术术语堆砌
- 验收标准具体可验证（"输入 X 应返回 Y"，不是"功能正常"）

### 4.2 CODE（代码实现）

- 每个文件的改动说明：文件名、改了什么、为什么这样改、影响什么
- 改动说明用业务语言，不要写成"修改了 foo() 函数的 bar 参数"

### 4.3 REVIEW（代码审查）

- 七维度各一句话结论即可，不要每个维度写一大段
- 发现的问题：具体到代码行、什么问题、怎么修的
- 安全/正确性维度如果有阻断性问题要详细说明

### 4.4 GIT（提交拆分）

- 每个 commit unit 必须包含业务上下文说明
- 每个 unit 附带开发者 Check 清单
- 风险说明要具体：哪个模块、什么场景、可能触发什么问题
- 不建议的写法："可能影响稳定性"（太泛）
- 建议的写法："如果 token 刷新请求在弱网环境下超时，旧代码会直接抛出异常导致用户被踢出，新代码会重试 3 次后才降级为匿名访问"

### 4.5 SUM（总结报告）

- 背景：让一个不了解项目的人看完能理解"为什么要花时间做这件事"
- 发现过程：有排查路径逻辑链条
- 解决方案：说明"为什么选 A 不选 B"，有 trade-off 分析
- 效果：有数据对比，不只是"测试通过了"
- 未来：诚实列出遗留问题，不假装完美

### 4.6 TALK（复盘分析）

- 5-Whys 追问要触及系统/流程/机制层面的根因
- 不停在"某人失误了"的层面
- 提炼的规范是具体可执行的，不是"以后要更仔细"

## 5. 范例对比

### 范例 1：GIT 拆分说明

❌ **坏**：
> ### 1. Logic — 修复登录超时
> - 文件：auth.ts, session.ts
> - 说明：修改了登录相关代码
> - 风险：低

✅ **好**：
> ### 1. 核心逻辑 — 修复登录超时
>
> 改动文件：auth.ts（3 处修改）、session.ts（1 处修改）
>
> 问题背景：线上用户反馈登录后约 15 分钟就会被踢出。排查发现 session 刷新逻辑中，token 过期时间判断使用了本地时间而非服务器时间——当用户设备时间比服务器慢几分钟时，session 在服务器已过期但客户端以为还在有效期内，下次请求直接被 401 拦截。
>
> 改动内容：
> - auth.ts:88 — token 过期判断从 Date.now() 改为响应头中的 Date 字段
> - auth.ts:142 — 新增 3 次重试逻辑，弱网环境下不直接抛异常
> - auth.ts:210 — token 刷新间隔从固定 10 分钟改为过期前 2 分钟动态刷新
> - session.ts:56 — session 状态增加 refreshing 中间态，防止并发刷新
>
> 引入风险：中等。改动涉及认证核心链路，如果 token 刷新逻辑有 bug 会导致全站登录失败。缓解措施——回滚策略：如果线上 token 刷新失败率超过 1%，立即回滚到旧版 session 处理逻辑（通过 feature flag 切换）。
>
> 验证证据：auth 模块 12 个测试全部通过；新增 3 个测试覆盖 token 过期/弱网重试/并发刷新场景。
>
> 开发者 Check：
> - [ ] 改动范围：符合需求，只改了 auth 和 session 模块
> - [ ] 无意外副作用：其他模块的 token 使用方式未变化
> - [ ] 测试覆盖：12 已有 + 3 新增，覆盖正常/过期/弱网/并发
> - [ ] 代码风格：与项目现有错误处理模式一致

### 范例 2：SUM 背景

❌ **坏**：
> 本次任务是优化系统性能，提升用户体验。经过分析发现了登录模块的性能瓶颈，通过调整 bcrypt 参数成功解决了问题。

✅ **好**：
> 6 月 15 日起，线上客服开始收到大量"登录太慢"的投诉。运维在 APM 中确认登录接口 P99 延迟达 2.8 秒——正常应该在 500ms 以内。影响面评估：日均约 3000 个独立用户登录，其中约 40% 在高峰期（午间 12:00-14:00、晚间 19:00-22:00）遇到超过 3 秒的等待。如果不修，预计每周流失约 200 个因登录体验差而放弃的用户。

## 7. 报告深度与类型指南（🆕 v2.7）

### 7.1 深度层级

| 深度 | 长度 | 适用场景 | 关键特征 |
|------|------|---------|---------|
| brief | 1-2页 | 微调、小bug、领导汇报 | 仅六要素概要(每要素1-2句)、无详细排查路径、无量化对比、无代码摘录 |
| standard | 3-5页 | 纯理解、纯审查、纯文档 | 完整六要素、含排查路径、量化对比、审查结论(每维度1句) |
| detailed | 6-10页 | 功能开发、重构、Bug修复 | 完整六要素 + 逐步骤MCR + 代码摘录(含路径行号) + 完整CTO问答 + 边界处理说明 |

### 7.2 报告类型

**executive_summary（领导层摘要）**：
- 目标读者：技术管理者、产品经理、业务方
- 语言：业务语言为主，技术术语需解释
- 重点：为什么做、效果如何、风险是什么、需要什么决策
- 省略：详细排查过程、代码细节、测试覆盖矩阵
- 始终 brief 深度。报告应回答"这件事对业务有什么价值"，而非"代码怎么写的"

**engineering_record（工程记录）**：
- 目标读者：开发团队成员、Code Review 同事、后续维护者
- 语言：技术语言为主，结合业务上下文
- 重点：怎么做的、为什么这样做、有什么 trade-off、踩了什么坑、以后要注意什么
- 必须包含（detailed 模式）：代码摘录(含路径行号)、MCR 全部条目、完整 CTO 问答
- standard 或 detailed 深度。报告应满足"同行审查就绪"标准

### 7.3 "同行审查就绪"标准

一份 engineering_record（detailed）报告应满足：**另一个工程师不阅读源码、不询问你，就能理解**：

- 改动解决了什么业务问题（业务上下文）
- 改动了哪些文件、每个文件改了什么、为什么（文件变更表 + 设计说明）
- 方案选择逻辑（为什么选 A 不选 B——有具体技术分析）
- 关键代码摘录（含文件路径和行号，可直接定位到源码）
- 测试覆盖了什么、没覆盖什么、为什么
- 已知风险和后续工作

满足这个标准的报告可以作为交接文档、PR 描述、事故复盘的基础材料。不满足的 → 回退补充。

## 8. 文档迭代与版本管理（🆕 v2.8）

当需求发生变更、需要对已有产物文档进行更新时，遵循以下增量更新原则。

### 8.1 核心原则：原位追加，不重建

**每一次需求变更都是文档的一次演进，不是一次重写。**

- 在原有标题结构下追加更新内容，不新建文档
- 不新增顶层标题（如 `## 第二次迭代`）
- 每次更新标注版本号和日期
- 过时信息标注而非删除——保留演进历史

### 8.2 更新记录格式

每次更新在文档末尾（footer 之前）的"更新记录"章节追加一条：

```
### 更新 v1.1 — 2026-06-25（本次）

**变更原因**：用户反馈登录页在 Safari 17 以下版本白屏

**在以下章节追加了内容**：
- 2. 发现过程 — 追加了 Safari 17 WebKit 渲染引擎兼容性排查路径
- 4. 解决方案 — 追加了 CSS Grid fallback 方案的取舍分析

**过时信息标注**：
- 3. 问题说明 — "浏览器兼容性已验证通过（Chrome/Firefox/Edge）"标注为 `[已过时 — v1.1 起]：遗漏 Safari 17 以下版本验证`

**当前版本状态**：v1.1
```

### 8.3 过时信息处理

**标注而非删除**。过时信息是重要的历史上下文——它解释了"当时为什么这样做"，帮助后来者理解决策演进。

格式：在过时段落前方追加 `[已过时 — vX.X 起]` 标记，后跟更新后的内容：

```
[已过时 — v1.1 起] 以下结论基于 v1.0 时仅测试了 Chrome/Firefox/Edge 的验证结果。
Safari 17 以下版本的兼容性问题已在 v1.1 修复，详见更新记录。

原内容：
> 浏览器兼容性已全面验证通过（Chrome 120+、Firefox 121+、Edge 120+）
```

### 8.4 自检清单

文档迭代时，Agent 必须逐项检查：

- [ ] 🆕 v2.8 未新建文档——在原文档基础上追加
- [ ] 🆕 v2.8 未新增顶层标题——在原有标题下追加更新记录子节
- [ ] 🆕 v2.8 更新标注了版本号和日期
- [ ] 🆕 v2.8 已检查过时信息——标注了 `[已过时 — vX.X 起]` 而非直接删除
- [ ] 🆕 v2.8 更新记录含变更原因、追加位置、过时标注、当前版本状态
- [ ] 🆕 v2.8 文档可追溯完整演进历史（非仅反映最新状态）

## 9. ETR 标准与反水文写作（🆕 v2.9）

### 9.1 核心原则

**产物不是过程打卡表，而是可供他人复盘、审查、接手的工程记录。**

每一段文字都必须有信息增量。删掉任何一句不影响理解 → 那句话就该删。

### 9.2 ETR 三段式

每个关键结论必须包含三段：

**Evidence（证据）** — 有什么证据？
- 测试输出片段（非"测试通过"）
- 命令 + 关键结果（非"go vet 通过"）
- 代码位置（文件+函数+行号——非"某处"）
- 数据对比（修改前 vs 修改后——非"提升了性能"）

**Thinking（推理）** — 为什么这样判断？
- 从现象到结论的推理链（非跳跃式"所以功能正常"）
- 排查路径（排除了什么→为什么排除→定位到了什么）

**Risk（风险）** — 还有什么风险？
- 未覆盖的场景 + 原因
- 已知副作用
- 假设前提（"这是在 X 假设下的结论"）
- 何时会失效

### 9.3 水文信号自查

在 finalize 报告前，逐段检查以下信号：

| 水文信号 | 为什么不能写 | 替换为 |
|---------|-------------|--------|
| "功能正常""逻辑清晰""代码质量较好" | 无标准、无证据、无推理 | 具体检查点+代码位置+为什么这样判断 |
| "测试通过"（无命令+输出） | 不知道跑了什么 | 命令 + 关键输出片段 + 覆盖矩阵 |
| "审查通过"（无检查点） | 不知道看了哪里 | 每维度≥2条检查点+位置+结论依据 |
