# 验收关卡

以下关卡用于判断全链路 9 步中每一步是否达标。未达标的步骤需要回退修正或补充验证后方可进入下一步。

## READ（需求理解）验收关卡
- [ ] 需求摘要包含明确的业务目标和上下文
- [ ] 修改范围清晰列出涉及的文件、模块或服务
- [ ] 验收标准具体、可验证（不是"功能正常"这种模糊表述，而是"输入 X 应返回 Y"这种可测试的标准）
- [ ] 约束与假设已明确列出（技术约束、业务假设）
- [ ] "不做什么"已明确划定边界，防止 scope creep
- [ ] 澄清问题不超过 3 个，且每个问题都是可回答、可验证的
- [ ] 如果输入包含图片/日志/代码，已正确解析其内容并关联到需求中

## CODE（代码实现）验收关卡
- [ ] 所有变更文件都有明确的改动原因说明
- [ ] 新增代码的注释使用中文
- [ ] 代码风格与项目现有模式保持一致（没有引入新的设计模式或代码组织方式）
- [ ] 没有引入新的第三方依赖（除非确实必要且已说明理由）
- [ ] 改动严格限定在需求划定的范围内，没有顺手"优化"无关代码
- [ ] 变更清单与验收标准逐项对齐，每一项验收标准都有对应的代码实现
- [ ] 🆕 若项目为 git 仓库，改动前已确保工作区干净或已 stash（checkpoint 机制）

## REVIEW（代码审查）验收关卡
- [ ] 七维度审查全部完成，每项都有明确的结论（通过/有问题/不适用）
- [ ] 发现的阻断性问题和主要问题已有修复记录
- [ ] 修复后已重新审查确认通过
- [ ] 审查报告引用了具体的代码位置（文件名+行号），而非笼统描述
- [ ] 如果全部通过，审查报告中有足够的具体检查证据，而非空洞的"看起来没问题"
- [ ] 🆕 二次独立抽查已完成：对最自信的 2 个 pass 维度做交叉验证，结论已记录
- [ ] 🆕 反合理化防御表已在审查前过目（9 条 Agent 自我欺骗话术及反驳）
- [ ] 🆕 可访问性维度（a11y）已检查（如涉及前端代码）：语义化HTML/ARIA/键盘导航/色彩对比
- [ ] 🆕 供应链检查已执行（如有新增依赖）：许可证兼容性/已知漏洞/维护状态

## EXAMINE（质量验证）验收关卡
- [ ] 有实际的终端测试执行记录（命令+输出），或明确说明了未运行测试的原因
- [ ] 如果项目无测试，已补充最小化的测试用例
- [ ] 所有测试结果均为通过（绿色），无失败或跳过的测试（除非跳过的原因已说明）
- [ ] 测试失败有修复记录，修复后有重新运行的通过记录
- [ ] 阻断项被标记并已回到修复闭环

## GIT（提交拆分）验收关卡
- [ ] 改动已按维度（配置/逻辑/UI/测试）合理分类
- [ ] 每个拆分单元都有：文件清单、改动说明、风险分析、验证方式
- [ ] 拆分后的每个单元可以独立编译（不需要跨单元依赖）
- [ ] 无法拆分的"混合变更文件"已被醒目标记，并附有特别审查提示
- [ ] 🆕 每个拆分单元附有 Conventional Commits 格式的提交消息建议
- [ ] 🆕 v2.4：git 命令已写入 `.claude/easywork/git-commands.sh`（或 `.bat`），未自动执行 add/commit/push
- [ ] 🆕 v2.5：每个 commit unit 的 business_context 非空且有实际业务说明（非泛泛的"修复bug"）
- [ ] 🆕 v2.5：每个 commit unit 的 developer_checklist 完备（5 项 Check 均已列出）
- [ ] 🆕 v2.5：每个 commit unit 的 risk_introduced 有具体的场景/影响/缓解说明
- [ ] 🆕 v2.5：每个 commit unit 的 verification_evidence 有测试覆盖场景和结果引用
- [ ] 🆕 v2.5：git_tracking 数据已产出（task_description/units/examine_result 非空）
- [ ] 🆕 v2.5：commit_message.body 包含改动原因/风险说明/验证方式三段
- [ ] 拆分方案已展示给用户并获得确认

## GRAPH（图表绘制）验收关卡
- [ ] Mermaid 代码语法正确，可以在渲染器中正常显示
- [ ] 图表类型选择合理（流程用 flowchart，调用用 sequenceDiagram，结构用架构图）
- [ ] 图表中的每个节点都能在本次改动代码中找到对应的实体
- [ ] 没有虚构或编造的组件/模块出现在图中
- [ ] 附有节点与代码对照表
- [ ] 图表附有文字解释，说明关键流程

## SUM（总结报告）验收关卡
- [ ] 六要素（背景、发现过程、问题说明、解决方案、最终效果、未来展望）齐全
- [ ] 背景部分说明了"为什么要做"而非仅仅"做了什么"
- [ ] 解决方案部分说明了"为什么选这个方案"而非仅仅"怎么做的"
- [ ] 效果部分有客观数据或测试结果支撑
- [ ] 未来展望诚实列出了遗留问题和后续建议
- [ ] 总结内容与前面各步骤的实际产出一致（不凭空出现未提及的改动）

## TALK（复盘分析）验收关卡
- [ ] 5-Whys 追问至少到达第 4 层，触及系统/流程/机制层面的根因（而非停在"某人失误"的层面）
- [ ] 每个 Why 的回答有逻辑链条，不是跳跃式的
- [ ] Trade-offs 分析诚实地列出了本次方案的代价和妥协
- [ ] 提炼的工程规范是具体的、可执行的（不是"以后要更仔细"这种废话）
- [ ] 复盘结果与前几步的实际过程和决策一致

## ASK（人工确认）验收关卡
- [ ] 六个维度（验收、兼容、数据、安全、测试、上线）各至少提出了 1 个具体问题
- [ ] 每个问题都关联到本次改动的具体内容，而非泛泛而问
- [ ] 每个问题都说明了"为什么问这个"和"如果答案为否则下一步做什么"
- [ ] 用户的确认/修改意见已记录
- [ ] 未确认的项被标记为"待确认"状态

## 全局验收关卡
- [ ] 所有 9 步的输出均符合对应模板的结构
- [ ] 没有跳过任何步骤（除非用户明确同意跳过）
- [ ] 没有在任何步骤中出现 Agent 自行猜测决定的情况（所有不确定都已询问用户）
- [ ] 最终产出的代码、文档、图表之间信息一致，无自相矛盾
- [ ] 🆕 每步产出自检已完成：所有 `[必填]` 字段均已产出，无空值占位符
- [ ] 🆕 结构化日志已追加到 `.claude/easywork/workflow.log.jsonl`
- [ ] 🆕 项目 Gotchas（`references/gotchas.md`）已扫描——如当前任务命中已知陷阱，已预警
- [ ] 🆕 团队策略覆盖（`references/team-policy.md`）已加载并应用（如存在）
- [ ] 🆕 v2.4 安全策略（`references/security-policy.md`）已加载并生效
- [ ] 🆕 v2.4 Git 安全：所有 git 写操作已获用户确认，拆分命令已写入脚本文件
- [ ] 🆕 v2.4 脱敏自检：HTML 报告和日志不含 Token/密钥/内部URL/大段源码/手机/邮箱
- [ ] 🆕 v2.4 自定义步骤：已列出清单并获用户确认（或已标注跳过）
- [ ] 🆕 v2.4 供应链检查：未将私有包名或内部 URL 发到外网搜索
- [ ] 🆕 v2.4 文件写入范围：所有写操作均在项目根目录内
- [ ] 🆕 v2.4 Gotchas 候选：如有新陷阱，已展示候选条目并获用户确认后才写入文件
- [ ] 🆕 v2.4 安全策略自保护：`security-policy.md` 未被任何技能修改
