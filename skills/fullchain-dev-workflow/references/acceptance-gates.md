# 验收关卡

以下关卡用于判断全链路 10 步中每一步是否达标。未达标的步骤需要回退修正或补充验证后方可进入下一步。

## READ（需求理解）验收关卡
- [ ] 需求摘要包含明确的业务目标和上下文
- [ ] 修改范围清晰列出涉及的文件、模块或服务
- [ ] 验收标准具体、可验证（不是"功能正常"这种模糊表述，而是"输入 X 应返回 Y"这种可测试的标准）
- [ ] 约束与假设已明确列出（技术约束、业务假设）
- [ ] "不做什么"已明确划定边界，防止 scope creep
- [ ] 澄清问题不超过 3 个，且每个问题都是可回答、可验证的
- [ ] 如果输入包含图片/日志/代码，已正确解析其内容并关联到需求中
- [ ] 🆕 v2.7 MCR-detailed：需求背景含业务上下文(business_background 非纯技术描述)
- [ ] 🆕 v2.7 MCR-detailed：用户目标明确(user_persona 包含谁在什么场景下做什么)
- [ ] 🆕 v2.7 MCR-detailed：MVP边界清晰(mvp_scope 列出本次做/不做)
- [ ] 🆕 v2.7 MCR-detailed：验收标准≥2条(standard)/≥3条(detailed)，均为可测试标准
- [ ] 🆕 v2.7 MCR-detailed：不做事项≥1条，非空泛的"不涉及其他模块"占位
- [ ] 🆕 v2.8 需求理解确认：Agent 已用自己的话重述对需求的理解（含业务目标、技术方案假设、不确定点）
- [ ] 🆕 v2.8 需求理解确认：如用户未指定实现方法，已列出≥1个澄清问题（不可自行假设实现方案）
- [ ] 🆕 v2.8 需求理解确认：已获得用户确认后才进入 CODE 阶段

## CODE（代码实现）验收关卡
- [ ] 所有变更文件都有明确的改动原因说明
- [ ] 新增代码的注释使用中文
- [ ] 代码风格与项目现有模式保持一致（没有引入新的设计模式或代码组织方式）
- [ ] 没有引入新的第三方依赖（除非确实必要且已说明理由）
- [ ] 改动严格限定在需求划定的范围内，没有顺手"优化"无关代码
- [ ] 变更清单与验收标准逐项对齐，每一项验收标准都有对应的代码实现
- [ ] 🆕 若项目为 git 仓库，改动前已确保工作区干净或已 stash（checkpoint 机制）
- [ ] 🆕 v2.7 MCR-detailed：文件变更表完整(路径+变更类型+行数+原因)
- [ ] 🆕 v2.7 MCR-detailed：核心设计说明解释了"为什么这样设计"(design_rationale 非空)
- [ ] 🆕 v2.7 MCR-detailed：关键函数职责说明≥2个(key_functions 有≥2项)
- [ ] 🆕 v2.7 MCR-detailed：重要边界处理说明≥1个(edge_case_handling 有≥1项)
- [ ] 🆕 v2.7 MCR-detailed：≥1段关键代码摘录(≤30行，含文件路径+行号)

## REVIEW（代码审查）验收关卡
- [ ] 七维度审查全部完成，每项都有明确的结论（通过/有问题/不适用）
- [ ] 发现的阻断性问题和主要问题已有修复记录
- [ ] 修复后已重新审查确认通过
- [ ] 审查报告引用了具体的代码位置（文件名+行号），而非笼统描述
- [ ] 如果全部通过，审查报告中有足够的具体检查证据，而非空洞的"看起来没问题"
- [ ] 🆕 二次独立抽查已完成：对最自信的 2 个 pass 维度做交叉验证，结论已记录
- [ ] 🆕 反合理化防御表已在审查前过目（9 条 Agent 自我欺骗话术及反驳）
- [ ] 🆕 可访问性维度（a11y）已检查（如涉及前端代码）：语义化HTML/ARIA/键盘导航/色彩对比
- [ ] 🆕 供应链检查已执行（如有新增依赖）：许可证兼容性/已知漏洞/维护状态
- [ ] 🆕 v2.7 MCR-detailed：七维度每维度≥2条具体检查点(dimensions.*.checkpoints 非空且非占位)
- [ ] 🆕 v2.7 MCR-detailed：发现的 issue 含具体代码位置(文件+行号，非"某处"笼统描述)
- [ ] 🆕 v2.7 MCR-detailed：无阻断问题时 blocking_issues_detail 说明了为什么没有(非空泛的"代码质量好")
- [ ] 🆕 v2.8 质量门禁：REVIEW 发现阻断问题后已回退 CODE 修复（非带着问题继续前进）
- [ ] 🆕 v2.8 回退循环：回退后已重新过完整七维度审查（非只查"修了什么"）
- [ ] 🆕 v2.8 回退上限：CODE↔REVIEW 回退未超过 3 轮，超过已挂起用户

- [ ] 阻断项被标记并已回到修复闭环

- [ ] 🆕 v2.8 交互式应用：如任务涉及 CLI/TUI/GUI/Web前端/游戏/交互式IO，已验证首屏稳定性(≥3次冷启动无闪烁/白屏)

- [ ] 🆕 v2.8 CLI/TUI：已验证 stdin 边界(关闭/空输入/非法输入/特殊字符/退出指令)

- [ ] 🆕 v2.8 自动刷新：已验证刷新频率不会造成刷屏、日志爆炸或 CPU 持续高负载

## SUM（总结报告）验收关卡

- [ ] 未来展望诚实列出了遗留问题和后续建议
- [ ] 总结内容与前面各步骤的实际产出一致（不凭空出现未提及的改动）
