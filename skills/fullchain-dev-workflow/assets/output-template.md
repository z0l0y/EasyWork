# 全链路输出模板

使用 `fullchain-dev-workflow` 时的完整输出结构。被跳过的步骤标注 `[skip]` 及跳过原因。

---

## 阶段 0：任务分类（每次启动时首先输出）

| 维度 | 判断 |
|------|------|
| 改动性质 | {纯理解/纯文档/微调/Bug修复/重构/功能开发} |
| 影响范围 | {低风险/中风险/高风险} |
| 预估文件数 | {N} 个 |

### 裁剪后步骤

```
[ ] 1. READ  — {执行/跳过} — {理由}
[ ] 2. CODE  — {执行/跳过} — {理由}
[ ] 3. REVIEW — {执行/跳过} — {理由}
[ ] 4. EXAMINE — {执行/跳过} — {理由}
[ ] 5. GIT — {执行/跳过} — {理由}
[ ] 6. GRAPH — {执行/跳过} — {理由}
[ ] 7. SUM — {执行/跳过} — {理由}
[ ] 8. TALK — {执行/跳过} — {理由}
[ ] 9. ASK — {执行/跳过} — {理由}
```

---

## 步骤 1：READ（需求理解）

- **核心目标**：{1-3 句话，可验证}
- **修改范围**：{涉及的文件/模块列表}
- **约束条件**：{技术约束 + 业务约束}
- **验收标准**：
  - [ ] {可验证标准 1}
  - [ ] {可验证标准 2}
- **不做什么**：{明确排除的范围}
- **澄清问题**：{如有，不超过 3 个}

---

## 步骤 2：CODE（代码实现）

- **实现方案**：{技术方案概述}
- **变更清单**：
  | 文件 | 改动类型 | 改动量 | 原因 |
  |------|---------|--------|------|
  | `path/to/file` | 修改/新增 | ~N行 | {原因} |
- **影响面评估**：{直接影响 + 间接影响}
- **新增依赖**：{如有，说明为什么必须引入}

---

## 步骤 3：REVIEW（代码审查）

- **审查结论**：✅ 通过 / ⚠️ 修复后通过 / ❌ 有未解决的阻断问题
- **六维度扫描**：
  - 正确性：{结论} — {发现的问题（如有）}
  - 安全性：{结论}
  - 兼容性：{结论}
  - 可维护性：{结论}
  - 性能：{结论}
  - 可观测性：{结论}
- **修复记录**：{回退到 CODE 修复的问题及修复内容}

---

## 步骤 4：EXAMINE（质量验证）

- **测试命令**：`{实际执行的命令}`
- **结果摘要**：通过 N / 失败 0 / 跳过 M
- **新增测试**：{补充了哪些测试用例}
- **质量结论**：✅ 通过 / ⚠️ 条件通过 / ❌ 未通过
- **测试输出凭据**：
  ```
  {关键输出片段}
  ```

---

## 步骤 5：GIT（提交拆分）

- **拆分概览**：

  | # | 维度 | 文件数 | 风险 | 审查优先级 |
  |---|------|--------|------|-----------|
  | 1 | 配置/构建 | N | 低 | ⭐ |
  | 2 | 核心逻辑 | N | 高 | ⭐⭐⭐ |
  | ... | ... | ... | ... | ... |

- **各单元详情**：{每单元：文件清单 + 改动说明 + 风险分析 + 验证方式}
- **混合变更警示**：{无法拆分的文件及原因}

---

## 步骤 6：GRAPH（图表绘制）

- **Mermaid 代码**：

  ```mermaid
  {代码}
  ```
- **节点对照表**：
  | 图中节点 | 代码实体 | 文件位置 |
  |---------|---------|---------|
  | ... | ... | ... |

## 步骤 7：SUM（总结报告）

- **背景**：{为什么做}
- **发现过程**：{怎么定位的}
- **问题说明**：{本质是什么}
- **解决方案**：{怎么修的 + 为什么这样修}
- **最终效果**：{验证结果，量化数据}
- **未来展望**：{遗留问题 + 后续建议}

---

  |--------|------|------|----------|------|
  | ... | ... | ... | ... | ... |

- **工程规范建议**：{具体、可执行的规则}

- **维度 1 — 需求与验收**：Q1 → 状态：⏳/✅/❌
