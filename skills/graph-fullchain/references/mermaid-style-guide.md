# Mermaid 图表样式指南

本文档提供 Mermaid 图表的绘制规范和最佳实践，确保产出的图表清晰、规范、可维护。

## 图表类型选择指南

### Flowchart（流程图）— 最常用
适用场景：
- 描述某个操作的执行步骤
- 包含条件判断和分支的流程
- 状态变化和阶段转换

语法要点：
- 使用 `flowchart TD`（Top-Down，从上到下）或 `flowchart LR`（Left-Right，从左到右）
- 节点类型：`[矩形]` 用于操作步骤、`{菱形}` 用于判断、`([圆角矩形])` 用于开始/结束、`[(数据库)]` 用于数据存储
- 连线标签：`-->|标签文字|` 在连线上添加说明

### SequenceDiagram（时序图）
适用场景：
- 多个参与者（前端、后端、数据库、缓存、第三方服务）之间的调用时序
- 请求-响应的完整链路
- 异步消息传递

语法要点：
- 使用 `participant` 定义参与者，可加别名：`participant FE as 前端应用`
- 实线箭头 `->>` 表示同步调用，虚线箭头 `-->>` 表示返回
- 使用 `Note over` 添加说明注释

### 架构图 / 系统关系图
适用场景：
- 展示系统的整体拓扑结构
- 展示模块/服务之间的依赖关系
- 新功能在系统中的位置

语法要点：
- 使用 `subgraph` 对节点进行分组，表示一个子系统或层级
- 连线方向保持一致（推荐从左到右或从上到下）
- 使用不同形状区分不同类型的节点（服务、数据库、外部系统、用户等）

## 节点命名规范

### 节点文案要求
- **简短**：每个节点的文案控制在 8 个字以内
- **动词化**：操作类节点使用动词开头（如"校验Token"而非"Token校验逻辑"）
- **名词化**：实体类节点使用名词（如"用户数据库"）
- **一致**：同一实体在不同图表中保持相同的名称

### 节点 ID 规范
- 使用有意义的英文 ID（如 `validateToken` 而非 `A`、`B`、`C`）
- 这样在修改图表时更容易定位节点
- ID 不会显示在渲染结果中，但会影响代码可维护性

## 连线与布局规范

### 连线方向
- 整个图表保持统一的流向（全部从上到下，或全部从左到右）
- 避免连线交叉——如果出现交叉，考虑调整节点排列顺序
- 回环和反馈线使用 `-->` 的不同变体区分

### 连线标签
- 仅在分支判断的出口添加标签（如"通过"/"失败"）
- 直通流程的连线不需要标签（减少视觉噪音）
- 标签文字同样简短（4 个字以内）

## 图表复杂度控制

### 复杂度红线
- 单张图超过 20 个节点 → **必须拆分**为多张子图
- 单张图超过 5 层嵌套 → **必须使用 subgraph** 进行层级归组
- 连线交叉超过 3 处 → **必须重排**节点顺序

### 拆分策略
- **按功能模块拆分**：每个子模块一张图
- **按层级拆分**：总览图展示大模块关系，详图展示模块内部流程
- **按时序拆分**：按时间阶段拆分为多张图

## 常见问题

### 如何在 Mermaid 中表示循环？
使用回环连线：`E -->|重试| B`（如果 B 是循环的起点）

### 如何处理过长的节点文字？
使用换行：`A[第一行<br/>第二行]`，但尽量避免——长文字说明应该在图表下方的文字解释中给出。

### 如何标记本次新增/修改的节点？
在节点文字前加标记，如 `A[🆕 新增校验逻辑]`，并在图表说明中解释标记含义。或者使用 `style` 给特定节点添加颜色。

## 验证方法

完成 Mermaid 代码后，复制到以下任一工具验证渲染效果：
- [Mermaid Live Editor](https://mermaid.live/) — 官方在线编辑器
- GitHub 的 Markdown 预览 — 直接在 `.md` 文件中查看
- VS Code 的 Mermaid Preview 插件

**确保图表可以正确渲染，没有语法错误，没有断开或孤立的节点。**
