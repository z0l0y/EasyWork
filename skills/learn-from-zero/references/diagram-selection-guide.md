# 图解策略指南（Diagram Selection Guide）

> 何时需要图？什么样的图？为什么选这种图？
> 参考 archify 的 5 类架构图选择逻辑 + graph-fullchain 的 Mermaid 风格指南

## 核心原则

1. **不是所有概念都需要图**——先判断"值得画吗"
2. **图不是替代文字，是补充文字**——每张图配"节点对照表"
3. **图的选择来自概念特征，不是来自画图能力**——先分析概念，再选图

---

## 决策流程

```
Step 1: 概念可视化判断

这个概念的核心是什么？
├── 一个流程/步骤序列（如"HTTP 请求的生命周期"）
│   → 值得画 ✅ → flowchart TD
│
├── 多个角色之间的交互（如"前端怎么跟后端通信"）
│   → 值得画 ✅ → sequenceDiagram
│
├── 一个东西的不同状态（如"TCP 连接的状态变化"）
│   → 值得画 ✅ → stateDiagram-v2
│
├── 知识的层级依赖关系（如"理解 React 需要先懂什么"）
│   → 值得画 ✅ → mindmap（但通常留给知识树用）
│
├── 数据的结构和关系（如"用户表和订单表的关系"）
│   → 值得画 ✅ → erDiagram
│
├── 多个方案的横向对比（如"Redis vs Memcached"）
│   → 更适合作表格 ⚠️ → Markdown table
│
├── 抽象的计算过程（如"递归怎么工作的"）
│   → 可以画但效果一般 → ASCII 图优先
│
└── 纯抽象概念（如"什么是多态"）
    → 不适合画图 ❌ → 强化类比+实例，不强行画图

Step 2: 层级 → 图复杂度

├── L0 (ELI5): 节点 ≤ 5, 零术语, 优先 ASCII 图
├── L1 (直觉): 节点 ≤ 8, 可有 1-2 个术语（配对照表）
├── L2 (技术): 节点 ≤ 15, 标准 Mermaid 图
└── L3 (深入): 节点 ≤ 20, 可拆分多图

Step 3: 生成图 → 配节点对照表 → 验证
```

## 七种图类型速查

### 1. Flowchart（流程图）— `flowchart TD/LR`

**适用场景**：
- 描述操作的执行步骤（如"用户登录流程"）
- 包含条件判断和分支（如"支付成功/失败的分支处理"）
- 数据从一个阶段流向另一个阶段（如"编译流水线"）

**Mermaid 语法要点**：
```
flowchart TD
    A[开始] --> B{判断条件}
    B -->|通过| C[做这件事]
    B -->|失败| D[做那件事]
    C --> E([结束])
    D --> E
```

**节点形状语义**：
- `[矩形]` — 操作/步骤
- `{菱形}` — 判断/分支
- `([圆角矩形])` — 开始/结束
- `[(圆柱)]` — 数据存储

### 2. SequenceDiagram（时序图）— `sequenceDiagram`

**适用场景**：
- 多个参与者之间的调用时序（如"前端→后端→数据库→缓存"）
- 请求-响应的完整链路
- 异步消息传递

**Mermaid 语法要点**：
```
sequenceDiagram
    participant U as 用户
    participant S as 服务器
    participant D as 数据库
    U->>S: 发送请求
    S->>D: 查询数据
    D-->>S: 返回结果
    S-->>U: 返回响应
```

**连线语义**：
- `->>` 实线箭头 — 同步调用
- `-->>` 虚线箭头 — 返回/异步响应
- `Note over` — 给参与者加注释

### 3. StateDiagram（状态图）— `stateDiagram-v2`

**适用场景**：
- 一个对象/连接的生命周期（如"TCP 连接：CLOSED→LISTEN→ESTABLISHED→CLOSED"）
- 多种状态和转换条件

**Mermaid 语法要点**：
```
stateDiagram-v2
    [*] --> 空闲
    空闲 --> 运行中: 收到启动信号
    运行中 --> 暂停: 收到暂停信号
    暂停 --> 运行中: 收到恢复信号
    运行中 --> [*]: 任务完成
```

### 4. Mindmap（思维导图）— `mindmap`

**适用场景**：
- 知识的前置依赖关系（学习路线、知识树）
- 概念的分类和子类

**注意**：mindmap 在 learn-from-zero 中主要用于 §2.4 知识树。如需额外使用，确保不与知识树重复。

### 5. ER Diagram（实体关系图）— `erDiagram`

**适用场景**：
- 数据的实体和关系（如"用户-订单-商品"三者的关系）
- 数据库表结构概览

**Mermaid 语法要点**：
```
erDiagram
    USER ||--o{ ORDER : 下订单
    ORDER ||--|{ ORDER_ITEM : 包含
    PRODUCT ||--o{ ORDER_ITEM : "被购买"
```

### 6. Markdown Table（表格）— 非 Mermaid

**适用场景**：
- 多个方案的横向对比（各方案 × 各维度）
- 概念分类和特征对比
- 术语对照表

**什么时候用表格而不是图**：
- 用户需要精确对比 → 表格（表格数据更精确，图更容易忽略细节）
- 用户需要直观印象 → 图（图比表格更容易形成直觉）

### 7. ASCII Art（ASCII 示意图）— 非 Mermaid

**适用场景**：
- 抽象计算过程（递归/排序/树遍历）
- L0/L1 层的简单结构示意
- Mermaid 不方便表达的"空间布局"（如内存布局、目录结构）

**示例**：
```
调用栈（递归计算阶乘）：
factorial(4)
  → 4 * factorial(3)
         → 3 * factorial(2)
                → 2 * factorial(1)
                       → 1（到底了，开始返回）
                ← 2 * 1 = 2
         ← 3 * 2 = 6
  ← 4 * 6 = 24
```

## L0-L1 vs L2-L3 图解策略差异

| 维度 | L0-L1（给零基础/非技术人员） | L2-L3（给开发者） |
|------|--------------------------|-----------------|
| 图类型 | ASCII 优先，简单 Mermaid | Mermaid 标准图 |
| 节点数上限 | ≤ 8（L0 不超过 5） | ≤ 20 |
| 节点文字 | 零术语，生活词汇 | 标准技术术语 |
| 节点对照表 | **必须有**——每个节点用白话解释 | 推荐有——至少对核心节点做解释 |
| 连线标签 | 不说"成功/失败"——说"对了/不对" | 标准技术描述 |
| subgraph | 不使用（太复杂） | 可使用（归组复杂子系统） |
| 颜色/样式 | 不使用（分散注意力） | 可用 style 标注关键节点 |

## 复杂度红线（参考 graph-fullchain 风格指南）

- 单张图超过 20 个节点 → **必须拆分**为多张子图
- 单张图超过 5 层嵌套 → **必须使用 subgraph** 进行层级归组
- 连线交叉超过 3 处 → **必须重排**节点顺序
- L0-L1 图违反任一红线 → **降级为 ASCII 图**或拆分为多图

**拆分策略**：
1. **按功能模块拆分**：如"用户系统"拆为"注册流程"和"登录流程"两张图
2. **按层级拆分**：总览图展示大模块，详图展示模块内部
3. **按时序拆分**：阶段 1 → 阶段 2 → 阶段 3 各一张图

## 常见错误

- ❌ "这个概念很重要，所以必须画图" → 重要性 ≠ 可可视化性。纯抽象概念画图适得其反
- ❌ 流程图拆成 30 个节点 → 没人看得懂，效果为零
- ❌ L0 层用 `sequenceDiagram`（时序图）→ 零基础用户看到"participant"这类词就跑
- ❌ 图没有标题说明 → 读者不知道这张图在说什么
- ❌ 图有语法错误 → 验证后再输出。在心里走一遍渲染结果

## 验证方法

完成 Mermaid 代码后，人工验证以下三件事：
1. **逻辑正确**：图中描述的流程/关系与概念的实际行为一致
2. **语法有效**：没有未闭合的引号、错误的箭头类型、缺失的节点引用
3. **可独立理解**：不看正文，只看图+标题+节点对照表，能理解核心意思吗？
