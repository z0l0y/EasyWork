# 深度复盘检查清单

本文档提供 TALK 阶段 5-Whys 分析和 Trade-offs 复盘的详细检查指导。

## 5-Whys 检查

### 追问深度检查
- [ ] 是否至少追问到了第 4 层？停留在第 1-2 层的通常是表面原因
- [ ] 每一层的答案是否为下一层的问题提供了自然的逻辑连接？
- [ ] 最终根因是否指向了系统性原因（流程/工具/机制/架构），而非个人失误？

### 常见追问陷阱

**陷阱一：停在"人犯错了"**
```
Why 1: 为什么会出这个 bug？ → 因为开发者忘记处理空值
（停在这里 — 这是典型的表面归因）
```
应该继续追问：
```
Why 2: 为什么忘记处理空值没有被发现？ → 因为 Code Review 没有覆盖这个文件
Why 3: 为什么 Code Review 没有覆盖？ → 因为这个文件在 CODEOWNERS 中没有指定审查者
Why 4: 为什么没有配置 CODEOWNERS？ → 因为团队迁移到 monorepo 时没有更新 CODEOWNERS 配置
→ 根因：monorepo 迁移的 checklist 缺失了 CODEOWNERS 更新这一项
```

**陷阱二：归因到外部不可控因素**
```
Why 3: 为什么这个第三方库会返回 null？ → 因为它的文档没有写清楚
（停在这里 — "别人没写清楚"不是一个可改进的根因）
```
应该继续追问：
```
Why 4: 既然第三方库文档不清，为什么我们没有做防御性编程？
→ 因为团队的前端最佳实践指南中缺少"所有外部数据必须做空值校验"这条规则
→ 根因：前端编码规范的覆盖度不足
```

**陷阱三：跳跃式追问**
```
Why 1: 接口超时 → 因为数据库慢
Why 2: 数据库为什么慢 → 因为没有索引
Why 3: 为什么没索引 → 因为没人建
（"没人建"跳跃到了"人的问题"，中间缺失了"为什么建索引的需求没有被提出和执行"）
```
正确的追问链：
```
Why 1: 接口超时 → 因为查询耗时 3 秒
Why 2: 查询为什么慢 → 因为 users 表缺少 status 字段索引
Why 3: 为什么缺少索引 → 因为建表脚本中没有定义这个索引
Why 4: 为什么建表脚本没定义 → 因为当初设计表结构时 status 只是枚举值，没想到数据量到百万级后需要按 status 过滤
Why 5: 为什么没有在设计评审时考虑数据量增长 → 因为设计评审 checklist 中没有"预估 6 个月后数据量并评估索引需求"这一项
→ 根因：设计评审流程缺少数据增长评估环节
```

### 追问质量检查
- [ ] 每一层追问都是基于上一层答案的自然延伸，不是跳跃到不相关的方向
- [ ] 采用了客观的归因逻辑（基于事实和证据），而非主观猜测
- [ ] 回避了"某人太菜"、"运气不好"等无法改进的归因
- [ ] 最终根因是可行动的（团队可以采取措施来系统性地防止这类问题）

## Trade-offs 分析检查

### 分析完整性
- [ ] 列出了本次方案的至少 1 项代价或副作用
- [ ] 区分了"主动选择"和"被迫妥协"
- [ ] 被迫妥协项有明确的原因说明和后续改进计划
- [ ] 没有假装方案是完美的（天下没有免费的午餐）

### 常见遗漏
Agent 容易遗漏的 Trade-off 类型：
- **可维护性 trade-off**：为了快速修复，代码写得不够优雅，增加了后续维护成本
- **通用性 trade-off**：针对特定场景的硬编码，未来扩展时需要重构
- **依赖 trade-off**：为了省事引入了一个新的第三方包，增加了项目的依赖负担
- **一致性 trade-off**：新代码的风格与老代码不一致，制造了认知分裂

## 工程规范提炼检查

### 规范可用性
- [ ] 提炼的规范是"可执行"的——一个初级工程师看了就知道该怎么做
- [ ] 不是空泛的口号（如"以后要更注意性能"——这没用）
- [ ] 包含了违规示例和合规示例（正反对比让人更容易理解）
- [ ] 如果可能，给出了自动化检查的方案

### 规范示例评估

❌ 无效的规范：
> "以后写代码时要注意安全性。"
> — 太模糊，无法执行，无法检查

❌ 同样无效：
> "所有代码都必须考虑安全。"
> — 什么样的代码有安全风险？怎么算"考虑了"？

✅ 有效的规范：
> "所有涉及用户输入的 API 参数，必须在 Controller 层使用 joi 或 zod 做 Schema 校验。校验规则必须包括：类型检查、必填/可选标注、字符串最大长度限制（防止恶意超长输入）、枚举值的白名单校验。此规则通过 ESLint 插件 `xxx` 在 CI 中强制检查，未通过校验的 PR 不允许合并。"
> — 具体到工具、层级、校验项、执行方式

## 质量红线

如果以下任一条不满足，TALK 阶段视为未完成：
1. 5-Whys 没有触及系统性根因（停在"人犯错了"或"第三方的问题"）
2. Trade-offs 分析为空（声称方案完美无代价）
3. 提炼的规范无法执行（太模糊、太空泛）
