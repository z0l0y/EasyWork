# 5-Whys 根因追溯方法论

> 用于 talk-retro 技能 · 深度复盘核心框架

## 原则

- **追问到系统性原因**，不停在表面症状
- 每一层 Why 必须有**证据支撑**（日志/代码/时间线）
- 避免"人的失误"作为终结点（追问"为什么系统允许这个失误发生？"）

## 六层追问模板

```
L1: 发生了什么？（症状）
  ↙
L2: 为什么会发生？（直接原因）
  ↙
L3: 为什么直接原因存在？（过程缺陷）
  ↙
L4: 为什么过程有缺陷？（系统/工具缺失）
  ↙
L5: 为什么系统缺失未被发现？（组织/文化根因）
  ↙
L6: 如何防止同类再次发生？（工程规范产出）
```

## 高质量 vs 低质量追问

| 层次 | ❌ 低质量 | ✅ 高质量 |
|------|----------|---------|
| L2 | "开发者写错了" | "类型检查未覆盖该路径，编译器未报错" |
| L3 | "没仔细看" | "Code Review checklist 无此项检查" |
| L4 | "流程不完善" | "CI 未配置该 linter 规则（具体规则名）" |
| L5 | "时间太紧" | "Sprint planning 低估了该模块复杂度（具体数据）" |

## 输出要求

每条复盘必须产出至少 **1 条可落地的工程规范**（具体到工具/配置/规则名称）：

```markdown
## 工程规范产出
- **规则**: [具体规则名]
- **落地方式**: [lint rule / CI check / code review checklist / team convention]
- **验证方式**: [如何确认该规则已生效]
```
