# Talk Retro（深度复盘）

## 前置判断

**必须执行**：Bug 修复（需要挖根因防复发）、重构（需要记录取舍）、生产事故。

**可以跳过**：功能开发（没有"为什么出问题"可追溯）、纯理解任务、微调。

**但**：如果 Bug 修复中发现了团队流程/规范的缺失，这一步是最有价值的产出。

## 阶段一：5-Whys 根因追溯

### 与 SUM 的边界

- **SUM §2（发现过程）**：说到"找到了出问题的代码行"为止。如"`auth.service.ts:88` 缺少空值判断"。
- **TALK §1（5-Whys）**：从这里开始继续往下追问——为什么那行代码会缺少空值判断？为什么 Code Review 没发现？为什么工具没拦住？一直追问到流程/机制层面的系统性根因。

### 方法

从 SUM 已定位的"代码行级根因"出发，连续追问直到触及流程/工具/机制层面的系统性根因。

### 关键规则

**不要停在"人犯错了"**。人犯错是症状，追问"为什么这个错误没被拦住"才是根因。

```
❌ 错误的追溯：
Why1: 为什么出bug？→ 开发者忘处理空值
（停在这里 → 归因到个人，无法系统性改进）

✅ 正确的追溯：
Why1: 为什么接口返回500？→ 因为变量为null时未做判断
Why2: 为什么没做空值判断？→ 开发者不知道这个字段可能为null
Why3: 为什么开发者不知道？→ 接口文档没有标注该字段的可选性
Why4: 为什么文档没标注？→ 团队的API文档模板没有"字段可空性"这一栏
Why5: 为什么模板有这个缺陷？→ 设计API文档模板时没有前端开发者参与审查
→ 根因：API文档模板的评审流程缺少跨角色视角
→ 行动：API文档评审必须包括1名前端+1名后端，检查字段可空性标注
```

### 根因类型检查

最终根因应指向以下之一（可行动的）：
- [ ] 流程缺失（缺Code Review、缺上线检查清单）
- [ ] 工具不足（缺静态分析、缺监控告警）
- [ ] 知识断层（文档缺失、onboarding 不完整）
- [ ] 设计缺陷（单点故障、过度耦合）

## 阶段二：Trade-offs 取舍分析

诚实地记录本次方案付出了什么代价。没有完美的方案。

| 取舍项 | 得到的好处 | 付出的代价 | 主动/被迫 | 后续建议 |
|--------|-----------|-----------|----------|---------|
| 降 bcrypt factor | 延迟↓88% | 安全性微降（仍安全） | 主动 | 6个月后安全审计确认 |

## 阶段三：工程规范提炼

从根因分析中提炼出**具体可执行**的规范，而非"以后要注意"这类废话。

❌ 无效规范："以后写代码要注意安全"
✅ 有效规范："所有涉及用户输入的API参数，必须在Controller层用zod做Schema校验，包括类型检查、必填标注、字符串最大长度限制。此规则通过CI中的 `npm run lint:schema` 强制检查。"

> **详细检查清单**：5-Whys追问质量检查、常见陷阱、Trade-offs分析完整性与工程规范可用性评估，见 `references/retro-checklist.md`。

## 遇到黑盒时

如果追问到第 3-4 层时触及闭源库/第三方黑盒，无法继续：
- 如实标注"根因分析在第 N 层中断，因为 XX 组件为外部黑盒"
- 将问题抛给用户补充信息

## 重复根因检测（🆕 v2.3）

如果本次任务的 `root_cause` 与 JSONL 日志中任一历史任务的 `root_cause` 相同：
- 标注 `[重复根因]`——说明团队在上次复盘后未能阻止同类问题再次发生
- 在 `engineering_rules` 中追加"上次提炼的规范未落地，需要强制执行措施"
- 建议将此根因对应的规范升级为 `references/team-policy.md` 中的强制规则（MUST）

## 反模式

- ❌ 5-Whys 停在"人写错了"——这不是根因，这是甩锅
- ❌ Trade-offs 表为空——没有完美的方案，只是你没认真想代价
- ❌ 规范写"以后多注意"——等于没写
- ❌ 追问是跳跃式的（Why1说A → Why2跳到不相关的Z）——逻辑要连贯
