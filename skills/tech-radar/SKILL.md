# Tech Radar（技术前沿雷达）v2.0

## 定位

**面向高级后端开发工程师的技术信息策展系统。不只是扫描——而是你可以自由缩放的技术望远镜：从 30 秒快速面扫到 10 分钟全面体扫到不限时间的聚焦点扫。**

你的时间有限，但每天产生的新技术信息是无限的。tech-radar 做三件事：

1. **替你扫**——从 20+ 个源头并行抓取最新动态，不用你一个个去刷
2. **替你筛**——按你的角色（高级后端+Agent 开发）打分，分三层：必读 / 值得关注 / 了解一下
3. **替你深读**——你挑出感兴趣的文章，agent 用 `read-paper` 级别的深度帮你分析

**🆕 v2.0 核心变化 —— 点面体三级粒度，你可以自己控制"望远镜"的焦距：**

```
🎯 点 (Point)  → 聚焦单一技术，穷尽深扫。   "MCP 协议生态现在是什么情况？"
📐 面 (Surface) → 快速浏览领域表层动态。     "这周 AI 和 Go 有什么新东西？"
🌐 体 (Volume) → 全领域 MRI 级全面扫描。   "帮我全面深扫最近一个月的技术动态"
```

**跟单纯用 WebSearch 的区别**：

| | WebSearch | tech-radar |
|---|----------|-----------|
| 覆盖 | 单一搜索引擎结果 | 20+ 源头并行（GitHub/ArXiv/HN/工程博客/Reddit/会议） |
| 排序 | 搜索引擎自己的算法 | 按**你的角色**打分（后端+Agent 开发） |
| 粒度 | 无粒度控制 | 点面体三级缩放——从单技术到全领域 |
| 输出 | 链接列表 | 结构化摘要 + 为什么值得看 + 建议动作 |
| 后续 | 你得自己一篇篇看 | 选中的文章自动深度分析 |
| 趋势 | 无 | 跨领域趋势综合（如"Go 和 Rust 在后端都在往 Wasm 走"） |

## 前置判断

### 覆盖领域

| 领域 | 关注重点 | 为什么 |
|------|---------|--------|
| 🤖 **AI/ML** | LLM/Agent 框架、推理优化、RAG、多模态、MLSys | 直接相关——你在开发 Agent 系统 |
| 🏗️ **后端架构** | 分布式系统、微服务、数据库、消息队列、可观测性 | 你的本职工作 |
| ☕ **Java** | 新版本特性、Spring 生态、虚拟线程、GraalVM、JVM 优化 | 后端主力语言 |
| 🔵 **Go** | 新版本特性、高性能框架、并发模式、工具链 | 后端主力语言 |
| 🔗 **Agent 开发** | Agent 框架、Tool Use、Multi-Agent、Agent 评测、MCP | 你在做 Agent 相关内容 |

### 扫描粒度（🆕 v2.0 点面体三级缩放）

用户控制的不只是"扫多少条"，而是**扫多宽**（范围）和**挖多深**（深度）两个维度：

| 粒度 | 触发词示例 | 范围 | 深度 | 采集策略 | 输出 |
|------|-----------|------|------|---------|------|
| 🎯 **点** Point | "深扫 MCP"、"聚焦 Virtual Threads"、"这个技术怎么样" | 单一主题/技术，跨所有相关领域搜索 | **最深**：穷尽采集、逐条深度分析、生态全景 | 按主题跨源深挖（WebSearch+WebFetch） | 8 段聚焦报告（~800-1500 字） |
| 📐 **面** Surface（默认） | "技术保鲜"、"扫一下技术动态"、"快速扫一下"、"有什么新东西" | 全部选定领域 | **最浅**：每领域 Top 3-5 条，标题+一句话摘要 | 每领域 1-2 个快速源轻量采集 | 压缩汇总表（~300-500 字，30 秒扫完） |
| 🌐 **体** Volume | "全面深扫"、"深度扫描"、"全面扫一下" | 全部选定领域 | **最深**：每领域 Top 15-20 条，全部来源，完整趋势分析 | 全源并行深采集 | 完整雷达报告（~2000-4000 字）+ 可选每领域深度附录 |

**关键语义区别**：
- **点** = 范围窄 + 深度最大 → 像拿着**放大镜**看一片叶子，看清楚叶脉纹路
- **面** = 范围广 + 深度最浅 → 像**雷达屏幕**快速扫一圈，只标记亮点位置
- **体** = 范围广 + 深度最大 → 像做**MRI 全身扫描**，每个部位都切到薄层

**用户可通过自然语言混合控制**：
- "AI 领域深度扫，其他领域快速扫" → AI=volume, 其余=surface
- "只扫 Go 和 Agent，要全面" → 仅 Go+Agent, 均为 volume
- "帮我聚焦看看 MCP 协议最近的进展" → 自动识别为 point 模式，topic="MCP 协议"
- 未指定粒度时默认**面模式**（最符合"保鲜"场景的日常需求）

### 扫描范围

每次扫描默认覆盖**过去 7 天**（可调整："扫最近一个月的" / "扫今天的"）。

用户可指定领域："只扫 AI 和 Go" / "全扫"（默认全扫）。

**点模式的特殊范围逻辑**：
- 点模式不使用领域过滤——它根据 `topic` 自动判断哪些领域相关
- 例如 `topic="MCP 协议"` 会跨 Agent、AI/ML、后端三个领域搜索
- Agent 在采集前先做**领域预判**："{topic} 跟哪几个领域有关？" → 只搜相关领域的数据源

## 执行流程

### 阶段 0：粒度判定（🆕 v2.0）

**在采集之前，先确定扫描粒度**：

```
用户输入分析:
  ├── 含 "深扫"/"聚焦"/"这个技术怎么样" + 具体技术名 → 🎯 点模式
  │   提取 topic（如"深扫 MCP 协议" → topic="MCP 协议"）
  │
  ├── 含 "快速扫"/"有什么新东西"/"扫一下"（无"全面""深"）→ 📐 面模式（默认）
  │
  ├── 含 "全面深扫"/"深度扫描"/"全面扫" → 🌐 体模式
  │
  └── 模糊（只说"技术保鲜"/"tech radar"）→ 📐 面模式（默认）
```

**粒度确认**（Agent 在采集前输出一行，不等待用户确认）：
```
🛰️ Tech Radar — {点/面/体}模式扫描
  {点: 聚焦 "{topic}" | 面: 快速浏览 {N} 个领域 | 体: 全面深扫 {N} 个领域}
  时间范围: {过去 7 天 / 30 天 / today}
```

### 阶段 1：按粒度分支采集

Agent 根据粒度采用不同的采集策略：

---

**🎯 点模式采集（主题聚焦深挖）**：

```
策略: WebSearch 为主（跨源精准搜索） + WebFetch 为辅（深挖关键源）

Step 1: 领域预判
  Agent 先判断 topic 与哪些领域相关：
  "MCP 协议" → Agent + AI/ML + 后端
  "Java 虚拟线程" → Java + 后端
  只搜相关领域的源，不搜无关领域。

Step 2: 多角度搜索（WebSearch，3-5 个 query）
  - "{topic} latest developments 2025"
  - "{topic} github trending"
  - "{topic} arxiv paper"
  - "{topic} engineering blog"
  - "{topic} 最新进展"（中文源）

Step 3: 关键源补充（WebFetch）
  - 如果 topic 涉及开源 → GitHub 搜索 topic 关键词
  - 如果 topic 涉及论文 → arXiv 搜索 topic 关键词
  - 如果 topic 涉及 Agent → Anthropic/LangChain 博客搜索

Step 4: 全量保留
  - 每个来源返回的条目全部保留（不限制数量）
  - 去重后全部进入阶段 2
  - 如果结果 > 30 条 → 按评分 Top 30 深度分析，其余列表附录
  - 如果结果 < 5 条 → 标注"本期该主题信号较弱"，但不降级到其他模式
```

---

**📐 面模式采集（领域表层快扫）**：

```
策略: 每领域最少源、最少条目，追求速度

Step 1: 每领域只选 1-2 个最快的源
  AI/ML → Hacker News + arXiv cs.AI（只看标题）
  后端 → Hacker News + Reddit r/programming
  Java → GitHub Trending Java + InfoQ Java（只看标题）
  Go → GitHub Trending Go + Go Blog（只看标题）
  Agent → Anthropic Blog + GitHub agent-framework topic

Step 2: 每源只取 Top 3-5 条目
  - 最小提取：标题 + 英文原标题 + 类型 + 一句话摘要（≤30 字）+ 来源链接
  - 不做深度提取（无核心看点、无推荐理由、无建议动作）

Step 3: 去重后每领域保留 3-5 条
  - 如某领域 < 3 条，标注"本周信号较弱"
  - 不展开跨源合并详情
```

---

**🌐 体模式采集（全源深采集）**：

```
策略: 全源并行深采集（当前 deep 模式行为，增强）

Step 1: 全源并行搜索（所有 20+ 源头）
  使用 §数据源完整列表 中的所有源头，分领域并行 WebFetch/WebSearch

Step 2: 每源取 Top 5-10
  - 完整结构化提取：标题（中英）+ 来源 + 类型 + 摘要 + 技术关键词
  - 去重合并（多源验证标注）

Step 3: 去重后每领域保留 15-20 条
  - 候选集上限: 每领域 ≤20 条，总条目 ≤100（避免 LLM 分析时信息过载）
  - 超出上限时: 按评分降序取前 N 条，其余列表附录
```

---

**并行采集通用规则**：
- Agent 同时对相关领域发起 WebSearch + WebFetch
- 面模式采集时间控制在 **15 秒内**
- 体模式采集时间控制在 **45 秒内**
- 点模式采集时间控制在 **30 秒内**（搜索 query 数较少但每个 query 更深）

### 阶段 2：去重 + 结构化提取

对采集到的所有条目：

```
每条目提取：
├── 标题（中英双语——英文标题+中文翻译）
├── 来源（GitHub/ArXiv/HN/...）+ 链接
├── 类型（论文/开源项目/博客文章/工具发布/会议演讲/新闻）
├── 一句话摘要（≤50 字中文，说清楚是什么）
└── 技术关键词（3-5 个，用于后续相关性打分）
```

**去重规则**：同一条消息在多个源头出现 → 合并，标注"多源验证"。

### 阶段 3：相关性打分

对每条目用以下维度打分（面向"高级后端+Agent 开发"角色）：

| 维度 | 权重 | 打分标准 |
|------|------|---------|
| **领域匹配** | 30% | 直接相关(AI/后端/Java/Go/Agent)=5 / 间接相关=3 / 弱相关=1 |
| **技术深度** | 25% | 有架构/设计决策/性能数据的深度技术内容=5 / 概述性=3 / 新闻稿=1 |
| **影响力** | 20% | 来自顶级会议/知名作者/大厂工程博客/社区热议=5 / 中等=3 / 小众=1 |
| **可行动性** | 15% | 看完能直接用/能改架构决策=5 / 增长见识=3 / 纯科普=1 |
| **时效性** | 10% | 本周内=5 / 本月=3 / 更早=1 |

**综合分 = Σ(维度得分 × 权重)**

### 阶段 4：三级分层

| 层级 | 分数 | 标签 | 含义 |
|------|------|------|------|
| 🔴 **Tier 1 — 必读** | ≥4.0 | `[必读]` | 直接相关 + 高影响力 + 可行动。不看可能错过重要方向。 |
| 🟡 **Tier 2 — 值得关注** | 2.5–3.9 | `[关注]` | 有学习价值，但不紧急。有空时读。 |
| 🟢 **Tier 3 — 了解一下** | <2.5 | `[了解]` | 知道有这个东西就行，不需要深读。 |

**分层调整**：
- 如果某个领域 Tier 1 为空 → 降低该领域阈值到 3.5，标注"本周无高影响力内容"
- 如果 Tier 1 > 10 条 → 只保留 Top 10，其余降级到 Tier 2

### #1 {标题}（{英文原标题}）

## 定期保鲜模式

### 手动定期

用户每次说"帮我保鲜"即触发一次标准扫描。适合不定期使用。

### Cron 自动保鲜（推荐）

用户可以设置 Cron 定时任务自动扫描：

```
用户：帮我每周一早上 9 点扫一次技术动态
Agent：创建 Cron 任务 → 每周一 9:00 自动执行 tech-radar
→ 产出报告 → 下次打开 Claude Code 时看到
```

**建议频率**：
| 频率 | 适合场景 |
|------|---------|
| 每天 | 极度关注前沿，时间充裕 |
| 每周（推荐） | 正常工作节奏，保持竞争力 |
| 每两周 | 时间紧张，但仍需保鲜 |
| 每月 | 最低限度，了解大方向 |

### 保鲜偏好学习

Agent 会从用户的深读选择中学习偏好：
- 用户总是深读 Agent 相关的 → 下期自动提高 Agent 领域的权重
- 用户从来没深读过 Java → 下期降低 Java 领域数量
- 用户跳过某种类型的条目（如纯新闻）→ 下期减少该类型

偏好记录在报告中（不写入磁盘，避免隐私问题）：
```
【你的阅读偏好（本期推断）】
- 高兴趣：Agent 框架对比、Go 并发模式、LLM 推理优化
- 低兴趣：Java 版本升级新闻、前端工具
```

## 反模式

- ❌ 把扫描变成 Google 搜索结果列表——每个条目必须有"为什么值得看"和"与你的关系"
- ❌ Tier 分级全凭感觉——必须有 5 维度评分表支撑，不能拍脑袋分
- ❌ Tier 1 太多（>10 条）——"必读"超过 10 条等于没有必读
- ❌ 趋势分析是套话——"AI 正在改变世界"不如"本周有 3 个新 Agent 框架都放弃了 LangChain 改用直接调用 LLM API"
- ❌ 只扫英文源不扫中文源——你读中文比英文快，中文技术社区（知乎/掘金/InfoQ中文）也应该覆盖
- ❌ 忽略工程博客——大厂工程博客（Uber/Meta/Netflix）的文章质量往往高于论文
- ❌ 数据源过少——只搜 GitHub Trending = 只看到工具，漏了论文和架构思想
- ❌ 深读时复制 read-paper 全文模板——对于开源项目/博客文章，用适配后的模板
- ❌ 输出完报告就走——必须提供后续交互选项（深读/对比/过滤/导出/定时）
- ❌ 不记录用户偏好——连续多次跳过的内容类型应该自动降权
- ❌ 定期保鲜的 Cron 任务没提醒用户手动确认——写磁盘的定时任务需要用户知情
- ❌ 对中文源和英文源用同样的搜索策略——中文社区的内容形态不同（知乎=深度长文，掘金=实战教程）
- ❌ **（🆕 v2.0）用点模式扫全领域**——点模式适合"我想搞清楚 MCP 协议"这种单一主题。全领域扫描请用体模式。如果用户说"帮我深扫所有领域"，自动纠正为体模式
- ❌ **（🆕 v2.0）面模式输出深度分析**——面模式用户要的是 30 秒扫完。不要在面模式下展开核心看点、推荐理由、建议动作。只给标题+一句话摘要
- ❌ **（🆕 v2.0）点模式结果太少就放弃**——如果某主题结果 <5 条，标注"本期该主题信号较弱"而不是自动降级到面模式。"没什么新东西"本身就是有效信息
- ❌ **（🆕 v2.0）体模式全量塞给用户**——总条目超过 50 条时，提示用户可以考虑切换到面模式或对某个领域做点模式聚焦
- ❌ **（🆕 v2.0）忽略粒度间的"缩放"引导**——面模式后必须引导"放大到体/聚焦到点"，体模式后必须引导"聚焦到点"，点模式后必须引导"追踪相关技术/放大到体"。粒度缩放是本技能的核心交互模式
- ❌ **（🆕 v2.0）点模式不判断领域**——点模式不要在 5 个不相关的领域里均匀搜索。"MCP 协议"只跟 Agent 和 AI/ML 有关，跟 Java 无关。先做领域预判再搜
